# Сервис биллинга Lyra — требования

## Назначение

Учёт баланса пользователей и контроль расходов на использование Claude. Bridge обращается к сервису биллинга для проверки и списания средств. Claude о биллинге не знает.

## Кто обращается к сервису

- **Bridge** — проверка баланса при подключении, списание по итогам сессии/запроса
- **Мобильное приложение** — просмотр баланса, пополнение, история

## API (Bridge → биллинг)

### Проверка баланса

```
GET /balance/{userId}
```
Ответ:
```json
{
  "userId": "...",
  "balance": 12.50,
  "currency": "USD",
  "maxBudgetUsd": 5.00
}
```

### Списание

Bridge передаёт сырые данные о токенах. Биллинг сам знает тарифы и считает стоимость.

```
POST /charge
{
  "userId": "...",
  "sessionId": "...",
  "model": "sonnet",
  "inputTokens": 1200,
  "outputTokens": 340
}
```
Ответ:
```json
{
  "ok": true,
  "charged": 0.03,
  "balance": 12.47
}
```
или
```json
{
  "ok": false,
  "error": "insufficient_funds",
  "balance": 0.00
}
```

Bridge не знает цен — только передаёт модель и количество токенов.

## Когда Bridge обращается к биллингу

1. **При подключении (hello)** — проверка баланса. Баланс > 0 → пускаем. Баланс = 0 → отказ.
2. **Перед каждым сообщением пользователя** — проверка баланса. Баланс > 0 (любой положительный) → пропускаем. Баланс <= 0 → отказ, сообщение не передаётся в Claude.
3. **После каждого ответа Claude** — Bridge получает количество токенов и модель из result, отправляет в биллинг. Биллинг считает стоимость и списывает.
4. **Баланс ушёл в минус** — допускается (запрос уже выполнен). На следующий запрос пользователя проверка не пропустит.

## Тарификация

- Биллинг хранит таблицу тарифов: модель → цена за inputToken, цена за outputToken
- Bridge не знает тарифов, передаёт только сырые данные (модель, токены)
- Биллинг сам рассчитывает стоимость и списывает
