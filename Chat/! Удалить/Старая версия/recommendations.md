# Рекомендации по развитию Lyra-Chat

> Анализ `src/Lyra-Chat/Forms/Форма/Ext/Form/Module.bsl` (1483 строки).
> Обработка состоит из одной главной формы и одной пустой `ФормаПодключение`.

---

## Часть 1. Что нужно сделать прямо сейчас

### 1.1 Разбить God-форму на модули

**Проблема:** 1483 строк в одной форме. Смешаны: UI-обработчики, WebSocket/Bridge, парсинг JSON, MCP-инструменты, утилиты, весь код фреймворка ЕХТ_Чат.

**Решение:** Вынести в отдельные формы-модули по паттерну `ПолучитьФорму("...", Параметры, ЭтаФорма)`.

Рекомендуемая разбивка:

| Форма-модуль | Что содержит |
|---|---|
| `МодульBridge` | WebSocket-соединение, отправка, закрытие, `КлючСоединения()` |
| `МодульПарсерClaude` | `ОбработатьОтветBridge()`, `ИзвлечьТекстДельты()`, `ИзвлечьЗначениеJSON()`, весь строковый парсинг |
| `МодульMCP` | `ОбработатьMCPЗапрос()`, `ВыполнитьЗапросНаСервере()`, `ПолучитьМетаданныеНаСервере()` и т.д. |
| `МодульЧат` | `ВывестиВЧатСообщениеАссистента()`, `ВывестиВЧатОшибку()`, `ДобавитьСообщениеВЧат()` и т.д. |

Главная форма после разбивки: только инициализация + связывание модулей + UI-обработчики команд (~100 строк).

```1c
// ПриСозданииНаСервере — главная форма после рефакторинга
Чат = ЕХТ_Чат_ПриСозданииНаСервере(...);
Чат.ЕХТ_Чат_Инициализация(...);

// ПриОткрытии — инициализация клиентских модулей
МодульBridge      = ПолучитьФорму(ПутьКФормам + "МодульBridge",      , ЭтаФорма);
МодульПарсерClaude = ПолучитьФорму(ПутьКФормам + "МодульПарсерClaude", , ЭтаФорма);
МодульMCP         = ПолучитьФорму(ПутьКФормам + "МодульMCP",          , ЭтаФорма);
МодульBridge.Инициализация(ЭтаФорма, МодульПарсерClaude);
МодульMCP.Инициализация(ЭтаФорма);
```

### 1.2 Убрать код ЕХТ_Чат из формы пользователя

**Проблема:** Секция `// === НАЧАЛО: Код фреймворка ЕХТ_Чат ===` (строки 1003–1481) — это копипаста. При обновлении фреймворка придётся вручную синхронизировать каждую форму-потребитель.

**Решение:**
- Вынести ЕХТ_Чат в отдельную форму-модуль `МодульЕХТЧат` или держать через существующий механизм `ЕХТ_Чат_ПриСозданииНаСервере` / `Чат.ЕХТ_Чат_Инициализация` — но тогда убрать дублирующий код из формы пользователя.
- Правило: в форме пользователя должны быть **только callbacks**: `ЕХТ_Чат_ПриОткрытии`, `ЕХТ_Чат_ПриДобавленииСообщения`, `ЕХТ_Чат_ПриНажатииСсылки`. Весь остальной код фреймворка — снаружи.

### 1.3 Исправить `ПолноеИмя()`

**Проблема:** Функция `ПолноеИмя()` (строка 997) возвращает `"ВнешняяОбработка.ЕХТ_Чат_БезРасширения"` — старое имя, скопированное из предыдущей версии. Реальное имя обработки — `Lyra_Chat`.

**Решение:** Использовать динамическое вычисление (как в Диадоке):

```1c
// Вместо хардкода

    ПутьКФормам = РеквизитФормыВЗначение("Объект").Метаданные().ПолноеИмя() + ".Форма.";
    // ...
КонецПроцедуры
```

Сохрани `ПутьКФормам` как реквизит формы — он понадобится клиентским процедурам для `ПолучитьФорму()`.

### 1.6 Реализовать стриминг

### 1.7 Заменить строковый парсинг JSON

**Проблема:** `ИзвлечьЗначениеJSON`, `ИзвлечьОбъектJSON`, `ИзвлечьТекстДельты`, `ДекодироватьJSONСтроку` — хрупкий самодельный парсер на `СтрНайти`. Ломается на пробелах, необычном форматировании, вложенных структурах.

Комментарий в коде объясняет причину: `ЕХТ_Значение.ИзJSON()` парсит в `Структуру`, а ключи с дефисами (`session_id`) невалидны для имён свойств Структуры.

**Правильное решение** — использовать `ЧтениеJSON` с `ТипСоответствие`:

```1c
// Парсим в Соответствие — принимает любые ключи включая с дефисами
ЧтениеJSON = Новый ЧтениеJSON;
ЧтениеJSON.УстановитьСтроку(СтрокаJSON);
ДанныеОтвета = ПрочитатьJSON(ЧтениеJSON, Истина);  // Истина = использовать Соответствие

ТипСобытия = ДанныеОтвета["type"];
// Для stream_event:
ТекстДельты = ДанныеОтвета["event"]["delta"]["text"]; // без хрупкого поиска подстрок
```

Это позволит убрать все 4 самодельные функции и `ДекодироватьJSONСтроку`.

### 1.8 Разделить временное состояние и сохраняемые настройки

**Проблема:** `<AutoSaveDataInSettings>Use</AutoSaveDataInSettings>` — форма сохраняет **все** реквизиты автоматически, включая `ТекущееСообщениеТекст`, `Занят`, `МожноЗакрытьФорму`. При следующем открытии форма может восстановить некорректное состояние.

**Решение:** Отделить настройки (что сохранять) от состояния (что не сохранять).

Варианты:
1. Оставить `AutoSaveDataInSettings = Use`, но убрать реквизиты состояния из формы — сделать их `Перем` в клиентском модуле (не реквизиты формы):
   ```1c
   &НаКлиенте
   Перем Занят;
   Перем МожноЗакрытьФорму;
   Перем ТекущееСообщениеТекст;
   Перем ТекущееСообщениеИД;
   ```
2. Оставить как реквизиты, но переключить `AutoSaveDataInSettings = DontUse` и вручную сохранять только нужные данные через `ХранилищеОбщихНастроек`.

### 1.9 Исправить `ЭкранироватьJSON` — переключиться на `ЕХТ_Значение.вJSON`

**Проблема:** MCP-ответ собирается конкатенацией строк (строки 589–594):

```1c
ОтветJSON = "{""type"":""mcp_response"",""requestId"":""" + ИДЗапроса
    + """,""result"":" + РезультатJSON + "}";
```

Это сломается если `ИДЗапроса` содержит специальные символы.

**Решение:** Собирать структуру и сериализовать:

```1c
Ответ = Новый Структура("type,requestId", "mcp_response", ИДЗапроса);
Если ЕстьОшибка Тогда
    Ответ.Вставить("error", Результат.error);
Иначе
    Ответ.Вставить("result", Результат);
КонецЕсли;
Соединение.ОтправитьСообщение(ЕХТ_Значение.вJSON(Ответ));
```

## Часть 2. Рекомендации по дальнейшей разработке

### 2.1 Архитектура при росте функциональности

При добавлении новых MCP-инструментов форма `МодульMCP` начнёт расти. Используй **Strategy-диспетч** вместо длинного `Если/ИначеЕсли`:

```1c
// МодульMCP — регистрация обработчиков
Процедура Инициализация()
    ОбработчикиИнструментов = Новый Соответствие;
    ОбработчикиИнструментов["1c_query"]    = "ВыполнитьЗапросНаСервере";
    ОбработчикиИнструментов["1c_metadata"] = "ПолучитьМетаданныеНаСервере";
    ОбработчикиИнструментов["1c_eval"]     = "ВычислитьНаСервере";
    ОбработчикиИнструментов["1c_exec"]     = "ВыполнитьКодНаСервере";
    // Новый инструмент — одна строка:
    ОбработчикиИнструментов["1c_post"]     = "ПровестиДокументНаСервере";
КонецПроцедуры

Функция ВыполнитьИнструмент(ИмяИнструмента, Ввод)
    ИмяОбработчика = ОбработчикиИнструментов[ИмяИнструмента];
    Если ИмяОбработчика = Неопределено Тогда
        Возврат Новый Структура("error", "Неизвестный инструмент: " + ИмяИнструмента);
    КонецЕсли;
    Возврат Вычислить("ЭтотОбъект." + ИмяОбработчика + "(Ввод)");
КонецФункции
```

### 2.2 Несколько независимых диалогов (мультисессионность)

Текущая архитектура: одна форма = одна сессия = одно соединение. Для нескольких диалогов:

- Список сессий как ТаблицаЗначений: `[ИД, Название, ПоследнееСообщение, Статус]`
- Каждый диалог — отдельный `ИдентификаторСессии` с отдельным WebSocket-соединением (ключ соединения включает ИД диалога)
- При переключении диалога — `loadMessages()` с историей нужной сессии

### 2.3 Сохранение истории диалога

Сейчас сообщения живут только в HTML-чате (JavaScript DOM) — они исчезают при закрытии формы.

**Варианты хранения:**

| Вариант | Плюсы | Минусы |
|---|---|---|
| `ХранилищеОбщихНастроек` | Просто, встроено в платформу | Ограничен объём, данные пользователя |
| `ХранилищеСистемныхНастроек` | Общее для всех пользователей | Конкурентный доступ |
| Файл JSON | Простой формат, diff в git | Нет транзакций |
| Таблица БД (через конфигурацию) | Полноценное хранение | Требует изменения конфигурации |

Оптимально для начала — `ХранилищеОбщихНастроек` с ключом `"Lyra.История." + ИдентификаторСессии`.

### 2.4 Системный промпт и контекст

Возможности для обогащения контекста Claude при каждом запросе:

- **System prompt** — передаётся через Bridge в заголовок сессии

- **Инструкции MCP** — список доступных инструментов с описанием в system prompt, чтобы Claude знал что умеет

```1c
- 1c_metadata(type: string): получить список объектов метаданных
- 1c_eval(expression: string): вычислить выражение 1С
- 1c_exec(code: string): выполнить блок кода 1С";
```

### 2.7 Обработка ошибок и переподключение

Текущее поведение: при ошибке WS форма просто закрывается.

**Нужно добавить:**
- Автопереподключение с exponential backoff
- Различать: Bridge недоступен (ошибка подключения) vs Claude вернул ошибку (agent_error)
- При `agent_error` — показывать кнопку «Повторить» последний запрос
- Сохранять последний отправленный текст до получения `result`

```1c
// При ошибке соединения — пытаемся переподключиться
&НаКлиенте

    Если ПопытокПереподключения <= 3 Тогда
        ЗадержкаСек = ПопытокПереподключения * 2;  // 2, 4, 6 сек
        ПодключитьОбработчикОжидания("ПереподключитьсяКBridge", ЗадержкаСек, Истина);
    КонецЕсли;
КонецПроцедуры
```

## Часть 3. Хранение настроек

### 3.1 Что сейчас есть в Lyra-Chat

В `Form.xml` включено `<AutoSaveDataInSettings>Use</AutoSaveDataInSettings>`. Платформа автоматически сохраняет и восстанавливает реквизиты, у которых есть секция `<Save>` в XML.

**Что сохраняется сейчас:**

| Реквизит | Сохраняется? | Нужно? |
|---|---|---|
| `Чат1_Настройка_ТемнаяТема` | ✅ | ✅ |
| `Чат1_Настройка_Аватары` | ✅ | ✅ |
| `Чат1_Настройка_Автосвертка` | ✅ | ✅ |
| `Чат1_ИдентификаторСессии` | ✅ | ✅ (для возобновления сессии) |
| `Модель` | ✅ | ✅ |
| `ТекущееСообщениеТекст` | ❌ (нет `<Save>`) | ❌ правильно |
| `Занят` | ❌ | ❌ правильно |
| `МожноЗакрытьФорму` | ❌ | ❌ правильно |

Текущая расстановка в целом **правильная** — временное состояние уже не сохраняется. Но механизм фрагилен: при добавлении нового реквизита формы платформа начнёт его сохранять автоматически, если явно не убрать `<Save>`.

### 3.2 Рекомендуемый паттерн: обёртка над хранилищем

Не обращаться к `ХранилищеОбщихНастроек` напрямую из разных мест — вынести в одну пару функций в ObjectModule. Паттерн взят из Диадока:

```1c
// ObjectModule.bsl
Функция НастройкиЗагрузить(КлючНастроек, ЗначениеПоУмолчанию = Неопределено) Экспорт
    Результат = ХранилищеОбщихНастроек.Загрузить("LyraChat", КлючНастроек);
    Если Результат = Неопределено Тогда
        Результат = ЗначениеПоУмолчанию;
    КонецЕсли;
    Возврат Результат;
КонецФункции

Процедура НастройкиСохранить(КлючНастроек, Значение) Экспорт
    ХранилищеОбщихНастроек.Сохранить("LyraChat", КлючНастроек, Значение);
КонецПроцедуры
```

Корневой ключ `"LyraChat"` — в одном месте. Если переименуешь продукт, меняешь одну строку.

### 3.3 Рекомендуемая структура ключей

```
"LyraChat" → "НастройкиПодключения"   // URL Bridge, порт
"LyraChat" → "НастройкиUI"            // тема, аватары, автосвертка
"LyraChat" → "НастройкиClaude"        // модель, системный промпт
"LyraChat" → "ИсторияСессий"          // список ИД сессий с метаданными
"LyraChat" → "История/" + SessionID   // сообщения конкретной сессии
```

Сессионный идентификатор в ключе — паттерн Диадока: `ProductID/SessionNumber/DataKey`.

---

### 3.4 Когда `AutoSaveDataInSettings` достаточно, а когда нет

**Достаточно** — для простых скалярных настроек (булево, строка, число), которые пользователь меняет через элементы формы и которые не нужны при закрытой форме.

**Недостаточно**, когда:
- Нужно читать настройки **до открытия формы** (например, в фоновом задании)
- Нужно сохранить **сложный объект** — Структуру, Массив, дерево (AutoSave не умеет)
- Нужна **история диалогов** — это точно требует `ХранилищеОбщихНастроек` с явными ключами
- Нужно **мигрировать** настройки при обновлении (паттерн из раздела 13.6 отчёта)

Для Lyra-Chat критичный случай — история диалога и системный промпт: они не помещаются в реквизит формы, их нужно хранить явно.

---

### 3.5 Сохранение истории диалога

```1c
// ObjectModule.bsl
Процедура ИсториюСохранить(ИдентификаторСессии, МассивСообщений) Экспорт
    НастройкиСохранить("История/" + ИдентификаторСессии, МассивСообщений);
КонецПроцедуры

Функция ИсториюЗагрузить(ИдентификаторСессии) Экспорт
    Возврат НастройкиЗагрузить("История/" + ИдентификаторСессии, Новый Массив);
КонецФункции

Функция СписокСессийЗагрузить() Экспорт
    Возврат НастройкиЗагрузить("СписокСессий", Новый Массив);
КонецФункции
```

`МассивСообщений` — массив структур `{роль, текст, время}`. Платформа сериализует массивы структур автоматически.

**Ограничение объёма:** `ХранилищеОбщихНастроек` хранит до 10 МБ на ключ. Длинные диалоги с большим количеством сообщений могут превысить лимит — стоит хранить только последние N сообщений (например, 100).

---

### 3.6 Безопасное удаление и очистка истории

По паттерну СБИС — удаление всегда в `Попытка/Исключение`:

```1c
Процедура ИсториюУдалить(ИдентификаторСессии) Экспорт
    Попытка
        ХранилищеОбщихНастроек.Удалить("LyraChat", "История/" + ИдентификаторСессии);
    Исключение
        // Нормально — может уже не существовать
    КонецПопытки;
КонецПроцедуры
```
