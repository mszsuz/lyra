# Полное код-ревью Lyra Chat

Дата: 2026-03-17
Область ревью: весь код проекта `Lyra-Chat-multiform.epf/src`
Формат: статическое ревью всех `.bsl`-модулей без end-to-end прогона в 1C/web-клиенте.

## Findings

### [P1] Серверный `lyra_data_query` ломается на любом непустом результате из-за итерации не по тому объекту
- В `Forms/МодульОбработка/Ext/Form/Module.bsl:108-123` массив имён колонок собирается в переменную `Колонки`, но дальше строки выборки читаются циклом `Для Каждого ИмяКолонки Из Колонка Цикл`.
- `Колонка` здесь — переменная из предыдущего цикла, а не массив имён колонок. Для непустого результата это приводит к ошибке итерации или пустому/некорректному наполнению строк.
- Итог: инструмент `lyra_data_query` ненадёжен именно в основном happy-path сценарии, когда запрос что-то вернул.

### [P1] Привилегированный режим включается и не гарантированно выключается
- В `Ext/ObjectModule.bsl:91-100` при исключении в блоке получения `ssl_id` режим включается через `УстановитьПривилегированныйРежим(Истина)`, но в ветке `Исключение` не выключается.
- В `Ext/ObjectModule.bsl:154-189` функция `ПолучитьИдентификаторКонфигурацииMD5()` включает привилегированный режим и вообще нигде его не выключает, включая ранний `Возврат ""`.
- Так как обе функции вызываются внутри одного серверного вызова `ПолучитьДанныеБазыДляHello()`, привилегии могут утечь на оставшуюся часть этого же вызова.

### [P1] Асинхронные запросы к 1С:Напарнику конфликтуют между собой по `request_id`
- В `Forms/МодульНапарник/Ext/Form/Module.bsl:76-82` модуль хранит только один текущий `ЭтотОбъект.ИДЗапроса`.
- В callback-обработке `result/error` используется текущее значение этого поля (`:269`, `:276`), а не идентификатор конкретного запроса из callback-пакета.
- Если прилетят два параллельных `lyra_ask_naparnik`, второй запрос перетрёт `request_id` первого, и результат уйдёт не в тот `tool_call`.

### [P2] Ошибка ответа не завершает текущую сессию стриминга и оставляет чат в зависшем состоянии
- В `Forms/МодульЧат/Ext/Form/Module.bsl:392-405` обработчик `error` показывает сообщение, но не вызывает `СброситьСостояние()` и не посылает событие завершения, в отличие от `assistant_end` (`:364-387`).
- Из-за этого `Активен()` продолжает возвращать `Истина` (`:250-251`), а следующая отправка уходит в ветку «прервать текущий стрим и отправить новый» (`:92-114`), где результат отправки вообще не проверяется.
- На практике это даёт зависший индикатор подготовки, несинхронное состояние UI и риск потери следующего сообщения после любой протокольной ошибки.

### [P2] Кнопка «Проверить токен Напарника» обращается к неэкспортируемой переменной главной формы
- `Forms/Настройки/Ext/Form/Module.bsl:49` вызывает `ФормаВладелец.МодульНапарник.ПроверитьДоступность(...)`.
- Но в `Forms/Форма/Ext/Form/Module.bsl:23` `МодульНапарник` объявлен как обычная переменная модуля формы, а экспортируется только `ОбщиеПараметры` (`:25`).
- Это создаёт риск runtime-ошибки именно по кнопке проверки токена, даже если основная инициализация формы работает корректно.

### [P2] Сценарии «отправлять нельзя» фактически не блокируют отправку
- `Forms/МодульСессия/Ext/Form/Module.bsl:301-302`, `:339-340`, `:386-389` используют событие `занят` как механизм блокировки ввода до авторизации и при нулевом балансе.
- Но в главной форме `занят` меняет только заголовок кнопки (`Forms/Форма/Ext/Form/Module.bsl:271-273`), а реальная доступность переключается другим событием `доступность_ввода` (`:232-234`).
- В результате после `hello_ack` без `auto_auth`, при `insufficient_balance` и при `balance <= 0` пользователь всё ещё может отправлять запросы.

### [P3] При первом открытии чат всегда стартует со светлым фоном, даже если включена тёмная тема
- В `Forms/МодульЧат/Ext/Form/Module.bsl:726-729` при `ЕХТ_Чат_ПриОткрытии()` фон жёстко инициализируется `КартинкаФонаСветлая`.
- При этом пользовательская настройка тёмной темы существует и применяется только в обработчике ручного переключения (`Forms/Форма/Ext/Form/Module.bsl:158-165`).
- Итог: после открытия/переинициализации виджет визуально стартует не в той теме, пока пользователь не переключит её заново.

### [P3] Проверка доступности 1С:Напарника создаёт реальные conversation-сессии и засоряет backend
- Комментарий `Forms/МодульНапарник/Ext/Form/Module.bsl:126-128` обещает лёгкую проверку доступности, но код на `:150-160` делает `POST /chat_api/v1/conversations/`.
- То есть каждая проверка токена создаёт новую conversation, которая далее нигде не используется и не закрывается.
- Это не ломает UI сразу, но создаёт лишние объекты и нагрузку на backend при каждой проверке/сохранении настроек.

## Residual Risks
- `lyra_data_eval` и особенно `lyra_data_exec` по-прежнему исполняют произвольное выражение/код из tool-call. Даже с `ОтменитьТранзакцию()` это не изолирует внешние побочные эффекты.
- В `МодульСессия` обработка `presence_leave` учитывает только активный стрим. Если backend исчез в idle-состоянии, пользователь узнает об этом только по следующему сбою отправки.

## Testing Gaps
- Не запускал живую проверку в 1C/web-клиенте.
- После исправлений я бы прогнал минимум:
  1. `lyra_data_query` на запросе, который гарантированно возвращает строки.
  2. Ошибку протокола во время подготовки ответа и повторную отправку после неё.
  3. Два параллельных `lyra_ask_naparnik` с разными `request_id`.
  4. Кнопку проверки токена в форме настроек.
  5. Первый запуск формы с уже включённой тёмной темой.
