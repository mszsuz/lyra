# Аудит соответствия `Lyra-Chat-multiform.epf`

Дата аудита: 2026-03-09

Проверенные материалы:
- `README.md`
- `ARCHITECTURE.md`
- `BACKLOG.md`
- `CLAUDE.md`
- `Общие архитектурные приемы при разработке внешних обработок.md`
- XML/BSL-исходники из `Lyra-Chat-multiform.epf/src`

Итоговая оценка:
Обработка соответствует требованиям частично. Базовая модульная архитектура, стриминг, JSON-парсинг и transport/MCP-каркас действительно реализованы, но есть несколько существенных расхождений между описанными требованиями и фактической реализацией. Самые критичные проблемы затрагивают закрытие формы, работоспособность `v8_query`, настройки подключения и безопасность MCP-инструментов.

## Ключевые замечания

### 1. Высокий риск: штатное закрытие формы, вероятно, сломано

Требование:
- модульная обработка должна корректно завершать жизненный цикл формы и каскадно отключать модули (`ARCHITECTURE.md:51-73`, `README.md:48-50`)

Факт:
- в главной форме закрытие блокируется условием `Если НЕ МожноЗакрытьФорму Тогда` (`Forms/Форма/Ext/Form/Module.bsl:79-97`)
- реквизит `МожноЗакрытьФорму` объявлен в форме (`Forms/Форма/Ext/Form.xml:547-557`)
- по всему проекту нет ни одного присваивания `МожноЗакрытьФорму = ...`

Риск:
- при значении по умолчанию `Ложь` форма не сможет закрыться штатно
- код отправки `disconnect` и отключения WebSocket выполнится внутри отмененного закрытия, но повторного `Закрыть()` после завершения очистки нигде нет

Рекомендация:
- либо инициализировать `МожноЗакрытьФорму = Истина` и закрывать форму штатно
- либо реализовать двухфазное закрытие: отмена первого закрытия, отключение транспорта, установка `МожноЗакрытьФорму = Истина`, повторный вызов `Закрыть()`

### 2. Высокий риск: `v8_query` содержит ошибку обхода колонок и, вероятно, падает на непустом результате

Требование:
- MCP-инструмент `v8_query` должен выполнять запрос и возвращать строки результата (`Forms/МодульMCP/Ext/Form/Module.bsl:140-150`)

Факт:
- список колонок собирается в массив `Колонки` (`Forms/МодульMCP/Ext/Form/Module.bsl:279-282`)
- затем строки результата заполняются циклом `Для Каждого ИмяКолонки Из Колонки Цикл` вместо обхода массива `Колонки` (`Forms/МодульMCP/Ext/Form/Module.bsl:288-293`)

Риск:
- код использует не ту переменную в цикле
- для непустого результата это с высокой вероятностью приведет к ошибке времени выполнения или некорректной сериализации строк

Рекомендация:
- исправить цикл на обход массива `Колонки`
- отдельно покрыть `v8_query` хотя бы минимальным smoke-тестом на результат с 1-2 колонками

### 3. Высокий риск: заявленные настройки подключения фактически не работают

Требование:
- реализована `ФормаПодключение` с URL, моделью и системным промптом, открываемая через `ОткрытьФорму(...)` и возвращающая настройки обратно в главную форму (`ARCHITECTURE.md:884-907`)
- конфигурируемые параметры не должны быть захардкожены (`ARCHITECTURE.md:568-581`)

Факт:
- форма настроек действительно существует (`Forms/ФормаПодключение/Ext/Form.xml:1-167`, `Forms/ФормаПодключение/Ext/Form/Module.bsl:1-33`)
- главная форма подключается по жестко заданному адресу `ws://localhost:11000/connection/websocket` (`Forms/Форма/Ext/Form/Module.bsl:119-125`)
- в главной форме нет вызова `ОткрытьФорму(...ФормаПодключение...)`
- обработчика `ОбработкаЗакрытияНастроек`, описанного в архитектуре, в главной форме нет
- в `chat`-сообщение отправляются только `type`, `form_id`, `content` (`Forms/Форма/Ext/Form/Module.bsl:158-162`)
- в `hello` отправляются только `form_id`, данные базы и `tools` (`Forms/Форма/Ext/Form/Module.bsl:692-708`)
- `Модель` отображается на главной форме и сохраняется в настройках (`Forms/Форма/Ext/Form.xml:107-139`, `Forms/Форма/Ext/Form.xml:529-545`), но в транспорт не передается
- `СистемныйПромпт` на главной форме вообще отсутствует

Риск:
- пользователь не может применить настройки, описанные в документации
- выбор модели и системный промпт не влияют на поведение обработки
- документация создает ложное ожидание готового UX-сценария

Рекомендация:
- реально открыть `ФормаПодключение` из главной формы
- хранить URL/модель/промпт в реквизитах главной формы и/или `ХранилищеОбщихНастроек`
- передавать выбранную модель и системный промпт в `hello`/`chat` по согласованному контракту

### 4. Высокий риск: контракт MCP-инструментов не совпадает с их реальной реализацией

Требование:
- `tools/list` должен описывать реальный контракт инструментов

Факт:
- `v8_query` объявляет параметр `params` (`Forms/МодульMCP/Ext/Form/Module.bsl:144-150`), но серверный обработчик использует только `query` (`Forms/МодульMCP/Ext/Form/Module.bsl:264-298`)
- `v8_metadata` объявлен как инструмент получения дерева/ветки по `path` (`Forms/МодульMCP/Ext/Form/Module.bsl:159-164`), но обработчик читает поле `type` и возвращает только плоские списки нескольких групп метаданных (`Forms/МодульMCP/Ext/Form/Module.bsl:309-359`)

Риск:
- клиент, который доверяет `tools/list`, будет формировать запросы по неверной схеме
- часть вызовов будет молча игнорироваться или возвращать не тот формат данных, который ожидается

Рекомендация:
- синхронизировать `tools/list` и серверные обработчики
- либо поддержать `params` и `path`, либо изменить схемы инструментов на реально поддерживаемые поля

### 5. Высокий риск: небезопасное выполнение произвольного кода на сервере 1С

Требование:
- в backlog уже зафиксирована необходимость ограничить опасные вызовы и проработать безопасный режим работы MCP (`BACKLOG.md:70-90`)

Факт:
- `v8_eval` напрямую вызывает `Вычислить(Выражение)` (`Forms/МодульMCP/Ext/Form/Module.bsl:363-375`)
- `v8_exec` напрямую вызывает `Выполнить(Код)` (`Forms/МодульMCP/Ext/Form/Module.bsl:382-394`)
- фильтра опасных слов, белого списка операций, подтверждения пользователя или sandbox-режима нет

Риск:
- при компрометации роутера, ошибке промпта или злоупотреблении инструментами обработка получает полный удаленный RCE по встроенному языку 1С
- это особенно опасно для production-баз

Рекомендация:
- как минимум добавить предвалидацию опасных конструкций и режим подтверждения
- стратегически заменить `v8_exec` набором специализированных безопасных инструментов

### 6. Средний риск: нарушено требование о динамическом имени обработки

Требование:
- путь к формам должен строиться через `Метаданные().ПолноеИмя()`, без хардкода имени внешней обработки (`ARCHITECTURE.md:914-923`, `README.md:44`)

Факт:
- `ObjectModule.ПолноеИмя()` реализован корректно (`Ext/ObjectModule.bsl:12-13`)
- однако формы-модули загружаются по жестко прописанным строкам `ВнешняяОбработка.Lyra_Chat_multiform...` (`Forms/Форма/Ext/Form/Module.bsl:57-70`)

Риск:
- переименование обработки или изменение технического имени сломает загрузку модулей
- архитектурное требование формально выполнено только частично

Рекомендация:
- строить путь к формам динамически через `ПолноеИмя()`/`ПолучитьПутьКФормам()`

### 7. Средний риск: `ObjectModule` и документация по архитектуре заметно разошлись

Требование:
- `ObjectModule` заявлен как серверный контейнер состояния и бизнес-логики с областями `Метаданные`, `Настройки`, `JSON`, `ЖурналРегистрации` (`ARCHITECTURE.md:722-741`, `Общие архитектурные приемы...:338-372`, `:1077-1188`)

Факт:
- реальный `ObjectModule` содержит не только метаданные и настройки, но и большой порт `ЕХТ_*`-утилит (`Ext/ObjectModule.bsl:125-381`)
- JSON-утилиты и ЖР-обертки, перечисленные как обязательные в архитектуре, отсутствуют
- `НастройкиЗагрузить/НастройкиСохранить` реализованы (`Ext/ObjectModule.bsl:33-52`), но по коду проекта не используются

Риск:
- документация больше не отражает реальное распределение обязанностей
- сложнее сопровождать серверную часть и понять, где должен жить общий код

Рекомендация:
- либо привести `ObjectModule` к описанной структуре
- либо обновить архитектурную документацию под фактическую реализацию

### 8. Низкий риск: документация о составе обработки устарела

Факт:
- README описывает 5 форм и 2 макета, а корневой XML включает 6 форм (`Форма`, `МодульBridge`, `МодульПарсер`, `МодульMCP`, `ФормаПодключение`, `МодульОбщий`) и только 1 подключенный шаблон (`README.md:40-45`, `Lyra-Chat-multiform.xml:27-33`)
- фактические размеры модулей сильно отличаются от README:
  - `ObjectModule.bsl`: 345 строк
  - `Форма/Module.bsl`: 1080 строк
  - `МодульBridge`: 393 строки
  - `МодульMCP`: 453 строки
  - `МодульОбщий`: 399 строк

Риск:
- документация затрудняет реальный аудит и вводит в заблуждение по составу сборки
- в репозитории лежит лишний шаблон `Templates/ДвоичныеДанныеПриложения.xml`, который не подключен в корневом XML

Рекомендация:
- синхронизировать README/ARCHITECTURE с текущим состоянием дерева `src`
- убрать неиспользуемые шаблоны или явно описать их назначение

## Что соответствует требованиям

Следующие части реализации выглядят соответствующими заявленной архитектуре:
- разбиение на отдельные формы-модули `МодульBridge`, `МодульПарсер`, `МодульMCP`, `ФормаПодключение`, `МодульОбщий`
- поддержка стриминга `thinking`/`text` и обновления текущего сообщения в реальном времени (`Forms/Форма/Ext/Form/Module.bsl:330-386`, `Forms/Форма/Ext/Form/Module.bsl:518-557`)
- JSON-парсинг через `ПрочитатьЗначениеJSON` с fallback на серверный `ПрочитатьJSON` (`Forms/МодульПарсер/Ext/Form/Module.bsl:18-31`, `Forms/МодульПарсер/Ext/Form/Module.bsl:249-269`)
- поддержка multi-JSON, ping/pong и `push.pub.data` в транспортном модуле (`Forms/МодульBridge/Ext/Form/Module.bsl:219-285`)
- поддержка `mcp_jsonrpc` и обратной отправки JSON-RPC-ответа в канал сессии (`Forms/Форма/Ext/Form/Module.bsl:442-458`)

## Ограничения аудита

Не выполнялось:
- реальная сборка EPF
- запуск в клиенте 1С
- проверка WebSocket/Centrifugo на живом стенде
- прогон MCP-инструментов на базе

Выводы отчета основаны на статическом анализе XML/BSL и сравнении с описанными требованиями в документации проекта.
