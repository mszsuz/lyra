# Архитектура Lyra-Chat-multiform.epf

> Модульная версия обработки Lyra-Chat. Монолитная форма (1521 строка) разнесена по служебным формам-модулям.
> Стриминг размышлений (thinking) и ответов Claude реализован в реальном времени.
> Паттерны взяты из анализа трёх production-обработок: СБИС (131 форма), Диадок (56 форм), Vanessa (108 форм).

## 1. Целевая архитектура

### Исходная версия (Lyra-Chat.epf)

```
Lyra-Chat.epf
├── Форма (1521 строка — god-object)
│   ├── UI-обработчики
│   ├── WebSocket/Bridge
│   ├── Парсинг JSON (строковый, хрупкий)
│   ├── MCP-инструменты (цепочка Если/ИначеЕсли)
│   ├── Вывод в чат
│   └── Копипаста ЕХТ_Чат (440 строк)
├── ФормаПодключение (пустая)
├── ObjectModule (пустой)
└── 2 макета
```

### Реализованная архитектура (Lyra-Chat-multiform.epf)

```
Lyra-Chat-multiform.epf
├── Форма (главная, 936 строк)
│   ├── Тонкий контроллер — UI-события и координация модулей
│   ├── ОбработатьСобытиеМодуля() — центральный диспетчер событий
│   ├── Стриминг: thinking/text дельты с обновлением пузыря в реальном времени
│   ├── Обработчики настроек чата (аватары, автосвертка, тёмная тема)
│   └── Встроенный код фреймворка ЕХТ_Чат (без изменений)
├── МодульBridge (174 строки) — WebSocket-соединение
├── МодульПарсер (279 строк) — JSON через ЧтениеJSON в Соответствие
├── МодульMCP (537 строк) — Strategy-dispatch + серверные обработчики
├── ФормаПодключение (48 строк) — UI настроек (URL, модель, промпт)
├── ObjectModule (54 строки) — обёртки настроек, ПолноеИмя()
└── 2 макета (HTML/JS чата, копия из оригинала)
```

**Источник паттерна:** Vanessa — главная форма хранит ссылки на модули в `Перем`, инициализирует их в `ПриОткрытии`, каскадно отключает в `ПередЗакрытием`.

**Отличие от плана:** МодульЧат не создан — код ЕХТ_Чат и стриминг остались в главной форме. Причина: ЕХТ_Чат тесно привязан к элементам формы (`Чат1`) и плохо выносится в отдельный модуль без переписывания фреймворка. Стриминг thinking/text реализован прямо в главной форме через `ОбновитьТекущееСообщение()`.

## 2. Разбиение на модули

### 2.1 Реализованное разбиение

| Модуль | Строк | Что содержит | Статус |
|---|---|---|---|
| **МодульBridge** | 174 | WebSocket-соединение, отправка, закрытие, обработчики WS_* | ✅ Реализован |
| **МодульПарсер** | 279 | `РазобратьJSON`, `ИзвлечьТекстДельты`, `ИзвлечьТипВнутреннегоСобытия`, `ИзвлечьТипБлока`, `СобратьJSON`, `ЭкранироватьJSON` | ✅ Реализован |
| **МодульMCP** | 537 | Strategy-dispatch, серверные обработчики `1c_query/metadata/eval/exec` | ✅ Реализован |
| **ФормаПодключение** | 48 | URL Bridge, выбор модели, системный промпт | ✅ Реализован |
| **МодульЧат** | — | Вывод в чат | ❌ Не создан (ЕХТ_Чат и стриминг остались в главной форме) |

**Взаимодействие модулей:**

```
Форма (контроллер)
  ├── ПолучитьФорму("МодульBridge")   → Bridge
  ├── ПолучитьФорму("МодульПарсер")   → Парсер
  ├── ПолучитьФорму("МодульMCP")      → MCP
  └── ОткрытьФорму("ФормаПодключение") → UI настроек
```

**Поток данных:**

```
1. Пользователь вводит текст
   → Форма.Чат1_ОтправитьСообщение
   → МодульПарсер.СобратьJSON (формирование {"type":"chat","content":"..."})
   → МодульBridge.ОтправитьСообщение (WebSocket)

2. Bridge отвечает
   → МодульBridge.WS_ПриПолученииСообщения
   → Владелец.ОбработатьСобытиеМодуля("Bridge", "bridge_message", JSON)

3. Форма обрабатывает ответ
   → МодульПарсер.РазобратьJSON (ЧтениеJSON → Соответствие)
   → обработка по типу события (session, stream_event, result, error, mcp_request...)

4. MCP-запрос
   → МодульMCP.ОбработатьЗапрос (strategy-dispatch по имени инструмента)
   → серверный обработчик (запрос/метаданные/вычисление/выполнение)
   → МодульBridge.ОтправитьСообщение (JSON-ответ)
```

### 2.2 Контракт модуля (из Vanessa)

Каждый модуль реализует минимальный контракт:

```1c
// Обязательно:
&НаКлиенте
Процедура Инициализация(Владелец) Экспорт    // DI — явная передача зависимости
&НаКлиенте
Функция ОтключениеМодуля() Экспорт

// Ссылка на владельца:
&НаКлиенте
Перем Владелец;  // главная форма, через которую модуль отправляет события
```

**Источник паттерна:** Vanessa — `ИнициализацияФормы(ВладелецФормы)` / `ОтключениеМодуля()` у каждого модуля.

**DI вместо Service Locator:** Зависимости передаются явно через `Инициализация(Владелец)`, а не через глобальный кеш. Это даёт понятный контракт модуля и отсутствие скрытых связей.

**Обратная связь через события:** Модули вызывают `Владелец.ОбработатьСобытиеМодуля("Bridge", "bridge_message", Данные)` — экспортный метод главной формы. Главная форма — единственный диспетчер событий.

**ВАЖНО: все экспортные методы — `&НаКлиенте`.** Методы с `&НаКлиентеНаСервереБезКонтекста` недоступны через `ПолучитьФорму()` (см. раздел 5).

### 2.3 Главная форма (реализованная структура)

```1c
// Форма/Ext/Form/Module.bsl — 936 строк (контроллер + стриминг + ЕХТ_Чат)

// === Объявление модулей ===
&НаКлиенте
Перем МодульBridge;
&НаКлиенте
Перем МодульПарсер;
&НаКлиенте
Перем МодульMCP;

// === Переменные стриминга ===
&НаКлиенте
Перем ТекущееСообщениеИД;        // ИД пузыря в чате (для updateMessage)
&НаКлиенте
Перем ТекущееРазмышлениеТекст;   // накопленный текст размышлений
&НаКлиенте
Перем ТекущийРежимБлока;         // "thinking" / "text" / ""

// === Инициализация ===
&НаСервере
Процедура ПриСозданииНаСервере(Отказ, СтандартнаяОбработка)
    ПутьКФормам = РеквизитФормыВЗначение("Объект").Метаданные().ПолноеИмя() + ".Форма.";
    Версия = РеквизитФормыВЗначение("Объект").ПолучитьВерсию();
    ЕХТ_Чат_ПриСозданииНаСервере(Отказ, СтандартнаяОбработка);
КонецПроцедуры

&НаКлиенте
Процедура ПриОткрытии(Отказ)
    // Инициализация модулей — DI через Инициализация(Владелец)
    МодульПарсер = ПолучитьФорму(ПутьКФормам + "МодульПарсер", , ЭтаФорма);
    МодульMCP    = ПолучитьФорму(ПутьКФормам + "МодульMCP",    , ЭтаФорма);
    МодульMCP.Инициализация(ЭтаФорма);
    МодульBridge = ПолучитьФорму(ПутьКФормам + "МодульBridge",  , ЭтаФорма);
    МодульBridge.Инициализация(ЭтаФорма);
КонецПроцедуры

// === Центральный диспетчер событий от модулей ===
&НаКлиенте
Процедура ОбработатьСобытиеМодуля(ИмяМодуля, Событие, Данные) Экспорт
    // Bridge → Парсер → обработка по типу → UI
    Если Событие = "bridge_message" Тогда
        ОбработатьСообщениеBridge(Данные);
    ИначеЕсли Событие = "bridge_connected" Тогда
        ВывестиВЧатИнформацию("Подключено к Bridge");
    // ... другие события ...
    КонецЕсли;
КонецПроцедуры

// === Каскадное отключение ===
&НаКлиенте
Процедура ПередЗакрытием(Отказ, ЗавершениеРаботы)
    Если ЗавершениеРаботы Тогда Возврат; КонецЕсли;
    Если МодульBridge <> Неопределено Тогда
        МодульBridge.Отключиться();
    КонецЕсли;
КонецПроцедуры
```


**Реализованный API (174 строки):**
- `Инициализация(Владелец)` — сохранение ссылки на главную форму
- `Подключиться(URL, SessionID, Модель)` — создание WebSocket с заголовками
- `Отключиться()` — закрытие соединения
- `ОтправитьСообщение(ТекстJSON)` — отправка через WebSocket
- `WS_ПриОткрытии/ПриЗакрытии/ПриОшибке/ПриПолученииСообщения` — обработчики
- Обратная связь через `Владелец.ОбработатьСобытиеМодуля("Bridge", событие, данные)`

### 3.2 МодульПарсер

МодульПарсер — **stateless-модуль** (без `Инициализация`/`ОтключениеМодуля`). Не хранит ссылку на владельца, содержит только чистые функции для работы с JSON. Получается через `ПолучитьФорму()` и вызывается напрямую.

```1c
// МодульПарсер/Ext/Form/Module.bsl — 279 строк

// --- Публичный API (все функции &НаКлиенте Экспорт) ---

// Парсинг JSON через ЧтениеJSON → Соответствие
Функция РазобратьJSON(СтрокаJSON) Экспорт
    // Делегирует на сервер: ЧтениеJSON + ПрочитатьJSON(, Истина)
    // Возвращает Соответствие (поддержка ключей с дефисами)

// Извлечение текста дельты из stream_event
// Поддерживает thinking_delta (поле "thinking") и text_delta (поле "text")
Функция ИзвлечьТекстДельты(Данные) Экспорт

// Тип события верхнего уровня: "session", "stream_event", "result", "error"...
Функция ИзвлечьТипСобытия(Данные) Экспорт

// Тип внутреннего события stream_event:
// "content_block_start", "content_block_delta", "content_block_stop"
Функция ИзвлечьТипВнутреннегоСобытия(Данные) Экспорт

// Тип блока контента из content_block_start:
// "thinking", "text", "tool_use"
Функция ИзвлечьТипБлока(Данные) Экспорт

// Универсальный доступ к полям верхнего уровня
Функция ИзвлечьЗначение(Данные, ИмяПоля) Экспорт

// Сериализация через ЗаписьJSON
Функция СобратьJSON(Значение) Экспорт

// Экранирование спецсимволов для вставки в JSON
Функция ЭкранироватьJSON(Текст) Экспорт

// Конвертация 1С-значений в JSON-совместимые типы
Функция ЗначениеДляJSON(ИсходноеЗначение) Экспорт
```

**Ключевая деталь — поддержка thinking:**

```1c
// ИзвлечьТекстДельты проверяет delta.type:
ТипДельты = Дельта.Получить("type");
Если ТипДельты = "thinking_delta" Тогда
    Текст = Дельта.Получить("thinking");
ИначеЕсли ТипДельты = "text_delta" Тогда
    Текст = Дельта.Получить("text");
Иначе
    Возврат "";
КонецЕсли;
```

```1c
    ОбработчикиИнструментов.Вставить("tools.metadata.get",    "Инструмент_Metadata");
    ОбработчикиИнструментов.Вставить("tools.code.eval",       "Инструмент_Eval");
    ОбработчикиИнструментов.Вставить("tools.code.exec",       "Инструмент_Exec");
КонецПроцедуры

&НаКлиенте
Функция ОтключениеМодуля() Экспорт
    ГлавнаяФорма = Неопределено;

КонецФункции

// --- Публичный API ---

&НаКлиенте
Процедура ОбработатьMCPЗапрос(Соединение, ДанныеЗапроса) Экспорт
    Метод = ДанныеЗапроса["method"];
    ИДЗапроса = ДанныеЗапроса["request_id"];
    Параметры = ДанныеЗапроса["params"];

    // Strategy-dispatch по реестру (паттерн СБИС)
    ИмяОбработчика = ОбработчикиИнструментов.Получить(Метод);
    Если ИмяОбработчика = Неопределено Тогда
        ОтправитьMCPОшибку(Соединение, ИДЗапроса, "Неизвестный метод: " + Метод);
        Возврат;
    КонецЕсли;

    // Вызов конкретного обработчика
    Результат = Неопределено;
    Попытка
        // Каждый инструмент — отдельная процедура
        Если ИмяОбработчика = "Инструмент_Query" Тогда
            Результат = Инструмент_Query(Параметры);
        ИначеЕсли ИмяОбработчика = "Инструмент_Metadata" Тогда
            Результат = Инструмент_Metadata(Параметры);
        // ... аналогично для остальных
        КонецЕсли;
    Исключение
        ОтправитьMCPОшибку(Соединение, ИДЗапроса, ОписаниеОшибки());
        Возврат;
    КонецПопытки;

    ОтправитьMCPРезультат(Соединение, ИДЗапроса, Результат);
КонецПроцедуры

// --- Серверные инструменты ---

&НаСервере
Функция Инструмент_Query(Параметры)
    // ... ВыполнитьЗапросНаСервере ...
КонецФункции
```

**Реализованный API (335 строк):**
- `Инициализация()` — регистрация обработчиков в `Соответствие`
- `ЗарегистрироватьОбработчик(Имя, ИмяМетода)` — расширяемый реестр
- `ОбработатьЗапрос(Данные)` — диспетч по имени инструмента
- `СформироватьОтвет(RequestId, Результат, Ошибка)` — единообразный JSON-ответ
- Обработчики (серверные): `Обработчик_1c_query`, `Обработчик_1c_metadata`, `Обработчик_1c_eval`, `Обработчик_1c_exec`

### 3.4 Стриминг (реализован в главной форме)

> **Статус:** ✅ реализован. Стриминг thinking и text дельт реализован непосредственно в главной форме, т.к. ЕХТ_Чат тесно привязан к элементам формы (`Чат1`).

**Переменные стриминга** (в секции `ПеременныеМодуля` главной формы):

```1c
&НаКлиенте
Перем ТекущееСообщениеИД;        // ИД пузыря в чате (для updateMessage)
&НаКлиенте
Перем ТекущееРазмышлениеТекст;   // накопленный текст размышлений
&НаКлиенте
Перем ТекущийРежимБлока;         // "thinking" / "text" / ""
```

**Протокол stream-событий Claude:**

```
content_block_start  {content_block.type: "thinking"}
content_block_delta  {delta.type: "thinking_delta", delta.thinking: "..."}  × N
content_block_stop
content_block_start  {content_block.type: "text"}
content_block_delta  {delta.type: "text_delta", delta.text: "..."}  × M
content_block_stop
```

**Обработка в `ОбработатьОтветBridge` (stream_event):**

```1c
ТипВнутренний = МодульПарсер.ИзвлечьТипВнутреннегоСобытия(ДанныеОтвета);

Если ТипВнутренний = "content_block_start" Тогда
    ТипБлока = МодульПарсер.ИзвлечьТипБлока(ДанныеОтвета);
    Если ТипБлока = "thinking" Тогда
        ТекущийРежимБлока = "thinking";
        // Создать пузырь "Размышляю..."
        Сообщение = ЕХТ_Чат_НовоеСообщение("Размышляю...", "Ассистент");
        ТекущееСообщениеИД = Сообщение.ИД;
        ЕХТ_Чат_ДобавитьСообщение(Элементы.Чат1, Сообщение);
    ИначеЕсли ТипБлока = "text" Тогда
        ТекущийРежимБлока = "text";
        // Переиспользовать пузырь thinking или создать новый
    КонецЕсли;

ИначеЕсли ТипВнутренний = "content_block_delta" Тогда
    ТекстДельты = МодульПарсер.ИзвлечьТекстДельты(ДанныеОтвета);
    Если ТекущийРежимБлока = "thinking" Тогда
        ТекущееРазмышлениеТекст += ТекстДельты;
        ОбновитьТекущееСообщение("Размышляю..." + ПС + ПС + ТекущееРазмышлениеТекст);
    ИначеЕсли ТекущийРежимБлока = "text" Тогда
        ТекущееСообщениеТекст += ТекстДельты;
        ОбновитьТекущееСообщение(ТекущееСообщениеТекст);
    КонецЕсли;
КонецЕсли;
```

**Вспомогательная процедура `ОбновитьТекущееСообщение`:**

```1c
&НаКлиенте
Процедура ОбновитьТекущееСообщение(НовыйТекст)
    Если ПустаяСтрока(ТекущееСообщениеИД) Тогда Возврат; КонецЕсли;
    Сообщение = Новый Структура;
    Сообщение.Вставить("ИД", ТекущееСообщениеИД);
    Сообщение.Вставить("Текст", НовыйТекст);
    ЕХТ_Чат_ОбновитьСообщение(Элементы.Чат1, Сообщение);
КонецПроцедуры
```

**Визуальное поведение:**

1. **Thinking фаза**: пузырь ассистента с текстом `Размышляю...\n\n<текст размышлений>` — обновляется в реальном времени
2. **Text фаза**: тот же пузырь заменяется на `<текст ответа>` — стримится посимвольно
3. **Result**: пузырь финализируется полным текстом, переменные сбрасываются
4. **Без thinking (Haiku)**: пустой пузырь создаётся при `content_block_start(text)`, далее стриминг текста

## 5. Ограничения платформы

Обнаружены при реализации Lyra-Chat-multiform.epf. Это не причины отказываться от паттерна форм-модулей — это ограничения, которые нужно учитывать.

### 5.1 `&НаКлиентеНаСервереБезКонтекста` невидима через `ПолучитьФорму()`

Методы с директивой `&НаКлиентеНаСервереБезКонтекста` **недоступны** при вызове через `ПолучитьФорму()`. Платформа не находит такие методы: ошибка `Метод объекта не обнаружен (ИмяМетода)`.

Через `ПолучитьФорму()` видны только методы с директивами `&НаКлиенте` и `&НаСервере`.

**Правило:** все экспортные методы форм-модулей, вызываемые из других форм, должны быть `&НаКлиенте`. Директива `&НаКлиентеНаСервереБезКонтекста` работает только для вызовов **внутри той же формы**.

```1c
// НЕПРАВИЛЬНО — не будет видна через ПолучитьФорму():
&НаКлиентеНаСервереБезКонтекста
Функция РазобратьJSON(СтрокаJSON) Экспорт  // ← ошибка при вызове из другой формы

// ПРАВИЛЬНО:
&НаКлиенте
Функция РазобратьJSON(Знач СтрокаJSON) Экспорт  // ← работает
```

**Совет:** если метод изначально был `&НаКлиентеНаСервереБезКонтекста`, можно оставить комментарий для документирования:
```1c
//*-+ &НаКлиентеНаСервереБезКонтекста
&НаКлиенте
Функция РазобратьJSON(Знач СтрокаJSON) Экспорт
```

### 5.2 ISP-нарушение форм-модулей

Форма-модуль формально нарушает принцип разделения интерфейсов (Interface Segregation Principle) — вызывающий код получает весь интерфейс `УправляемаяФорма` (сотни свойств и методов), хотя использует лишь 3–5 экспортных методов модуля.

**Однако в EPF это единственный механизм модуляризации**, и при соблюдении конвенций (минимальный экспортный интерфейс, документирование публичных методов) он надёжно работает. Альтернативы — макеты с `Выполнить()` — значительно хуже: теряется автодополнение, статический анализ и навигация по коду.

### 5.3 Клиент-серверный «пинг-понг»

При вызове `&НаСервере`-метода формы-модуля через `ПолучитьФорму()` происходит дополнительный клиент-серверный переход. Если главная форма уже на сервере и вызывает серверный метод модуля — получается цепочка клиент → сервер (главная) → клиент → сервер (модуль).

**Решение:** для межформенных вызовов использовать `&НаКлиенте`-обёртки, а серверную логику инкапсулировать внутри формы-модуля. Так сделано в МодульMCP: экспортные методы `&НаКлиенте`, серверные обработчики — внутренние `&НаСервере`.

## 16. Сборка EPF

### 16.1 Конфигурация `.1c-devbase.bat`

Файл `.1c-devbase.bat` в корне проекта `Chat/` задаёт путь к платформе и базе для сборки:

```bat
@echo off
set "ONEC_PATH=C:\Program Files\1cv8\current\bin\1cv8.exe"
set "ONEC_FILEBASE_PATH=C:\1ext.ru\tools\1C-Services\TEMP\lyra-build"
```

**Важно:**
- Используется **файловая временная база** (не серверная) — для сборки .epf серверная база не нужна
- Каждый проект/агент должен использовать **свою папку** в `TEMP/`, т.к. базой могут пользоваться несколько агентов
- Путь **не должен содержать пробелов**
- Симлинк `C:\Program Files\1cv8\current\` — не хардкодить версию платформы

### 16.2 Создание временной базы

```bash
cmd //c "C:\\1ext.ru\\tools\\1C-Services\\DB\\create-temp-base.bat C:\\1ext.ru\\tools\\1C-Services\\TEMP\\lyra-build"
```

Временная база создаётся один раз и переиспользуется для всех сборок.

### 16.3 Сборка через `/1c-batch`

```bash
# Из папки Chat/ (где лежит .1c-devbase.bat):
build-epf.bat <XML-исходник> <Результирующий-EPF>
```

**Путь результата** — EPF должен собираться в корень папки EPF:

```
Chat/
├── .1c-devbase.bat
├── Lyra-Chat-multiform.epf/
│   ├── Lyra-Chat-multiform.epf    ← сюда собирается готовый файл
│   └── src/                        ← исходники XML/BSL
│       └── Lyra-Chat-multiform/
```

Пример вызова:

```bash
cd "C:/WORKS/2026-01-31 Lyra/Chat"
build-epf.bat "Lyra-Chat-multiform.epf/src" "Lyra-Chat-multiform.epf/Lyra-Chat-multiform.epf"
```

### 16.4 Запуск в веб-клиенте

1. Убедиться что Bridge запущен: `node C:/WORKS/2026-01-31 Lyra/Bridge/bridge.js`
2. Открыть обработку через URL веб-клиента:

```
http://localhost:8300/?C="Execute=C:\WORKS\2026-01-31 Lyra\Chat\Lyra-Chat-multiform.epf\Lyra-Chat-multiform.epf"
```

3. Нажать «Подключиться» — Bridge подключится к Claude
4. Ввести сообщение — увидеть стриминг thinking → text

### 16.5 Возможные проблемы

| Проблема | Причина | Решение |
|----------|---------|---------|
| Кириллица `1С-Services` ломается в cmd | Путь содержит символ `С` (кириллица) | Использовать латинскую `C`: `1C-Services` |
| `build-epf.bat` не находит `.1c-devbase.bat` | Текущая директория не `Chat/` | `cd` в `Chat/` или использовать bat-обёртку |
| Пути с пробелами ломают аргументы | Git Bash разбивает аргументы по пробелам | Использовать bat-обёртку с хардкодом путей |
| Сервисная база не отвечает после сборки | Designer блокирует базу при сборке | `srv1c-status.bat`, при необходимости `srv1c-start.bat` |

## 17. Как использовать
