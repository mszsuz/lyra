# План: Рефакторинг Chat EPF — модульная архитектура

## Контекст

Главная форма Чата — god-object (1628 строк). Стриминг, транспорт, JSON, инструменты, UI — всё в одном месте. МодульПарсер существует отдельно от Bridge без причины, имена на английском, дублирование JSON-сериализации во всех модулях. Нужна чистая модульная архитектура для дальнейшего развития (МодульДесктоп, клиентские команды и т.д.).

**Было → Станет:**

## Фаза 4: МодульЧат — извлечение стриминга

**Цель:** вся логика диалога в отдельном модуле. Самая сложная фаза.

**Ключевое решение:** МодульЧат НЕ вызывает ЕХТ_Чат напрямую. Вычисляет что показать → отправляет UI-команды владельцу через `ОбработатьСобытиеМодуля("Чат", событие, данные)`.

**Переменные (были в Форме → теперь в МодульЧат):**
```
ТекущееСообщениеИД, ТекущееСообщениеТекст, ТекущийРежимБлока,
НачалоРазмышления, ДлительностьРазмышленияМс, НачалоПодготовкиОтвета,
СтатусыИнструментов, ВремяПоследнегоСигнала, ТаймаутОтветаСработал
```

**API МодульЧат:**
```bsl
Процедура Инициализация(ФормаВладелец, Транспорт, Обработка) Экспорт
Процедура ОбработатьСобытие(ТипСобытия, ДанныеОтвета) Экспорт
Процедура НовыйВопрос() Экспорт
Процедура Прервать() Экспорт
Функция Активен() Экспорт
```

**События от МодульЧат → Форма:**
- `("Чат", "добавить_сообщение", {Текст, Шаблон})` → ЕХТ_Чат_ДобавитьСообщение
- `("Чат", "обновить_сообщение", {ИД, Текст})` → ЕХТ_Чат_ОбновитьСообщение
- `("Чат", "завершён", Неопределено)` → УстановитьЗанят(Ложь)
- `("Чат", "tool_result", {Ответ})` → МодульТранспорт.ОтправитьСообщение
- `("Чат", "предупреждение", {Текст})` → ВывестиВЧатПредупреждение

**Переносим из Формы → МодульЧат:**
- `ОбработатьОтветBridge` (основная логика стриминга)
- HTML-билдеры: `СобратьHTMLПодготовки`, `СобратьСпойлерПодготовки`, `SVGПульсация`, `ФорматВремениПодготовки`
- Группировка статусов: `СгруппироватьСтатусы`, `СгруппироватьСтатусыИнструментов`
- Таймер: `ОбновитьТаймерПодготовки`
- Логика abort из `Чат1_ОтправитьСообщение`

**Переносим из МодульПарсер → МодульЧат:**
- `ИзвлечьТекстДельты`, `ИзвлечьТипСобытия`, `ИзвлечьТипВнутреннегоСобытия`, `ИзвлечьТипБлока`, `ИзвлечьЗначение`

**Маршрутизация в Форме упрощается:**
```bsl
// bridge_message handler:
ДанныеОтвета = МодульТранспорт.РазобратьJSON(Данные);
ТипСобытия = МодульТранспорт.ПолучитьПоле(ДанныеОтвета, "type");
Если ТипСобытия = "hello_ack" Тогда ОбработатьHelloAck(ДанныеОтвета);
ИначеЕсли ТипСобытия = "auth_ack" Тогда ОбработатьAuthAck(ДанныеОтвета);
ИначеЕсли ТипСобытия = "balance_update" Тогда ОбработатьBalanceUpdate(ДанныеОтвета);
Иначе МодульЧат.ОбработатьСобытие(ТипСобытия, ДанныеОтвета);
КонецЕсли;
```

**Проверка:** полный smoke-test: подключение → вопрос → thinking-анимация → стриминг текста → спойлер → assistant_end. Abort. Tool_call. Reconnect.

## Порядок работы с Конфигуратором

Я пишу BSL-код модулей. Пользователь создаёт формы в Конфигураторе (Form.xml), выгружает через dump-epf. Я не генерирую Form.xml с нуля — только BSL.

**Для каждой новой формы-модуля:**
1. Пользователь создаёт пустую форму в Конфигураторе
2. Пользователь выгружает через dump-epf
3. Я пишу Module.bsl
4. Сборка через build-epf
5. Тест в браузере

## Верификация после каждой фазы

1. `build-epf.bat` — сборка EPF
2. Открыть в 1С (веб-клиент `http://localhost:8300/`)
3. Подключиться к Centrifugo
4. Отправить сообщение → стриминг работает
5. Tool_call → результат приходит
6. Abort → "прервано" отображается
7. Reconnect → сессия восстанавливается
