### Безопасность: работа через GUID вместо строковых представлений (высокий приоритет)

Источник: анализ конкурента [msrv-tech AI Agent](../docs/competitors/msrv-tech--AI_agent/Анализ%20решения.md).

| # | Задача | Описание | Сложность |
|---|---|---|---|
| G1 | **Проработать концепцию GUID-ориентированного взаимодействия** | Агент не должен видеть персональные данные (ФИО, названия контрагентов). Входные параметры MCP-инструментов — GUID + тип объекта. 1С автоматически конвертирует GUID → ссылка перед выполнением, ссылка → GUID в ответах. Результаты `v8_query` — отдельный вопрос: агенту нужны представления для анализа данных, но они содержат ПДн. Варианты: маскирование, псевдонимы, whitelist полей. **Требуется анализ и RFC.** | Высокая |
| G2 | **Маскировка данных в Роутере (Privacy Layer)** | Слой маскировки персональных данных между tool_result и Claude. Числа (суммы, количества) остаются открытыми, текстовые значения (ФИО, ИНН, названия контрагентов) заменяются детерминированными псевдонимами (`Контрагент_a3f2b1c0`). Расшифровка — при отправке ответа пользователю в Чат. Детерминированность через HMAC-SHA256: одно значение → один псевдоним в рамках сессии. Правила маскировки: числовые строки с ведущими нулями (коды, серийники) маскируются, чистые числа — нет. Настраиваемый whitelist/blacklist полей. Референс: [alonehobo/1c-trusted-gateway](https://github.com/alonehobo/1c-trusted-gateway) (Go, аналогичная концепция). **Критично для enterprise-клиентов.** | Высокая |

### DSL vs безопасные MCP-инструменты — анализ подходов (средний приоритет)

Источник: анализ конкурента [msrv-tech AI Agent](../docs/competitors/msrv-tech--AI_agent/Анализ%20решения.md).

| # | Задача | Описание | Сложность |
|---|---|---|---|
| D1 | **Провести сравнительный анализ DSL vs MCP-инструменты для операций записи** | Два подхода к безопасному изменению данных: (1) JSON DSL — LLM генерирует сценарий целиком, 1С выполняет пакетом. Плюсы: атомарность, можно валидировать весь сценарий до выполнения. Минусы: нет обратной связи между шагами, дублирует логику LLM, сложный интерпретатор (~4000 LOC у конкурента). (2) Набор безопасных MCP-инструментов (v8_create_document, v8_set_field, v8_write, v8_post) — Claude вызывает по одному с обратной связью. Плюсы: стандарт MCP, обратная связь на каждом шаге, Claude сам обрабатывает ошибки. Минусы: больше round-trip, нет атомарности без явной транзакции. **Ключевой вопрос:** `v8_exec` небезопасен для продакшна — чем заменить? Белый список MCP-инструментов = DSL по сути, но через стандартный протокол. **Требуется RFC с плюсами/минусами и рекомендацией.** | Средняя |

## Обнаруженные ограничения платформы

Подробно описаны в [ARCHITECTURE.md, раздел 4](ARCHITECTURE.md#4-ограничения-платформы). Краткая сводка:

1. **`&НаКлиентеНаСервереБезКонтекста` невидима через `ПолучитьФорму()`** — все экспортные методы модулей должны быть `&НаКлиенте`
2. **ISP-нарушение** — форма-модуль экспонирует весь интерфейс `УправляемаяФорма`, но альтернатив в EPF нет
3. **Клиент-серверный пинг-понг** — серверные методы формы-модуля вызываются с дополнительным переходом, решается `&НаКлиенте`-обёртками
