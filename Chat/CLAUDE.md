# Lyra-Chat — внешняя обработка 1С

Общайся с пользователем на русском языке.

## Что это

Чат-клиент для 1С:Предприятие 8. Внешняя обработка (EPF) с HTML-интерфейсом на базе фреймворка ЕХТ_Чат. Подключается к Centrifugo по WebSocket, стримит ответы Claude в реальном времени.

Это работающий MVP — уже использовался для подключения к нескольким базам и помощи с разбором проблем.

## Требования к платформе

- **1С:Предприятие 8.3.27+** — требуется встроенный WebSocket-клиент (`WebSocketКлиентСоединения`)
- Тонкий клиент, управляемое приложение

## Архитектура

Модульная: главная форма (диспетчер) + служебные формы-модули.
Подробности: [ARCHITECTURE.md](ARCHITECTURE.md), [README.md](README.md)

### Модули

| Модуль | Назначение |
|--------|-----------|
| **Форма** (диспетчер) | Тонкий контроллер: маршрутизирует события между модулями, серверная инициализация ЕХТ_Чат |
| **МодульСессия** | Управление сессией: подключение, протокол (hello_ack, auth_ack, balance_update), настройки, QR, подключение/отключение |
| **МодульЧат** | Жизненный цикл диалога + ЕХТ_Чат: стриминг, отправка, tool_call, abort, вывод в чат, callback'и виджета |
| **МодульТранспорт** | Centrifugo клиент на BSL: connect/JWT, publish, ping/pong, мульти-JSON, push-конверт, hello/reconnect, watchdog, JSON-сериализация |
| **МодульОбщий** | Единое место для утилит, нужных нескольким модулям (JSON, форматирование) |
| **МодульОбработка** | Обработчики всех команд от модели — серверных (v8_query, meta_list) и клиентских (контекст, открытые формы, активные элементы) |

### Поток данных

```
1. Пользователь вводит текст
   → Форма → МодульЧат.ОтправитьСообщениеПользователя → МодульТранспорт.Publish

2. Ответ от сервера
   → Centrifugo push на канале сессии
   → МодульТранспорт разворачивает push-конверт
   → Форма (диспетчер) → МодульСессия.ОбработатьСобытиеТранспорта
   → МодульСессия маршрутизирует: системные → сам, стриминг → МодульЧат

3. Команда от модели (exec)
   → push на канале → МодульСессия → МодульЧат → МодульОбработка.ВыполнитьИнструмент → результат
   → МодульТранспорт.Publish (JSON-ответ обратно)
```

## Транспорт: Centrifugo (реализовано)

МодульТранспорт — мини-клиент Centrifugo на BSL: connect с JWT, publish, ping/pong, разбор мульти-JSON (`\n`), разворачивание push-конверта, JSON-сериализация. Реализация клиентского протокола Centrifugo непосредственно в коде BSL, используя `WebSocketКлиентСоединения`. Без внешних компонент или SDK. Subscribe на lobby не нужен (`allow_publish_for_client: true`), subscribe на канал сессии не нужен (авто-подписка через `channels` claim в JWT).

Общий JWT для lobby зашит в обработку. При hello Роутер возвращает персональный chat_jwt — Чат переподключается с ним.

**Два канала:**

| Канал | Назначение |
|-------|-----------|
| `session:lobby` (общий) | hello-рукопожатие. Общий JWT зашит в обработку |
| Канал сессии (персональный) | Создаётся Роутером. Доступ через chat_jwt (для Чата) / mobile_jwt (для мобильного) |

**JWT-схема:**

| JWT        | Откуда                    | Назначение                                                    |
|------------|---------------------------|---------------------------------------------------------------|
| Общий JWT  | Зашит в EPF               | Только для lobby — connect к Centrifugo, publish hello        |
| chat_jwt   | От Роутера (в hello_ack)  | Переподключение к Centrifugo для канала сессии (Чат). Содержит `channels: ["session:<session_id>"]`, срок жизни 1 год |
| mobile_jwt | От Роутера (в hello_ack)  | Кодируется в QR, мобильное подключается с ним к каналу сессии. Содержит `channels: ["session:<session_id>"]`, срок жизни 1 год |

**Протокол Centrifugo (JSON поверх WebSocket):**

```json
{"id": 1, "connect": {"token": "JWT..."}}
{"id": 2, "publish": {"channel": "session:lobby", "data": {"type": "hello", ...}}}
```

**Важно:** Centrifugo может отправлять **несколько JSON в одном WebSocket-фрейме**, разделённых `\n`. Мини-клиент BSL должен разбивать входящее сообщение по `\n` и парсить каждую строку отдельно.

**Флоу подключения (MVP с auto_auth):**

1. Обработка открывается — собирает данные о базе (конфигурация, версия, компьютер, строка подключения)
2. Подключается к Centrifugo по WebSocket (`ws://host:11000/connection/websocket`), отправляет `connect` с общим JWT (lobby)
3. Публикует `hello` с данными о базе в `session:lobby` (subscribe не нужен — `allow_publish_for_client: true`)
4. Роутер возвращает `hello_ack` с `chat_jwt`, `mobile_jwt` и `auto_auth: true`
5. Чат переподключается к Centrifugo с `chat_jwt` (персональный канал сессии)
6. При `auto_auth` — ввод сразу доступен, QR не показывается (MVP)
7. Канал сессии остаётся на всю сессию (нет переключения на другой канал)

**Флоу подключения (продакшен, будущее):**

Шаги 1–5 те же, затем:
6. Чат отображает QR-код = `mobile_jwt` (срок жизни 1 год), ввод заблокирован
7. Мобильное приложение сканирует QR — подключается к Centrifugo с `mobile_jwt` — отправляет `{user_id, device_id}`
8. Роутер завершает авторизацию — `auth_ack` → ввод разблокирован

**Переподключение** определяется по `form_id` (UUID формы, `ЭтотОбъект.УникальныйИдентификатор`). `form_id` включается во все сообщения от Чата (`hello`, `chat`). Если `form_id` известен Роутеру и сессия жива (TTL 30 мин) — `hello_ack` с `status: reconnected`, новый `chat_jwt`, без QR. Закрытие формы = новый `form_id` = новая сессия.

## Данные о базе (hello)

При открытии обработка собирает:

| Поле | Пример | Описание |
|------|--------|----------|
| `form_id` | `e5f6a7b8-...` | UUID формы (ЭтотОбъект.УникальныйИдентификатор) |
| `config_name` | `БухгалтерияПредприятия` | Метаданные.Имя |
| `config_version` | `3.0.191.41` | Метаданные.Версия |
| `config_id` | `2xxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` | Идентификатор конфигурации. MD5 от платформенного ПолучитьИдентификаторКонфигурации() без последней строки. У типовых на поддержке одной версии совпадает, у доработанных/снятых с поддержки — свой. Вычисляется в обработке, без зависимостей |
| `computer` | `BUHPC-01` | ИмяКомпьютера() |
| `connection_string` | `Srvr="srv1c";Ref="buh_prod";` | СтрокаСоединенияИнформационнойБазы() |
| `base_ids` | `{ssl_id, user_id, storage_id, connect_id}` | Идентификаторы базы для мульти-ID матчинга (см. ниже) |

### Идентификация базы (base_ids)

Чат собирает все доступные идентификаторы базы через `ПолучитьИдентификаторыБазы()` (ObjectModule.bsl). Роутер матчит по любому из них, приоритет: ssl_id > user_id > storage_id > connect_id.

| ID | Источник | Стабильность |
|----|----------|-------------|
| `ssl_id` | `Константы.ИдентификаторИнформационнойБазы` (БСП) | Высокая — привязан к базе |
| `user_id` | `ПользователиИнформационнойБазы.ТекущийПользователь().УникальныйИдентификатор` | Высокая — привязан к пользователю ИБ |
| `storage_id` | `ХранилищеОбщихНастроек`, ключ `lira_base_id` (UUID, per-user) | Средняя — создаётся при первом запуске |
| `connect_id` | MD5 от `ИмяКомпьютера() + "\|" + СтрокаСоединенияИнформационнойБазы()` | Низкая — зависит от способа подключения |

При матчинге Роутер мержит все новые ID в существующую запись (например, ssl_id появился после обновления БСП).

## exec-команды (MCP → 1С)

Когда Claude нужны данные из базы пользователя, Роутер отправляет exec-команду через Centrifugo в канал сессии. Чат получает команду, выполняет на сервере 1С, отправляет результат обратно.

MCP-инструменты:

| Инструмент | Назначение |
|---|---|
| `lyra_data_query` | Запрос на языке запросов 1С. Валидация через `СхемаЗапроса`, лимит 100 строк |
| `lyra_meta_list` | Список объектов метаданных (справочники, документы, регистры) |
| `lyra_meta_describe` | Структура объекта метаданных (реквизиты, табличные части, типы) |
| `lyra_data_read` | Все реквизиты объекта по ссылке (`_ref`) |
| `lyra_data_eval` | Вычисление одиночного выражения 1С |
| `lyra_data_exec` | Выполнение блока кода 1С в транзакции-песочнице. Возврат через переменную `Результат` |
| `lyra_nav_resolve` | Навигационная ссылка на объект (кликабельная) |
| `lyra_diag_read_log` | Журнал регистрации с фильтрами |

## Распространение

Обработка распространяется любым способом (файл EPF). Общий JWT для Centrifugo зашит в обработку — один для всех экземпляров, даёт доступ только к lobby. Роутер адресует конкретный экземпляр по `client` UUID из `pub.info.client` (приходит в push при publish). Авторизация пользователя — через мобильное приложение (QR).

## Ключевые файлы

- `Lyra-Chat-multiform.epf/src/` — исходники (XML + BSL)
- `Форма/Module.bsl` — главная форма (тонкий диспетчер ~470 строк)
- `МодульСессия/Module.bsl` — управление сессией: подключение, протокол, баланс, QR, настройки (~350 строк)
- `МодульЧат/Module.bsl` — жизненный цикл диалога + ЕХТ_Чат: стриминг, отправка, tool_call, abort, вывод в чат (~1230 строк)
- `МодульТранспорт/Module.bsl` — мини-клиент Centrifugo + JSON (~420 строк, бывш. МодульBridge + МодульПарсер)
- `МодульОбработка/Module.bsl` — MCP strategy-dispatch, серверные обработчики инструментов (~640 строк, бывш. МодульMCP)
- `МодульОбщий/Module.bsl` — общие утилиты для нескольких модулей (~300 строк)
- `ARCHITECTURE.md` — архитектура и паттерны
- `BACKLOG.md` — бэклог задач
