# Анализ конкурента: msrv-tech / AI Agent for 1C:Enterprise

## 1. Общее описание

**AI Agent for 1C:Enterprise** — автономный интеллектуальный агент для работы с данными 1С через естественный язык. Не просто чатбот, а **action-oriented агент**, способный самостоятельно планировать, выполнять и проверять действия в базе 1С.

**Слоган:** *"Превратите вашу 1С в интеллектуальную экосистему"*

## 2. Технологический стек

| Компонент | Технология |
|-----------|-----------|
| Язык | 1C BSL (основной) + Python (CLI/автоматизация) |
| LLM | OpenAI-совместимые провайдеры (Gitsell AI Proxy по умолчанию) |
| Развёртывание | Расширение 1С (.cfe) для платформы 8.3.26+ |
| Связь с LLM | Прямые HTTP-запросы из 1С (без внешнего моста) |
| Async | Фоновые задания 1С + серверные оповещения (8.3.26+) |

## 3. Архитектура

```
┌─────────────────────────────┐
│     UI (Управляемые формы)  │
│  ИИА_Агент │ ИИА_RAG │ ... │
└──────────────┬──────────────┘
               │
┌──────────────▼──────────────┐
│   Оркестратор (фон. задание)│
│  Планирование → Генерация → │
│  → Выполнение → Проверка    │
└──────────────┬──────────────┘
       ┌───────┼───────┐
       │       │       │
┌──────▼─┐ ┌──▼────┐ ┌▼──────────┐
│DSL     │ │RAG    │ │LLM API    │
│Движок  │ │Индекс │ │(HTTP)     │
│(JSON)  │ │(TF-IDF)│ │           │

```

**Ключевое отличие:** прямое HTTP-соединение с LLM из 1С, без внешнего Bridge/Node.js.

## 5. Основные возможности

### 5.1 Два режима работы
- **Запрос1С** — только чтение, безопасные запросы к данным
- **Агент** — полноценный агент с созданием/изменением документов

### 5.2 DSL (Domain Specific Language)
Вместо MCP используется собственный JSON DSL:
```json
{
  "steps": [
    {"action": "GetMetadata", "filter": "Контрагенты"},
    {"action": "RunQuery", "query": "ВЫБРАТЬ ... ИЗ ..."},
    {"action": "FindReferenceByName", "object_name": "Контрагенты", "name": "ООО Ромашка"},
    {"action": "CreateDocument", "object_name": "РеализацияТоваровУслуг"},
    {"action": "SetField", "field": "Сумма", "value": 10000},
    {"action": "Write"},
    {"action": "ShowInfo", "message": "Документ создан!"}
  ]
}
```

**Доступные действия:**
- Чтение: GetMetadata, GetObjectFields, CheckObjectExists, RunQuery, FindReferenceByName/GUID, SelectObject
- Запись: CreateReference, CreateDocument, SetField, Write
- Системные: ShowInfo, ForEach, SaveToStorage/LoadFromStorage

### 5.3 RAG (Retrieval-Augmented Generation)
Встроенная система семантического поиска по метаданным:
- TF-IDF индексация (без внешних vector DB)
- Чанкирование: заголовки, реквизиты, табличные части
- Стоп-слова + расширение синонимов
- Хранение в регистрах 1С (ИИА_Чанки, ИИА_ТокенИндекс)

### 5.4 Автономное планирование
- Агент создаёт **видимый план** перед выполнением
- План сохраняется в справочнике ИИА_Диалоги
- Шаги отмечаются по мере выполнения — полная прозрачность

### 5.5 State Summary (оптимизация контекста)
Вместо сырой истории сообщений — сжатое резюме:
- Последняя задача пользователя
- Последний DSL
- Результаты выполнения
- Текущий статус

### 5.6 Intelligent Error Recovery
- Обнаружение галлюцинаций (несуществующие объекты)
- Автоповтор с другими параметрами
- Детекция циклов (повторяющиеся CheckObjectExists)
- Чекпоинты во временном хранилище

### 5.7 Контроль токенов и стоимости
- Бюджет токенов на диалог/пользователя
- Хард-лимиты для контроля расходов
- Mock-режим для тестирования без API-вызовов

## 6. Безопасность

Три уровня защиты:
1. **Права пользователя** — работает под правами текущего пользователя 1С (RLS)
2. **Тип диалога** — режим "Запрос" блокирует все операции записи
3. **Пользовательские настройки** — индивидуальные ограничения в регистре ИИА_НастройкиПользователей

## 7. Тестирование

- **Бесплатные тесты** (без AI): парсинг DSL, доступ к метаданным, readonly-режим
- **AI-тесты**: полный цикл с реальным LLM
- **Mock-тесты**: предопределённые ответы для воспроизводимости
- **Python CLI**: run_tests.py, run_dialog.py, test_examples.py (с отчётами в Telegram)
- **Vanessa Automation**: UI-тесты (.feature файлы)

## 8. Python CLI инфраструктура

- **test_examples.py** — тестирование примеров из README через COM + отчёт в Telegram
- **run_dialog.py** — запуск диалога из CLI
- **build_extension_83.py** — автосборка .cfe
- **update_1c.py** — обновление конфигурации без UI
- **com_1c** — обёртка COM для произвольных запросов к 1С

# Сравнение с Lyra Chat

## Архитектурные отличия

| Аспект | msrv-tech (AI Agent) | Lyra Chat |
|--------|---------------------|-----------|
| **Подключение к LLM** | Прямой HTTP из 1С | WebSocket через Bridge (Node.js) |
| **Стриминг** | Нет (запрос-ответ через фон. задание) | Да, real-time дельты (thinking + text) |
| **Протокол инструментов** | Собственный JSON DSL | MCP 2025-11-25 (стандарт) |
| **Async** | Фоновые задания 1С | WebSocket (мгновенная связь) |
| **UI** | Стандартная управляемая форма | HTML/JS чат (ЕХТ_Чат фреймворк) |
| **Развёртывание** | .cfe расширение | .epf внешняя обработка |
| **Автономность** | Полностью автономный (план → выполнение) | Управляется Claude (MCP tool_use) |
| **Провайдер LLM** | OpenAI-совместимые (Gitsell) | Claude (Anthropic) |
| **Thinking** | Нет отображения | Спойлеры с таймером |

## Наши преимущества

1. **Стриминг в реальном времени** — пользователь видит ответ по мере генерации
2. **Thinking блоки** — визуализация хода мыслей AI со спойлерами и таймером
3. **MCP стандарт** — совместимость с экосистемой (Claude Code, IDE)
4. **Современный UI** — HTML/JS чат вместо управляемой формы
5. **Модульная архитектура** — DI паттерн (Bridge, Parser, MCP как отдельные модули)
6. **Claude** — более мощная модель для сложных задач
7. **Простота развёртывания** — .epf без модификации конфигурации

## Преимущества конкурента

1. **Автономный агент** — сам планирует, выполняет и проверяет
2. **Встроенный RAG** — семантический поиск по метаданным без внешних зависимостей
3. **DSL для записи** — безопасное создание документов через JSON
4. **Прозрачный план** — пользователь видит план до выполнения
5. **Error Recovery** — автовосстановление, детекция циклов, чекпоинты
6. **Контроль стоимости** — бюджет токенов на диалог/пользователя
7. **Без внешних зависимостей** — не нужен Bridge/Node.js
8. **State Summary** — оптимизация контекста вместо полной истории
9. **Режим только чтение** — явное разделение безопасных/опасных операций
10. **Аудит** — полный лог действий в регистре

# Что полезного можно добавить в Lyra Chat

## Приоритет 1 — Высокий (значительное улучшение)

### 1. MCP-инструменты для записи данных
**Что:** Инструменты `v8_create_document`, `v8_create_reference`, `v8_set_field`, `v8_write` для безопасного создания и модификации объектов через MCP.
**Зачем:** Сейчас есть `v8_query` (чтение), `v8_eval`, `v8_exec`. Специализированные инструменты записи дадут Claude структурированный и безопасный способ модифицировать данные — с валидацией на каждом шаге.
**Реализация:** Новые обработчики в МодульMCP с проверкой прав и режимом readonly.

### 2. Режим "Только чтение" (Safe Mode)
**Что:** Переключатель в UI, блокирующий все операции записи на уровне MCP-обработчика.
**Зачем:** Безопасность для новых пользователей и демонстраций. Сейчас `v8_exec` может выполнить любой код.
**Реализация:** Флаг в настройках формы → проверка в МодульMCP перед выполнением `v8_exec`/`v8_eval`/write-инструментов.

### 3. Автономный режим агента
**Что:** Режим, в котором Claude получает задачу и автономно выполняет цепочку: анализ → план → шаги → верификация.
**Зачем:** Сейчас Claude делает по одному MCP-вызову за раз. Автономный режим позволит решать сложные задачи одним запросом.
**Реализация:** Системный промпт с инструкцией планировать и выполнять шаги. Claude уже умеет multi-step tool_use — нужно правильно оформить промпт.

## Приоритет 2 — Средний (заметное улучшение)

### 4. Прозрачный план действий в UI
**Что:** Перед выполнением многошаговой задачи — показать план: "1. Найду контрагента → 2. Создам документ → 3. Заполню реквизиты".
**Зачем:** Доверие пользователя. Он видит что будет делать AI до того, как это произойдёт.
**Реализация:** Специальный блок в UI чата (чекбоксы с шагами, обновляющиеся по мере выполнения).

### 5. RAG по метаданным конфигурации
**Что:** Индексация метаданных (справочники, документы, реквизиты) для семантического поиска.
**Зачем:** Сейчас `v8_metadata` возвращает плоский список. RAG позволит Claude быстрее находить нужные объекты по смыслу.
**Реализация:** Индексация при первом запуске + MCP-инструмент `v8_search_metadata` с fuzzy-поиском.

### 6. State Summary (сжатие контекста)
**Что:** Вместо полной истории — сжатое резюме предыдущего контекста после N сообщений.
**Зачем:** Экономия токенов в длинных диалогах.
**Реализация:** Запросить у Claude резюме, использовать его вместо полной истории.

### 7. Контроль стоимости (Token Budget)
**Что:** Лимит токенов на диалог с предупреждением при приближении.
**Зачем:** Контроль расходов, особенно при использовании несколькими пользователями.
**Реализация:** Счётчик в настройках, проверка перед вызовом, отображение в UI.

## Приоритет 3 — Низкий (nice to have)

### 8. Аудит-лог действий AI
**Что:** Запись действий AI (запросы, создание, изменение) в регистр сведений.
**Зачем:** Отслеживание и контроль, возможность отката.
**Реализация:** Логирование в МодульMCP перед выполнением каждого инструмента.

### 9. Fuzzy-поиск метаданных (Левенштейн)
**Что:** Нечёткое сопоставление имён объектов ("контрагент" → "Справочник.Контрагенты").
**Зачем:** Толерантность к опечаткам и вольному именованию.
**Реализация:** Функция расстояния Левенштейна в отдельном модуле.

### 10. Mock-режим для тестирования
**Что:** Предопределённые ответы LLM для воспроизводимого тестирования без API-вызовов.
**Зачем:** Регрессионное тестирование без расходов на API.
**Реализация:** Очередь mock-ответов, подменяющая реальные вызовы.
