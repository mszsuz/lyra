# Lyra-Mobile — мобильное приложение

Общайся с пользователем на русском языке.

## Что это

Приложение-компаньон для Lyra на Flutter (iOS + Android). Не дублирует чат — работает как устройство ввода для Чата (1С) и личный кабинет пользователя. Единственный способ авторизации сессии 1С.

## Платформа

**Flutter** — единая кодовая база для iOS и Android. Ключевые пакеты:

- `speech_to_text` — нативный STT на обеих платформах
- `mobile_scanner` — сканер QR-кодов
- `camera` / `image_picker` — фото
- `centrifuge` — WebSocket-подключение к Centrifugo
- `firebase_core` + `firebase_messaging` — push-уведомления (FCM)

## Функции

### MVP

1. **Автоматическая регистрация** — device_id = user_id, создаётся при первом запуске (zero friction)
2. **Сканер QR** — сканирование QR-кода с формы Чата (1С) для авторизации сессии
3. **Баланс** — текущий остаток пользователя от биллинга
4. **Статус подключения** — какая база подключена / не подключена
5. **Список баз** — история подключений пользователя
6. **Push-уведомления** — FCM (Firebase Cloud Messaging)

### Будущие

- **Чат в мобильном** — полноценный чат с Claude: отображение стриминга ответов + голосовой ввод. Пользователь может отойти от компьютера и продолжить общение через телефон. Чат 1С остаётся открытым
- **Микрофон** — речь → текст → канал → Роутер → Claude (Чат видит на канале)
- **Камера** — фото → канал → Роутер → Claude (Чат видит на канале и отображает)
- **Статистика** — расходы по сессиям, архив чата
- **Пополнение баланса** — внешний эквайринг (детали TBD)
- **Push-уведомления** — баланс заканчивается, статус сессий

## Регистрация пользователя

Регистрация **автоматическая** — без SMS, без телефона, без подтверждений. Максимально нулевой порог входа.

### Принцип

`device_id` и `user_id` — **независимые** сущности:
- **device_id** — UUID, генерируется на телефоне при первом запуске, хранится в secure storage. Меняется при смене/потере устройства.
- **user_id** — UUID, генерируется роутером при первой регистрации. Постоянный идентификатор пользователя (баланс, базы, настройки привязаны к нему).

Связь `device_id → user_id` хранится на сервере. Пользователь сразу попадает на главный экран.

### Флоу регистрации

```text
1. Первый запуск → генерация device_id (UUID), сохранение в secure storage
2. Мобильное подключается к Centrifugo (mobile:lobby JWT)
3. Мобильное → mobile:lobby: register {device_id}
4. Роутер: device_id известен → возвращает существующий user_id
         device_id новый → создаёт user_id, привязывает к device_id
5. Роутер → mobile:lobby: register_ack {status: "ok", user_id}
6. user_id сохраняется в secure storage → переход на Главную
```

### Восстановление доступа (новое устройство)

Email можно добавить **опционально** в профиле пользователя (с подтверждением). На новом устройстве:

```text
1. Пользователь выбирает «Восстановить доступ»
2. Вводит email → Роутер отправляет код на email
3. Подтверждает код → Роутер возвращает user_id
4. Новый device_id привязывается к существующему user_id
```

Если email не был добавлен — восстановление невозможно, создаётся новый аккаунт.

## Авторизация сессии 1С (QR-флоу)

Мобильное приложение — **единственный** способ авторизации. Альтернатив (логин/пароль) нет.

### QR-код = JWT для Centrifugo

Роутер генерирует 2 JWT при обработке hello от Чата:

- `chat_jwt` (sub=`chat-<uuid>`) — для Чата
- `mobile_jwt` (sub=`mobile-<uuid>`) — для мобильного

Оба дают доступ к одному каналу сессии. `mobile_jwt` кодируется в QR-код.

`mobile_jwt` — **срок жизни 1 год**. При истечении потребуется новая авторизация (повторный hello + QR).

### Флоу

```text
1. Чат (1С) → Centrifugo (общий JWT) → lobby → hello
2. Роутер → hello_ack: {session_id, chat_jwt, mobile_jwt}
3. Чат переподключается с chat_jwt
4. Чат показывает QR = mobile_jwt
5. Мобильное сканирует QR → подключается к Centrifugo с mobile_jwt
6. Мобильное отправляет {type: "auth", user_id, device_id} на канале
7. Роутер связывает пользователя с сессией → авторизация завершена
```

Мобильное подключается к `mobile:lobby` только для регистрации. Для авторизации сессии — сразу попадает на канал сессии через JWT из QR.

**Переподключение Чата:** мобильное приложение НЕ участвует в переподключении Чата. Мобильное авторизует сессию через QR при первом подключении. При reconnect (тот же `form_id`) повторная авторизация не нужна — Роутер отправляет `hello_ack` со статусом `reconnected` (без `mobile_jwt`).

## Несколько сессий

Один пользователь может работать с **несколькими базами 1С одновременно**. Мобильное хранит список баз с историей подключений.

## Медиа-поток (голос, фото)

```text
Мобильное → публикует медиа в канал сессии
    → Роутер подхватывает → передаёт Claude
    → Чат видит (подписан на тот же канал) → отображает в чате
```

Все подписчики канала видят все сообщения — дополнительная маршрутизация не нужна.

## MDM: структура данных

- **Справочник Пользователи** — привязан к device_id, опциональный email для восстановления
- **Подчинённый справочник Базы** — история подключений пользователя к базам 1С

## Что видит мобильное на канале

- Баланс на счёте пользователя (от биллинга, в реальном времени)
- Статус подключения (база подключена / не подключена)
- Возможно другие статусные сообщения в будущем

Биллинг подключается к каналу сессии как **WebSocket-клиент Centrifugo** и отправляет баланс в реальном времени.

Стриминг ответов Claude — пока только в Чате (1С). В будущем — полноценный чат и в мобильном (см. бэклог).

## Транспорт

Подключается к **Centrifugo** по WebSocket через `centrifuge-dart` (пакет `centrifuge: ^0.14.0`).

- Регистрация — общий mobile-lobby JWT → канал `mobile:lobby`
- Авторизация сессии — персональный JWT из QR-кода → канал сессии

Все серверные компоненты (Роутер, MDM, Биллинг, Цензор) общаются между собой через Centrifugo и могут быть разнесены на разные серверы.

### Формат сообщений от Centrifugo (centrifuge-dart)

centrifuge-dart обрабатывает протокол Centrifugo и отдаёт уже разобранные данные:

- `client.publication` — данные из `pub.data` (server-side subscription). Приходит `List<int>` (UTF-8 bytes) — **один JSON-объект** на publication. Split по `\n` **не нужен** (в отличие от raw WebSocket, где Centrifugo шлёт мульти-JSON во фрейме).
- `client.connected` / `client.disconnected` / `client.connecting` — события состояния соединения.

Пример raw WS-трафика при регистрации (подключение с mobile-lobby JWT, publish register в mobile:lobby):

```
<< {"id":1,"connect":{"client":"d42eb19c-...","version":"6.6.2 OSS","expires":true,"ttl":31532436,"ping":25,"pong":true,"subs":{"mobile:lobby":{}}}}
<< {"id":2,"publish":{}}
<< {"push":{"channel":"mobile:lobby","pub":{"data":{"type":"sms_sent","reg_id":"<reg_id>","phone":"+79001234567"}}}}
```

centrifuge-dart из последнего push отдаёт в `publication.data` только `{"type":"sms_sent","reg_id":"<reg_id>","phone":"+79001234567"}`. Фильтрация по phone/reg_id на клиенте.

**ВАЖНО:** Роутер может отправлять pretty-printed JSON (с переносами строк внутри объекта). Это **не** мульти-JSON — это один объект. `jsonDecode()` парсит его корректно без split.

## Экраны и сценарии

### Карта экранов

```text
┌─────────────┐
│   Splash    │  / — проверка user_id в secure storage
│  (авто)     │      нет user_id → авто-регистрация → /home
└──────┬──────┘
       │
       └── всегда ────────────► /home (Главная)


┌──────────────────┐     ┌──────────────────┐
│     Главная      │────►│  Сканер QR       │
│     /home        │     │  /scanner        │
└────────┬─────────┘     └──────────────────┘
         │                        │
         │                        ▼
         │               ┌──────────────────┐
         ├──────────────►│  Сессия          │
         │               │  /session/:id    │
         │               └──────────────────┘
         │
         └──────────────►┌──────────────────┐
                         │  Профиль         │
                         │  /profile        │
                         └──────────────────┘
```

### 1. Splash (/)

Автоматический экран при запуске:
- user_id есть → переход на Главную
- user_id нет → авто-регистрация (генерация device_id, register в mobile:lobby) → переход на Главную

UI: спиннер загрузки по центру. Регистрация прозрачна для пользователя — никаких форм.

### 3. Главная (/home)

Основной экран после авторизации.

**Элементы:**
- AppBar: заголовок «Лира», кнопка выхода (logout) справа
- Большая кнопка «Сканировать QR» с иконкой — главное действие
- Заголовок «Активные сессии»
- Список карточек сессий (или заглушка «Нет активных сессий»)

**Карточка сессии:**
- Иконка статуса: зелёное облако (подключено) / серое (отключено)
- Название базы (или сокращённый ID сессии)
- Баланс: «150.00 руб»
- Стрелка → переход к экрану сессии

**Список сессий — с Роутера, не локальный:**
- При входе на Главную мобильное запрашивает список активных сессий у Роутера через mobile:lobby (publish `get_sessions {user_id}` → ответ `sessions_list`)
- Роутер возвращает только живые сессии этого пользователя (с балансом, статусом, названием базы)
- Если Чат (1С) закрыл форму — сессия исчезает из списка при следующем запросе
- Pull-to-refresh для обновления списка
- JWT сессий хранятся локально в secure storage (нужны для подключения к каналу), но список актуальных сессий — всегда с сервера

**Выход (logout):**
- Диалог подтверждения «Данные авторизации будут удалены. Восстановление возможно только через email (если был добавлен).»
- Очистка secure storage → авто-регистрация нового аккаунта

**Сценарий:**
1. Мобильное запрашивает у Роутера список активных сессий пользователя
2. Пользователь видит актуальный список подключённых баз 1С
3. Нажимает «Сканировать QR» → переход на Сканер
4. Нажимает на карточку сессии → переход к экрану Сессии
5. Pull-to-refresh — обновить список

### 4. Сканер QR (/scanner)

Полноэкранная камера для сканирования QR-кода из Чата 1С.

**Элементы:**
- AppBar: «Сканировать QR», кнопка «Назад»
- Видоискатель камеры на весь экран
- Подсказка внизу: «Наведите камеру на QR-код в 1С»

**Состояния после сканирования:**
- «Подключение...» — подключение к Centrifugo с JWT из QR
- «Авторизация...» — отправка auth (user_id + device_id)
- «Готово» → автоматический переход к экрану Сессии
- Ошибка → иконка ошибки + текст + «Попробовать снова» / «На главную»

**Сценарий:**
1. Камера сканирует QR (ищет строку начинающуюся с `eyJ` — JWT)
2. Подключение к Centrifugo с mobile_jwt из QR
3. Отправка auth {user_id, device_id} в канал сессии
4. Получение auth_ack → сессия сохраняется → переход к экрану Сессии

### 5. Сессия (/session/:id)

Экран активной сессии — мониторинг подключения к базе 1С.

**Элементы:**
- AppBar: название базы, кнопка «Назад»
- Карточка «Подключение»: иконка статуса + текст (Подключено / Подключение... / Отключено)
- Карточка «Баланс»: сумма крупным шрифтом (зелёный > 0, красный ≤ 0)
- Статус сессии — зависит от серверного статуса:
  - `active` — зелёная иконка, «Активна», голос и камера доступны
  - `insufficient_balance` — жёлтая иконка, «Пополните баланс», ввод заблокирован
  - `disconnected` — серая иконка, «Чат отключён», ввод заблокирован
- Панель ввода внизу экрана: кнопка микрофона + кнопка камеры (активна только при статусе `active`)

**Голосовой ввод (микрофон):**
- Кнопка микрофона → нативный STT (speech_to_text)
- При нажатии: анимация записи (пульсирующий круг), распознавание речи в реальном времени
- Распознанный текст отправляется в канал сессии → Роутер → Claude
- Чат (1С) видит текст на канале и отображает как сообщение пользователя

**Отправка фото (камера):**
- Кнопка камеры → выбор: сделать снимок / выбрать из галереи
- Превью снимка + кнопка «Отправить» / «Отмена»
- Фото отправляется в канал сессии (base64 или URL) → Роутер → Claude
- Чат (1С) видит фото на канале и отображает в чате

**Жизненный цикл сессии:**
- Сессия создаётся при сканировании QR, хранится в secure storage
- Мобильное автоматически переподключается к Centrifugo при входе на экран
- Сессия пропадает из списка, когда сервер её закрывает (закрытие формы в 1С или TTL 30 мин без клиента)
- Удалять сессию вручную не нужно — её время жизни определяет Роутер

**Обновления в реальном времени:**
- Баланс обновляется через balance_update от Биллинга
- Статус подключения обновляется автоматически (connected/connecting/disconnected)

**Сценарий:**
1. Пользователь видит состояние подключения к базе 1С
2. Баланс обновляется в реальном времени после каждого запроса к Claude
3. При нулевом балансе — статус «Недостаточно средств» (сессия не закрывается, можно пополнить)
4. Нажимает микрофон → говорит вопрос → текст уходит в Claude через канал сессии
5. Нажимает камеру → делает/выбирает фото → фото уходит в Claude через канал сессии
6. Чат (1С) отображает все сообщения от мобильного в реальном времени
7. Кнопка «Назад» — возврат на Главную (сессия остаётся активной)

### 6. Профиль (/profile)

Экран профиля пользователя — email для восстановления, настройки.

**Элементы:**
- Email: поле ввода + кнопка «Подтвердить» (отправляет код на email)
- Статус email: «Не указан» / «Подтверждён» (зелёная галочка)
- device_id (readonly, для поддержки)

**Восстановление на новом устройстве:**
- Кнопка «Восстановить доступ» на splash-экране (если нет user_id)
- Ввод email → код на email → подтверждение → привязка нового device_id к user_id

### Будущие экраны (после MVP)

- **Статистика** — расходы по сессиям, архив чатов
- **Пополнение баланса** — внешний эквайринг
- **Микрофон** — запись голоса → STT → отправка текста в канал сессии
- **Камера** — фото → отправка в канал сессии
- **Настройки** — выбор сервера (для dev), уведомления
