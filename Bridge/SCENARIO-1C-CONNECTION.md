# Сценарий подключения 1С к Lyra — Agent SDK

## Что изменилось

Bridge больше не запускает `claude` CLI как child process. Вместо этого — программный вызов через `@anthropic-ai/claude-agent-sdk`. Это даёт:

- Стриминг от всех агентов (Haiku и Sonnet) в реальном времени
- MCP-инструменты для 1С создаются прямо в Bridge (без отдельного процесса)
- Роутер (Haiku) вызывает специалиста (Sonnet) через встроенный механизм `agents`
- Трекинг токенов по моделям, лимит бюджета

## Участники системы

| Компонент | Роль | Реализация |
|---|---|---|
| **1С** | WebSocket-клиент, чат-форма | Платформа 8.3.18+ |
| **Bridge** | Диспетчер, прокси, оркестратор агентов, биллинг | Node.js + Agent SDK |
| **Haiku** | Первая линия — роутер, живёт всю сессию | `ClaudeSDKClient`, model: haiku |
| **Sonnet** | Аналитик 1С — вызывается Haiku через `agents` | Определён как agent в SDK |
| **1С MCP** | exec-инструменты (query, eval, metadata, exec) | `createSdkMcpServer` внутри Bridge |
| **Vega MCP** | Метаданные конфигураций (общий на хосте) | Подключается через `mcpServers` |
| **mcp-1c-docs** | Документация 1С (общий на хосте) | Подключается через `mcpServers` |
| **Мобильное приложение** | Авторизация через QR, баланс | Отдельный проект |
| **Сервис биллинга** | Баланс, тарифы, списание | Отдельный сервис |

## Общая схема

```
Мобильное              1С (WebSocket-клиент)
приложение                    │
    │                         │ ws://host:3003
    │ POST /auth/confirm      │
    │                  ┌──────┴──────┐
    └─────────────────►│  Диспетчер  │◄──► Биллинг
                       │  порт 3003  │
                       └──────┬──────┘
                              │ hello_ack → connectUrl
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
     ┌────────────────┐ ┌────────────────┐ ┌────────────────┐
     │ Рабочий Bridge │ │ Рабочий Bridge │ │ Рабочий Bridge │
     │  порт 4001     │ │  порт 4002     │ │  порт 4003     │
     │                │ │                │ │                │
     │  Agent SDK     │ │  Agent SDK     │ │  Agent SDK     │
     │  ├─ Haiku      │ │  ├─ Haiku      │ │  ├─ Haiku      │
     │  └─ Sonnet     │ │  └─ Sonnet     │ │  └─ Sonnet     │
     │     ├ Vega MCP │ │     ├ Vega MCP │ │     ├ Vega MCP │
     │     ├ docs MCP │ │     ├ docs MCP │ │     ├ docs MCP │
     │     └ 1С MCP ──┼─┤     └ 1С MCP ──┼─┤     └ 1С MCP  │
     └────────┬───────┘ └────────┬───────┘ └────────┬───────┘
              │                  │                  │
              ▼                  ▼                  ▼
          1С (база А)       1С (база Б)       1С (база В)
```

Диспетчер — точка входа, авторизация, балансировка. Рабочие Bridge — отдельные процессы/машины, каждый с Agent SDK.

## Шаг 1. Запуск Bridge

**Кто:** Администратор

Bridge стартует:
- Диспетчер: WebSocket-сервер на порту 3003 (авторизация, маршрутизация)
- Рабочие Bridge: на портах 4001+ (Agent SDK, сессии)
- Каждый рабочий Bridge создаёт один `1С MCP` сервер через `createSdkMcpServer` (инструменты query, eval, metadata, exec) — общий для всех сессий на этом воркере, обработчики маршрутизируют по сессии
- Подключение к сервису биллинга

Vega MCP и mcp-1c-docs работают на хосте как отдельные процессы (общие для всех пользователей).

## Шаг 2. Подключение и авторизация

### 2а. 1С подключается, отправляет hello

```json
{
  "type": "hello",
  "config": "БухгалтерияПредприятия",
  "version": "3.0.150.27",
  "processingVersion": "1.0.3",
  "baseId": "base-a1b2c3",       // опционально, отсутствует при первом подключении
  "userName": "Иванов И.И."
}
```

### 2б. Bridge валидирует, генерирует JWT, отправляет QR

```json
{
  "type": "auth_required",
  "qrData": "<JWT: sessionId + exp>"
}
```

1С показывает QR на форме.

### 2в. Мобильное приложение сканирует QR, подтверждает

Мобильное → Bridge: JWT + userId
Bridge проверяет JWT, запрашивает баланс в биллинге.

### 2г. Bridge отвечает 1С

**Баланс > 0:** диспетчер выбирает рабочий Bridge (по нагрузке, доступности) и отвечает:
```json
{
  "type": "hello_ack",
  "baseId": "base-a1b2c3",
  "sessionId": "...",
  "connectUrl": "ws://host:4017/"
}
```

**Баланс = 0:**
```json
{
  "type": "hello_reject",
  "reason": "zero_balance",
  "message": "Пополните баланс в приложении Lyra"
}
```

**Архитектура:** диспетчер (порт 3003) — точка входа, только авторизация и маршрутизация. Рабочие Bridge — отдельные экземпляры на своих портах/машинах. Это даёт отказоустойчивость (упал рабочий — остальные живы), масштабирование (новые Bridge по нагрузке) и изоляцию сессий.

## Шаг 3. Запуск Haiku и приветствие

### 3а. 1С переподключается на рабочий Bridge

1С закрывает соединение с диспетчером, открывает WebSocket на `connectUrl` из hello_ack.

### 3б. Рабочий Bridge создаёт сессию через Agent SDK

Рабочий Bridge вызывает `ClaudeSDKClient` — долгоживущая сессия Haiku:

Параметры:
- **model:** haiku
- **systemPrompt:** промпт Лиры (из `lyra-router-prompt.md`) + контекст сессии (конфигурация, версия, имя пользователя, роль)
- **includePartialMessages:** true (стриминг)
- **tools:** `respond_to_user(response)` — кастомный инструмент для прямого ответа пользователю
- **agents:** `{ analyst: { description, prompt (из lyra-analyst-prompt.md), model: "sonnet" } }` — субагент-аналитик, Haiku вызывает его через встроенный Task tool
- **mcpServers:** Vega, mcp-1c-docs (общие), 1С MCP (общий на воркере, маршрутизация по сессии)
- **allowedTools:** respond_to_user, Task, mcp__vega__*, mcp__docs__*, mcp__1c__*

### 3в. 1С MCP — маршрутизация по сессии

1С MCP один на рабочий Bridge (создаётся при старте воркера). Инструменты: 1c_query, 1c_eval, 1c_metadata, 1c_exec. Когда Sonnet вызывает 1c_query — обработчик определяет сессию вызова, находит нужный WebSocket 1С, отправляет `mcp_request`, ждёт `mcp_response`, возвращает результат агенту.

### 3г. Haiku приветствует пользователя

Bridge отправляет первый запрос в Haiku (пустой или "поприветствуй пользователя").

Haiku вызывает инструмент:
```
respond_to_user(response: "Привет, Иванов И.И.! Я Лира — ИИ-ассистент для 1С. Чем могу помочь?")
```

Bridge получает `content_block_start` с `name: "respond_to_user"` → стримит аргумент `response` побуквенно в 1С через WebSocket.

### 3д. Что приходит в 1С (стриминг)

```json
{"type":"stream_event","event":{"type":"content_block_delta","delta":{"text":"Привет"}}}
{"type":"stream_event","event":{"type":"content_block_delta","delta":{"text":", Иванов"}}}
...
{"type":"result","result":"Привет, Иванов И.И.! Я Лира — ..."}
```

## Шаг 4. Простой вопрос (Haiku отвечает сам)

### 4а. 1С отправляет

```json
{"type":"chat","content":"Что такое УСН?"}
```

### 4б. Bridge проверяет баланс

Запрос в биллинг по userId. Ответ: `{ balance, maxBudgetUsd }`. Баланс > 0 → продолжаем. Баланс <= 0 → отказ.

### 4в. Bridge передаёт вопрос в Haiku

Через `client.query("Что такое УСН?", { maxBudgetUsd })` — лимит на запрос из биллинга.

### 4г. Haiku вызывает respond_to_user

```
respond_to_user(response: "УСН — упрощённая система налогообложения...")
```

### 4д. Bridge обрабатывает

1. Получает `content_block_start` с `name: "respond_to_user"` → начинает стриминг
2. Стримит аргумент `response` в 1С (дельты по буквам через `input_json_delta`)
3. После завершения — отправляет `result` в 1С

### 4е. Bridge списывает токены

Из `message.usage` берёт inputTokens, outputTokens, model → отправляет в биллинг.

## Шаг 5. Сложный вопрос (Haiku → Sonnet)

### 5а. 1С отправляет

```json
{"type":"chat","content":"Почему у контрагента Ромашка не заполнен КПП?"}
```

### 5б. Bridge проверяет баланс, передаёт Haiku

### 5в. Haiku решает: нужен специалист

```
Task(subagent_type: "analyst", prompt: "Пользователь спрашивает про КПП контрагента Ромашка. Нужно проверить в базе.")
```

Bridge получает `content_block_start` с `name: "Task"` → знает что это вызов субагента, не стримит аргументы в 1С.

### 5г. Что происходит внутри Agent SDK

Agent SDK:
1. Собирает промпт Sonnet: фиксированный системный промпт + context от Haiku + вопрос пользователя
2. Запускает Sonnet (API-вызов, не процесс)
3. Sonnet получает доступ к MCP: Vega, mcp-1c-docs, 1С MCP

**Критическое отличие от CLI:** Agent SDK стримит ВСЕ события от Sonnet — текст, вызовы инструментов — через тот же `for await` цикл. `parent_tool_use_id` указывает что событие от субагента.

### 5д. Sonnet работает — стриминг в 1С в реальном времени

Bridge получает StreamEvent от Sonnet и пробрасывает в 1С:

```
→ 1С: {"type":"stream_event","event":{"delta":{"text":"Ищу контрагента "}}}
→ 1С: {"type":"stream_event","event":{"delta":{"text":"Ромашка в базе..."}}}
```

Пользователь видит текст по буквам — никаких пауз.

### 5е. Sonnet вызывает 1c_query — exec через Bridge → 1С

Sonnet вызывает инструмент `mcp__1c__1c_query`. Обработчик внутри Bridge:

1. Получает вызов от Agent SDK
2. Отправляет в WebSocket 1С:
   ```json
   {"type":"mcp_request","requestId":"...","tool":"1c_query","params":{"query":"ВЫБРАТЬ ..."}}
   ```
3. Ждёт ответа от 1С:
   ```json
   {"type":"mcp_response","requestId":"...","result":"[{...}]"}
   ```
4. Возвращает результат в Agent SDK → Sonnet получает данные

Для пользователя exec-вызовы скрыты. Видны только текстовые комментарии Sonnet:

```
→ 1С: {"type":"stream_event","event":{"delta":{"text":"Нашёл: Ромашка — физлицо. "}}}
→ 1С: {"type":"stream_event","event":{"delta":{"text":"У физлиц КПП не бывает."}}}
```

### 5ж. Sonnet завершает

Ответ Sonnet возвращается в Haiku (через Agent SDK). Haiku видит результат в своём контексте — сможет суммаризовать для следующего вызова специалиста.

### 5з. Bridge списывает токены

Из `result.modelUsage` — отдельно Haiku и Sonnet:
```json
{
  "claude-haiku-4-5": { "inputTokens": 500, "outputTokens": 50 },
  "claude-sonnet-4-5": { "inputTokens": 5000, "outputTokens": 800 }
}
```
Два списания в биллинг — за Haiku и за Sonnet.

## Шаг 6. Продолжение диалога (follow-up)

Пользователь: "А у Василька?"

1. Bridge проверяет баланс
2. `client.query("А у Василька?")` — Haiku получает вопрос в той же сессии (контекст сохранён)
3. Haiku помнит: обсуждали КПП, Ромашка — физлицо
4. Haiku вызывает `Task(subagent_type: "analyst", prompt: "Выясняли КПП у Ромашки — физлицо. Теперь проверить Василёк.")`
5. Agent SDK запускает нового Sonnet с context + вопрос
6. Sonnet делает запрос по Васильку, стримит ответ
7. Списание

## Шаг 7. Эскалация Sonnet → Opus (будущее)

Два варианта:
- **MVP:** Кнопка "плохой ответ" в 1С → Bridge запускает отдельный `query()` с model: opus и контекстом предыдущей попытки
- **Позже:** Sonnet возвращает `confidence: low` → Bridge автоматически переспрашивает через Opus

Можно определить Opus как ещё одного agent:
```
agents: {
  analyst: { model: "sonnet", ... },
  senior_analyst: { model: "opus", ... }
}
```

## Шаг 8. Переподключение при обрыве

1. WebSocket разрывается
2. 1С переподключается к диспетчеру (порт 3003), отправляет hello с baseId + sessionId
3. Авторизация (QR или кэшированная)
4. Диспетчер по маппингу session→worker находит рабочий Bridge, на котором живёт `ClaudeSDKClient`
5. hello_ack с `connectUrl` этого воркера
6. 1С подключается к рабочему Bridge, он привязывает новый WebSocket к существующей сессии
7. Диалог продолжается, Haiku жив, контекст сохранён

Если ClaudeSDKClient не найден (воркер упал, таймаут) — создаётся новая сессия на доступном воркере.

---

## Шаг 9. Завершение сессии

| Событие | Что происходит |
|---|---|
| 1С закрывает форму | WebSocket закрывается. Bridge держит ClaudeSDKClient на случай reconnect |
| Таймаут неактивности | Bridge вызывает `client.disconnect()`, освобождает ресурсы |
| Баланс ушёл в ноль | Следующий запрос не пройдёт проверку. Сессия не закрывается |
| maxBudgetUsd исчерпан | Agent SDK сам остановит агента. Bridge уведомляет 1С |

## Главные отличия от CLI-версии

| | CLI (было) | Agent SDK (стало) |
|---|---|---|
| Запуск Claude | `spawn('claude', [...])` child process | `query()` / `ClaudeSDKClient` — API-вызов |
| MCP для 1С | Отдельный процесс `bridge.js --mcp` | `createSdkMcpServer` внутри Bridge |
| Стриминг Sonnet | Нет (только финальный результат) | Да (StreamEvent в реальном времени) |
| Роутер → специалист | Не работало нативно | `agents: { analyst: {...} }` |
| Маршрутизация exec | Через WebSocket MCP→Bridge→1С | Обработчик в `createSdkMcpServer` → WebSocket → 1С |
| Трекинг токенов | `total_cost_usd` в result | `modelUsage` — раздельно по моделям |
| Лимит бюджета | Нет | `maxBudgetUsd` в опциях |
| Кол-во процессов | 3 (bridge + claude + bridge --mcp) | 1 (bridge с Agent SDK) |
| Архитектура | Один процесс, один порт | Диспетчер (3003) + рабочие Bridge (4001+) |

## Вопросы для обсуждения

1. ~~**Haiku отвечает JSON**~~ — **Решено:** Haiku отвечает не JSON, а вызовом инструментов. Два инструмента: кастомный `respond_to_user(response)` для прямого ответа и встроенный `Task` (Agent SDK) для вызова субагента-аналитика. При стриминге `content_block_start` приходит **до** аргументов и содержит имя инструмента — Bridge сразу знает действие. `respond_to_user` → стримит аргумент `response` побуквенно в 1С. `Task` → субагент, Agent SDK стримит события Sonnet. Подтверждено тестом (`test-subagent-output.json`, строка 177: `type: "tool_use"`, `name: "Task"` приходит отдельным блоком до аргументов).

2. ~~**Haiku видит ответ Sonnet?**~~ — **Решено:** да, видит. Agent SDK возвращает результат субагента как tool_result в контекст Haiku. Оставляем — Haiku дешёвый ($0.80/M input), а контекст нужен для follow-up вопросов. Подтверждено тестом (`test-subagent-output.json`: tool_result на строке 224, пересказ Haiku на строке 280).

3. ~~**createSdkMcpServer и async-ответы**~~ — **Решено:** паттерн уже реализован и протестирован в текущем `bridge.js` (строки 280-311). `pending` Map + `call1c()` возвращает Promise, резолвится при получении `mcp_response` по `requestId`. Таймаут 30 сек. Переносится в `createSdkMcpServer` без изменений — async обработчики стандартны для MCP SDK.

4. ~~**Один порт vs два**~~ — **Решено:** диспетчер (3003) + рабочие Bridge на отдельных портах/машинах. connectUrl в hello_ack.

5. ~~**maxBudgetUsd**~~ — **Решено:** `maxBudgetUsd` приходит из сервиса биллинга вместе с проверкой баланса. Биллинг знает тарифы и может рассчитать разумный лимит на один запрос. Bridge передаёт значение в `query()` Agent SDK как есть. Это страховка от бесконечного цикла агента, не связана с балансом напрямую.
