# Навык: Где заполняется / очищается / проверяется реквизит

Применяй этот алгоритм когда пользователь спрашивает:
- «Где заполняется реквизит X?»
- «Почему реквизит X пустой / обнуляется?»
- «Кто устанавливает значение X?»
- «Где проверяется заполнение X?»

## Алгоритм (3-5 вызовов)

### Шаг 1. Определи владельца реквизита

Если пользователь назвал объект (документ, справочник) — используй его как `owner`.

Если нет — найди:
```json
{"op": "find_objects_by_attribute", "attr": "Себестоимость", "match": "exact"}
```
Результат: список объектов, у которых есть этот реквизит (может быть в шапке или в табличной части).

### Шаг 2. Найди код, который работает с реквизитом

```json
{"op": "find_routines_by_code", "text": "Себестоимость", "search_mode": "fulltext", "owner": "Документы.ВозвратТоваровОтПокупателя"}
```

**Важно:**
- Всегда `search_mode: "fulltext"` — ищем точное имя, не семантику
- Всегда `owner` — сужает до конкретного объекта, иначе найдёт сотни процедур по всей конфигурации
- Смотри на имена найденных процедур: `ПередЗаписью`, `ОбработкаЗаполнения`, `ОбработкаПроверкиЗаполнения`, `ПриИзменении` — это типичные точки входа

### Шаг 3. Прочитай код найденных процедур

```json
{"op": "get_routine_body", "routine_name": "ПередЗаписью", "owner": "Документы.ВозвратТоваровОтПокупателя"}
```

Или по `routine_id` из результатов шага 2.

Анализируй код:
- Присваивание `Реквизит = 0` или `Реквизит = Неопределено` → очистка
- `Если ... Тогда Реквизит = ...` → условное заполнение
- `МассивНепроверяемыхРеквизитов.Добавить("Реквизит")` → отключение проверки заполнения
- `Движения.РегистрX.Добавить()` с этим реквизитом → использование при проведении

### Шаг 4. (опционально) Проследи цепочку вызовов

Если процедура вызывает другую функцию, которая выполняет работу:
```json
{"op": "call_graph_subtree", "routine_id": "<id из шага 2>", "direction": "out", "depth": 2}
```

Или если хочешь понять, кто вызывает найденную процедуру:
```json
{"op": "call_graph_subtree", "routine_id": "<id>", "direction": "in", "depth": 2}
```

## Типичные точки заполнения реквизитов в 1С

| Событие | Когда срабатывает | Что обычно делает |
|---|---|---|
| `ОбработкаЗаполнения` | Создание нового документа (Ввести на основании) | Заполняет реквизиты из документа-основания |
| `ПередЗаписью` | Перед записью в базу | Очистка, пересчёт, нормализация данных |
| `ОбработкаПроведения` | При проведении документа | Формирование движений по регистрам |
| `ОбработкаПроверкиЗаполнения` | Перед записью/проведением | Проверка обязательных реквизитов |
| `ПриИзменении` (форма) | Пользователь изменил поле на форме | Пересчёт зависимых полей |
| `ПриУстановкеНовойСтроки` (форма) | Добавление строки в табличную часть | Значения по умолчанию |

## Пример: расследование «Себестоимость» в ВозвратТоваровОтПокупателя

**Вопрос:** Где заполняется реквизит Себестоимость в документе ВозвратТоваровОтПокупателя?

**Шаг 1:** `find_objects_by_attribute(attr="Себестоимость")` → нашли: это реквизит табличной части Товары

**Шаг 2:** `find_routines_by_code(text="Себестоимость", search_mode="fulltext", owner="Документы.ВозвратТоваровОтПокупателя")` → 8 процедур, включая ПередЗаписью, ОбработкаПроверкиЗаполнения

**Шаг 3:** `get_routine_body(routine_name="ПередЗаписью", owner="Документы.ВозвратТоваровОтПокупателя")` → нашли: если указана партия (Сделка заполнена валидным документом), Себестоимость обнуляется

**Ответ:** Себестоимость — ручной реквизит. Если есть документ-основание (Сделка), он очищается до 0 в ПередЗаписью. Если основания нет — обязателен для заполнения вручную.
