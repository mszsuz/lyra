# Lyra — ИИ-ассистент для 1С

## Концепция
- ИИ-ассистент на движке Claude Code, встроенный в 1С:Предприятие через чат
- Пользователи задают любые вопросы через чат прямо в интерфейсе 1С
- Часть экосистемы Lyra + 1ext.com

## Архитектура
```
1С → WebSocket → Bridge → Haiku (первая линия, основной инстанс)
                   │        │
                   │        ├── простой вопрос → отвечает сам → Bridge → 1С
                   │        │
                   │        └── сложный вопрос → Bridge запускает Sonnet (одноразовый)
                   │              │       │
                   │              │       ├── Vega MCP — метаданные, код, семантика
                   │              │       ├── mcp-1c-docs — документация, примеры
                   │              │       └── ...другие MCP (общие, на хосте)
                   │              │
                   │              ├── exec (1c_query/1c_eval) → Bridge → 1С → результат → Sonnet
                   │              ├── промежуточные шаги (стрим) → Bridge → 1С (по буквам)
                   │              └── финальный ответ → Bridge → 1С
                   │
                   └── exec-маршрутизатор: знает пару "Claude-процесс ↔ WebSocket 1С"
```

## Bridge
- Посредник между 1С и Claude — тупой прокси, ничего не меняет в сообщениях
- 1С подключается к Bridge по WebSocket, Bridge запускает Claude (Haiku) для каждого пользователя
- Публикация базы НЕ нужна — 1С-клиент инициирует соединение наружу
- Если 1С отключилась и подключилась обратно — сессия сохраняется, Claude помнит контекст
- Весь лог записывается в файл
- Два типа сообщений:
  - `display` — текст для вывода в чат пользователю (стрим по буквам)
  - `exec` — служебная команда (1c_query, 1c_eval) → 1С выполняет, сериализует, возвращает результат
- **exec-маршрутизация**: Bridge знает пару "Claude-процесс ↔ WebSocket 1С", поэтому exec-команды от Sonnet попадают в правильную 1С-базу. Это НЕ отдельный MCP-сервер, а часть протокола Bridge.

## Haiku — первая линия поддержки
- Основной Claude-инстанс, "лицо" Лиры — пользователь всегда общается с ним
- **Постоянный процесс** — живёт всю сессию, накапливает историю в контексте
- Представляет команду: пользователь не знает о существовании специалистов
- На простые вопросы отвечает сам (приветствия, общие вопросы, справка по 1С)
- Сложные вопросы → возвращает JSON с `specialist` + `context` (суммаризация предыдущего обсуждения)
- Bridge по JSON запускает нужного специалиста с фиксированным промптом + context + вопрос
- Контекст 200K токенов хватит с запасом (50 сообщений ≈ 5-25K токенов, auto-compaction при необходимости)
- Системный промпт: `docs/lyra-router-prompt.md`

## Три слоя знаний
1. **Как устроено** (Vega) — структура конфигурации, код модулей, связи между объектами
2. **Как должно быть** (mcp-1c-docs) — документация, best practices, справочник языка
3. **Как есть сейчас** (1c-tools через WS) — реальные данные в базе пользователя

## Sonnet — аналитик 1С (специалист)
- **Одноразовый процесс** — запускается Bridge на каждый эскалированный вопрос, завершается после ответа
- Получает: фиксированный системный промпт + context (от Haiku) + точный вопрос пользователя
- Имеет доступ к MCP-инструментам (Vega, mcp-1c-docs — общие на хосте)
- exec-команды (1c_query, 1c_eval) маршрутизируются Bridge в 1С пользователя
- Итеративно исследует вопрос, комментируя шаги и промежуточные результаты
- Вывод стримится по буквам через Bridge → WebSocket → чат 1С
- Знает конфигурацию и версию пользователя
- В будущем команду можно расширять: консультант-методолог (Opus), разработчик (Sonnet) и др.

## Связь с Vega
- Vega и Lyra живут на одном хосте
- Vega обновляется при выходе новых релизов 1С (`vega import --release ...`)
- Инкрементальные embeddings (v4) обеспечивают быстрое обновление (~20 сек)
- Лира знает из какой конфигурации и версии поступил вопрос

## Мультитенантность и сессии
- При подключении WebSocket клиент передаёт: конфигурация, версия, ИД сессии
- Лира по этим данным выбирает нужный Vega-инстанс (БП/УТ/ERP + версия релиза)
- Контекст диалога сохраняется в рамках сессии (follow-up вопросы работают)

## Безопасность
- 1С-запросы безопасны: в языке запросов 1С нет UPDATE/DELETE — только SELECT
- 1c_eval — вычисление выражений (Вычислить()), не процедуры
- Деструктивных операций через MCP-канал нет

## Монетизация
- Каждый запрос уменьшает баланс клиента
- Пока баланс > 0 — всё работает
- Стоимость зависит от сложности (Haiku дешевле, Sonnet + MCP round-trips дороже)

## Стоимость моделей Claude API (за 1M токенов)
| Модель | Input | Output |
|--------|-------|--------|
| Haiku 4.5 | $0.80 | $4 |
| Sonnet 4.5 | $3 | $15 |
| Opus 4.6 | $15 | $75 |

### Расчёт стоимости одного запроса
```
Стоимость = (input_tokens × input_price / 1_000_000) + (output_tokens × output_price / 1_000_000)
```

### Примерные сценарии
- **Haiku-роутер** (классификация): ~500 in + ~50 out ≈ **$0.0006** (~0.05 руб)
- **Sonnet-агент** (сложный вопрос, 5 MCP round-trips): ~20K in + ~2K out ≈ **$0.09** (~7 руб)
- **Opus** (самые сложные): ~20K in + ~2K out ≈ **$0.45** (~36 руб)

### Стратегия
- Haiku-роутер на каждый запрос — копеечная классификация
- 90% простых вопросов — Haiku сам (~0.05 руб)
- Сложные вопросы по 1С — Sonnet (~7 руб)
- Opus — только если Sonnet не справился (см. эскалацию ниже)

## Эскалация Sonnet → Opus
Как определить, что Sonnet не справился:

### Подходы
1. **Self-assessment** — Sonnet в служебном поле возвращает `confidence: low/medium/high`. При `low` → автоматически Opus. Минус: модели склонны завышать уверенность.
2. **Эвристики по поведению** — >8 MCP-вызовов без ответа, пустые результаты, слишком короткий/общий ответ, Sonnet сам пишет "не удалось определить".
3. **Обратная связь пользователя** — "это неправильно" / кнопка "плохой ответ" → переспрос через Opus с контекстом предыдущей попытки.

### Рекомендуемый план
- **MVP**: вариант 3 (обратная связь) — простейший в реализации
- **Позже**: добавить 1+2 когда будет статистика по реальным вопросам
- Заранее сложно предсказать где Sonnet ошибается — лучше собрать данные

### Комбинированная схема (целевая)
```
Sonnet отвечает
  → confidence: high → отправить пользователю
  → confidence: low  → автоматически Opus (пользователь не заметит)
  → пользователь недоволен → Opus с контекстом "предыдущий ответ был неверным"
```

## Архитектурные решения (Q&A)

### Haiku — постоянный, с полной историей
Haiku — долгоживущий процесс на всю сессию. Накапливает все сообщения в контексте.
Именно поэтому может суммаризовать предыдущее обсуждение в поле `context` для специалиста.
Сессия переживает переподключение 1С (Claude-процесс живёт независимо от WebSocket).

### Sonnet — одноразовый, без состояния
Запускается заново на каждый эскалированный вопрос. Получает фиксированный промпт + context + вопрос.
Если бы Sonnet был постоянным, поле `context` было бы не нужно.
Плюсы: нет накопления контекста, дешевле, проще, каждый вопрос — чистый лист.

### exec-команды маршрутизируются через Bridge
Vega и mcp-1c-docs — общие MCP-серверы на хосте (одни на всех пользователей).
Но 1c_tools (exec) должен попасть в конкретную 1С. Bridge знает пару
"Claude-процесс ↔ WebSocket 1С" и маршрутизирует exec-команды в нужный WebSocket.
Это часть протокола Bridge, НЕ отдельный MCP-сервер.

### Стриминг промежуточных шагов
Bridge стримит stdout Sonnet-процесса напрямую в WebSocket пользователя, посимвольно.
MCP-вызовы обрабатываются серверно (скрыты). Текстовый вывод Sonnet
("Ищу в метаданных...", "Нашёл регистр...") виден в чате в реальном времени.

### Контекст Haiku не переполнится
200K токенов Haiku хватит с запасом. 50 сообщений ≈ 5-25K (2.5-12.5%).
Auto-compaction в Claude Code справится с edge cases.
Пользователи редко пишут 200+ сообщений в одной сессии.

## Стиль общения (критично!)
- **Язык пользователя**, НЕ программиста: "звёздочка", "пункт меню", "галочка"
- **НИКОГДА** не говорить: "регламентное задание", "регистр сведений", "функциональная опция", "план обмена"
- **Уточняй размытые вопросы** — сначала спроси, потом действуй
- **Думай о простом** — "пропала синхронизация" может означать удалённую звёздочку в Избранном
- Направляй по интерфейсу: "Администрирование → Синхронизация данных"

## Примеры циклов

### Простой (Haiku сам, с уточнением)
```
Пользователь: "У меня пропала синхронизация, что делать?"
  → Haiku: вопрос размытый → уточняет
  → "Подскажите: пропала ссылка из меню? Данные не приходят? Ошибка?"
  → Пользователь: "Пропала звёздочка"
  → Haiku: это просто! Отвечает сам
  → "Откройте Администрирование → Синхронизация данных, нажмите звёздочку."
```

### Сложный (Haiku → Sonnet)
```
Пользователь: "Почему у контрагента Ромашка не заполнен КПП?"
  → Haiku: нужны данные → передаёт аналитику
  → Sonnet: Vega search_metadata → Справочники.Контрагенты, атрибут КПП
  → Sonnet: 1c_query (через WS) → SELECT КПП, ЮрФизЛицо WHERE Наименование LIKE "Ромашка%"
  → Sonnet: видит ЮрФизЛицо = ФизЛицо → понимает причину
  → Пользователь: "КПП не заполнен, потому что Ромашка — физлицо. У физических лиц КПП не бывает."
```

### С уточнением → эскалация
```
Пользователь: "У меня пропала синхронизация"
  → Haiku: уточняет
  → Пользователь: "Данные перестали приходить из Бухгалтерии"
  → Haiku: нужен анализ → передаёт аналитику с контекстом
  → {"specialist":"analyst","context":"Пропала синхронизация. Уточнили — данные не приходят из БП."}
```
