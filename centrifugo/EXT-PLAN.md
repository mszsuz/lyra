## Порядок разработки

### Этап 1. Справочник + Конфигурация

**Цель:** можно создать элемент справочника с настройками и прочитать их программно.

1. Добавить реквизиты в справочник `ЕХТ_Центрифуга`:
   - `URLВебСокет` (Строка, 250)
   - `URLСерверAPI` (Строка, 250)
   - `КлючAPI` (Строка, 250)
   - `СекретHMAC` (Строка, 250)

2. Обновить форму элемента — вывести реквизиты на форму

3. Создать общий модуль `ЕХТ_Центрифуга_Конфигурация`:
   - `СерверПоУмолчанию()` — первый (единственный) элемент справочника
   - `URLВебСокет(Сервер)`, `URLСерверAPI(Сервер)`, `КлючAPI(Сервер)`, `СекретHMAC(Сервер)` — чтение реквизитов через `ЕХТ_Ссылка.ЗначенияРеквизитов`

**Проверка:** создать элемент справочника, заполнить, прочитать программно.

### Этап 2. Server API (HTTP)

**Цель:** из 1С можно вызвать Centrifugo Server API (publish, subscribe и т.д.).

1. Служебная функция `ВызовServerAPI(Сервер, Метод, ТелоЗапроса)`:
   - HTTP POST на `URLСерверAPI(Сервер)/метод`
   - Заголовок `X-API-Key: КлючAPI(Сервер)`
   - Тело: JSON
   - Возврат: структура ответа

2. Экспортные функции в `ЕХТ_Центрифуга`:
   - `Publish(Сервер, Канал, Данные)` — POST `/api/publish`
   - `Subscribe(Сервер, Пользователь, КлиентИД, Канал)` — POST `/api/subscribe`
   - `Unsubscribe(Сервер, Пользователь, КлиентИД, Канал)` — POST `/api/unsubscribe`
   - `Presence(Сервер, Канал)` — POST `/api/presence`
   - `History(Сервер, Канал)` — POST `/api/history`

**Проверка:** запустить Centrifugo, вызвать `ЕХТ_Центрифуга.Publish(Сервер, "test:ch", Новый Структура("msg", "hello"))` — увидеть в Admin UI.

### Этап 3. JWT

**Цель:** генерировать JWT-токены для клиентов Centrifugo.

Используется платформенный объект `ТокенДоступа` (с 8.3.21) — ручная реализация HMAC/Base64URL не нужна.

Экспортная функция `СоздатьJWT(Сервер, Sub, Каналы, ВремяЖизни)`:

```bsl
Токен = Новый ТокенДоступа();
Токен.КлючСопоставленияПользователя = Sub;  // claim "sub"
Токен.ВремяЖизни = ВремяЖизни;              // claim "exp"
Токен.ПолезнаяНагрузка.Вставить("channels", Каналы);

СекретBase64 = СекретHMAC(Сервер);  // из справочника, в формате Base64
Токен.Подписать(АлгоритмПодписиТокенаДоступа.HS256, СекретBase64);

Возврат Строка(Токен);  // готовый JWT-строка
```

**Нюанс:** `СекретHMAC` в справочнике должен храниться в формате Base64 (метод `Подписать` ожидает Base64-строку).

**Проверка:** сгенерировать JWT, подключиться к Centrifugo с ним через Node.js тест.

### Этап 4. WebSocket-клиент

**Цель:** серверный компонент (Роутер, Биллинг) может подключиться к Centrifugo по WebSocket, подписаться на каналы и получать/отправлять сообщения.

1. Создать общий модуль `ЕХТ_Центрифуга_ВебСокетКлиент`

2. `Подключиться(Сервер, JWT)`:
   - `НовыйОбъект("WebSocketКлиентСоединения")` на `URLВебСокет(Сервер)`
   - Отправить `{"id":1,"connect":{"token":"JWT..."}}`
   - Дождаться ответа `{"id":1,"connect":{...}}`
   - Запомнить `ping` интервал из ответа
   - Вернуть структуру-клиент (соединение, счётчик id, очередь сообщений)

3. `Подписаться(Клиент, Канал)`:
   - Отправить `{"id":N,"subscribe":{"channel":"..."}}`

4. `Опубликовать(Клиент, Канал, Данные)`:
   - Отправить `{"id":N,"publish":{"channel":"...","data":{...}}}`

5. `ПолучитьСообщения(Клиент)`:
   - Прочитать входящие фреймы
   - Разбить по `\n` (мульти-JSON)
   - Отфильтровать ping `{}` (отправить pong)
   - Развернуть push-конверт до `data`
   - Вернуть массив сообщений

6. `Отключиться(Клиент)`:
   - Закрыть WebSocket-соединение

**Проверка:** Роутер подключается к lobby, подписывается, получает hello от Node.js-клиента.

**Платформенный API (8.3.27):** `WebSocketКлиентСоединения.ОткрытьСоединение(Ключ, URL, Обработчики)` — **асинхронный** с обработчиками событий. Обработчики задаются через `ОбработчикиWebSocketКлиентСоединения`:
- `ОбработчикОткрытияСоединения` — вызывается при установке соединения
- `ОбработчикПолученияСообщения` — вызывается при получении каждого WS-фрейма
- `ОбработчикОшибки` — вызывается при ошибке
- `ОбработчикЗакрытияСоединения` — вызывается при закрытии

Модуль обработчиков = `ОбщийМодуль` (можно указать `ЕХТ_Центрифуга_ВебСокетКлиент`). Это значит, что API модуля — **событийный**, а не поллинговый (вместо `ПолучитьСообщения` — коллбэк `ОбработчикПолученияСообщения`).

Метод `ОтправитьСообщение(Данные)` у `WebSocketКлиентСоединение` — для отправки (publish, subscribe, ping).

### Этап 5. Форма настроек + UX

2. Форма элемента справочника:

## Граф зависимостей

```
Справочник ЕХТ_Центрифуга (реквизиты)
    │
    ▼
ЕХТ_Центрифуга_Конфигурация (чтение настроек)
    │
    ├──────────────────┐
    ▼                  ▼
ЕХТ_Центрифуга     ЕХТ_Центрифуга_ВебСокетКлиент
(Server API + JWT)  (WebSocket-клиент)
```

Этапы 1 → 2 → 3 строго последовательные (каждый зависит от предыдущего).
Этап 4 зависит только от Этапа 1 (Конфигурация), можно параллельно с 2-3.
Этап 5 — после всех остальных.

## Тестирование

Для каждого этапа — ручная проверка с запущенным Centrifugo (`centrifugo/centrifugo.exe --config=config.json`).

Автотесты (Node.js, `centrifugo/test-*.mjs`) уже покрывают протокол Centrifugo. Для проверки BSL-реализации использовать их как reference: запускать Node.js-клиент как «Чат», а 1С-код как «Роутер», и проверять что они общаются.

## Решённые вопросы

1. **JWT** — платформенный `ТокенДоступа` (с 8.3.21), `АлгоритмПодписиТокенаДоступа.HS256`. Ручной HMAC/Base64URL не нужен.
2. **WebSocket** — `WebSocketКлиентСоединения.ОткрытьСоединение()` (8.3.27), асинхронный с обработчиками событий. Готовый паттерн в `Chat/МодульBridge` (175 строк).
3. **Справочник** — в скелете назван `ЕХТ_Центрифуга` (не `ЕХТ_Центрифуга_Серверы`). Спецификация EXT-CLAUDE.md уже обновлена.
