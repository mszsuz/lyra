# Ревью плана `PLAN-PROFILES-MCP`

Дата: 2026-03-11

## Вывод

План идёт в правильную сторону: он закрывает реальные пробелы текущего контура `Router -> centrifugo_stdio -> Claude` и хорошо делит работу на профили, MCP-relay и конфиги.

Но в текущем виде я бы не начинал реализацию с "шага 1". Сначала плану нужны несколько уточнений, потому что часть предпосылок уже расходится с кодом и документацией проекта.

## Находки

### [P1] Не зафиксирован единый source of truth для topology и путей

Что видно в проекте:

- Текущий Router реально работает через `pipe:<session_id>` и сам транслирует `pipe -> session`: `Router/ЕХТ_Лира_Роутер/src/CommonModules/ЕХТ_Лира_Роутер/Ext/Module.bsl:142`, `:144`, `:474-517`.
- `docs/lyra-architecture.md` описывает другой контур: `stdio-bridge` как более автономный адаптер, а не тот поток, который сейчас зашит в Router.
- `Router/CLAUDE.md` всё ещё содержит старое описание callback-пути через `LyraActiveChannel`.
- В плане путь к профилям задан абсолютным `C:\1ext.ru\projects\github.com\ЕХТ_Лира_Роутер\Профили\Основной\`, но в репозитории уже есть локальная папка `Router/ЕХТ_Лира_Роутер/Профили/Основной/`, и она сейчас пустая.

Почему это важно:

- Без явного решения, какая схема считается канонической, легко получить реализацию "по плану", а тесты и документация останутся на другом контуре.
- Абсолютный путь к профилям ломает воспроизводимость локальной разработки и CI.

Что бы я поправил:

- Добавить короткий ADR: какая topology каноническая, где живут профили, кто владеет MCP-relay.
- Для профилей выбрать один вариант:
  - repo-local `Router/ЕХТ_Лира_Роутер/Профили/...`,
  - либо внешний путь из настроек Router.
- После этого привести `docs/lyra-architecture.md`, `Router/CLAUDE.md` и сам план к одному описанию.

### [P1] Вариант Г недооценивает объём изменений в Chat

Что видно в проекте:

- Главная форма Chat сейчас подключена к простому протоколу `tool_call` / `tool_result`, а не к `mcp_jsonrpc`: `Chat/Lyra-Chat-multiform.epf/src/Lyra-Chat-multiform/Forms/Форма/Ext/Form/Module.bsl:435-451`.
- В `hello` Chat отправляет Router только имена инструментов: `.../Форма/Ext/Form/Module.bsl:694`.
- Полноценные JSON-RPC helper-методы в `МодульMCP` действительно есть: `.../Forms/МодульMCP/Ext/Form/Module.bsl:68`, `:140`.
- Legacy-схема `mcp_jsonrpc` описана в `Bridge/SPEC-MCP-1C.md` и поддержана в `Bridge/bridge.js`: `Bridge/SPEC-MCP-1C.md:119-126`, `Bridge/bridge.js:176`, `:406`.

Почему это важно:

- Формулировка "упростить МодульMCP" скрывает реальный объём работ. Для варианта Г менять придётся не только `МодульMCP`, но и form-level contract в Chat.
- Иначе получится половинчатая миграция: JSON-RPC код есть, но UI/Bridge-слой всё ещё ждёт `tool_call`.

Что бы я поправил:

- Добавить отдельный подпункт "миграция Chat-контракта":
  - новый тип события `mcp_jsonrpc`,
  - где и как хранить `tools_definitions`,
  - что делать с текущими `tool_call/tool_result`,
  - как работает reconnect и повторная выдача описаний инструментов.

### [P1] Пункт `env={CLAUDECODE:""}` не решает реальную проблему наследования окружения

Что видно в проекте:

- `centrifugo_stdio` умеет только добавлять env-переменные из config: `centrifugo/ЕХТ_Центрифуга/centrifugo_stdio/src/main.rs:58`, `:108`, `:272`.
- Удаления inherited env там нет.
- Текущий `claude-wrapper.mjs` решает проблему иначе: делает именно `delete env.CLAUDECODE`: `centrifugo/claude-wrapper.mjs:23-25`.

Почему это важно:

- Если вложенный Claude проверяет сам факт существования переменной, то пустая строка не поможет.
- В плане одновременно есть правильная идея "удалять `CLAUDECODE`" и слабое техническое решение "передать `CLAUDECODE: ""`".

Что бы я поправил:

- Сделать удаление `CLAUDECODE` явным изменением `centrifugo_stdio`, а не надеяться на пустое значение.
- Альтернатива: расширить schema config файла, например `unsetEnv: ["CLAUDECODE"]`.

### [P2] Текущий Router собирает config JSON строковой конкатенацией, и план резко повышает риск экранирования

Что видно в проекте:

- `ЗапуститьМодельДляСессии` сейчас пишет JSON-конфиг для `centrifugo_stdio` вручную строкой: `Router/ЕХТ_Лира_Роутер/src/CommonModules/ЕХТ_Лира_Роутер/Ext/Module.bsl:506`.
- План добавляет сразу несколько потенциально хрупких значений:
  - `--system-prompt-file <путь>`,
  - `--mcp-config <путь>`,
  - `--allowedTools <список>`,
  - `--settings {"disableAllHooks": true}`.

Почему это важно:

- Здесь легко получить трудноуловимый сбой на пути, кавычке или JSON-экранировании.
- Особенно на Windows-путях и при генерации временных файлов из 1С.

Что бы я поправил:

- Перед расширением набора аргументов внести технический шаг: "рефакторинг генерации config JSON через структурную сериализацию, без конкатенации строк".

### [P2] Открытые вопросы по Claude CLI нужно перевести в upfront-gates

Что видно в плане:

- Вопрос по wildcard в `--allowedTools`.
- Вопрос по `--dangerously-skip-permissions`.
- Подразумеваемая зависимость от поддержки `--system-prompt-file`, `--strict-mcp-config`, `--settings`.

Почему это важно:

- Эти флаги определяют не только запуск, но и сам формат `model.json`.
- Если хотя бы одно предположение окажется неверным, придётся переделывать схему профиля и Router-side сборку аргументов.

Что бы я поправил:

- Добавить "Этап 0 / spike" до создания финального формата профилей:
  - локально прогнать текущий установленный Claude CLI,
  - проверить wildcards,
  - проверить permission behavior для MCP tools,
  - проверить приём нужных флагов.

### [P2] Раздел тестирования слишком общий для изменения контракта между четырьмя компонентами

Что видно в проекте:

- В репозитории уже есть хорошие транспортные тесты: `centrifugo/test-centrifugo-stdio.mjs`, `centrifugo/test-chat-via-router.mjs`, `centrifugo/test-chat-claude.mjs`.
- В плане тестирование сведено к одному пункту "проверить полную цепочку".

Почему это важно:

- Здесь риски сидят на стыках:
  - profile -> temp files,
  - Router -> `centrifugo_stdio`,
  - `mcp:` channel -> Chat,
  - JSON-RPC `initialize/tools/list/tools/call`.
- Один e2e-тест не подскажет, в каком месте сломался контракт.

Что бы я поправил:

- Явно добавить контрактные тесты:
  - шаблонизация `system-prompt.md` и генерация temp files;
  - `centrifugo_stdio` в raw JSON-RPC сценарии без `type`;
  - Router proxy `mcp:<id> -> Chat -> mcp:<id>`;
  - полный цикл `initialize -> tools/list -> tools/call`;
  - регрессия текущего сценария `chat -> text_delta/thinking_delta -> assistant_end`.

## Что в плане сильное

- Направление с профилями хорошее: в текущем Router действительно нет системного промпта, MCP-конфига и model-level настроек: `Router/.../Module.bsl:474-517`.
- Идея вынести описания инструментов из hardcode тоже правильная: сейчас `tools/list` в Chat собирается руками в `МодульMCP`: `.../МодульMCP/Ext/Form/Module.bsl:140+`.
- Разделение `pipe:` и `mcp:` каналов выглядит чище текущей попытки решать всё одним transport-контуром.

## Как бы я пересобрал порядок работ

1. Зафиксировать topology и source of truth для профилей.
2. Сделать spike по фактическому поведению Claude CLI.
3. Рефакторнуть генерацию Router config в структурный JSON.
4. Доработать `centrifugo_stdio`:
   - удаление `CLAUDECODE`,
   - подтверждённый raw JSON-RPC режим или чётко описанный аналог.
5. После этого выбрать между вариантом Г и В.
6. Только затем менять контракт Chat и добавлять profile-driven tools definitions.
7. Финально закрыть всё e2e и контрактными тестами.

## Итог

План хороший по направлению, но пока ещё слишком оптимистичен по объёму миграции и по степени определённости runtime-контрактов.

Если коротко: сначала нужно зафиксировать topology, поведение Claude CLI и механику `centrifugo_stdio`, а уже потом делать профили и окончательно выбирать MCP-relay вариант.
