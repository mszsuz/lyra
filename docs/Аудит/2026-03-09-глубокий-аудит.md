# Глубокий аудит Lyra

Дата: 2026-03-09
Формат: статический аудит кода и конфигурации

Краткий вывод: текущая версия репозитория одновременно содержит критические security-риски, незавершенные stub-контуры и несколько блокирующих протокольных расхождений. В текущем виде систему нельзя считать безопасной для внешнего использования, а часть заявленного пользовательского флоу неработоспособна даже в MVP-сценарии.

## Ответ на аудит

Дата: 2026-03-09
Автор: Claude Opus 4.6 (по запросу владельца проекта)

### Общая оценка

Аудит качественный, находки по существу. Большинство P0/P1 подтверждены проверкой кода. Ниже — разбор по каждой находке с указанием статуса и плана.

### Контекст

Lyra находится в стадии **активной разработки** (pre-alpha). Сейчас идёт последовательная проверка шагов 1-14 архитектуры с немедленным исправлением ошибок. На момент аудита проверены и подтверждены шаги 1-3. Система **не предназначена для внешнего использования** в текущем виде — это dev-окружение на localhost.

### [P0] Секреты в репозитории

**Статус: принято с уточнением.**

Находка смешивает два разных класса данных:

1. **Lobby JWT** (зашиты в EPF и мобильное приложение) — это **не секреты авторизации**, а bootstrap-токены для входа в публичный pre-auth контур. Centrifugo требует токен для любого соединения, но lobby — общедоступная точка входа до начала авторизации. Lobby-токен даёт только: открыть соединение, опубликовать hello/register, получить персональный канал. Он не даёт доступа к каналам сессий, Server API или Admin UI. Подробнее — раздел «Модель безопасности lobby-токенов» в `lyra-architecture.md`.

2. **hmac_secret_key, http_api.key, admin credentials** — это действительно серверные секреты, которые не должны попадать в git перед внешним использованием.

Репозиторий приватный, всё работает на localhost. Для dev-стадии это осознанный компромисс.

**План:** перед любым внешним использованием — ротация серверных секретов (hmac, api key, admin), вынос в env/secret store, `config.json` в `.gitignore`. Lobby JWT можно пересобрать, но сама по себе их «утечка» не является компрометацией — это публичные пропуски в тамбур, а не ключи от квартиры.

**Приоритет:** серверные секреты — перед первым внешним деплоем. Lobby JWT — низкий приоритет (пересобрать при ротации hmac).

### [P0] Глобальный `LyraActiveChannel` — межсессионная утечка

**Статус: подтверждено, известное ограничение Фазы 1.**

Это задокументировано в `Router/CLAUDE.md`:

> Фаза 1 поддерживает **одну активную сессию** (для нескольких сессий потребуется маппинг bridge → channel).

Singleton-канал — осознанное упрощение для MVP одного пользователя. Multi-session изоляция запланирована: маппинг `process_id → channel` через регистр сведений или параметры callback.

**План:** при переходе к multi-session (Фаза 2) — передавать `session_id` в параметрах запуска stdio-bridge и в callback URL, маршрутизировать ответы по нему.

**Приоритет:** после завершения проверки шагов 1-14 текущего MVP.

### [P1] Мобильный клиент — subscribe vs publish

**Статус: частично верно.**

Аудит указывает, что клиент делает `newSubscription('mobile:lobby')`, а сервер запрещает subscribe. Но для `mobile:lobby` настроен `allow_publish_for_client: true` — клиенту нужен publish, не subscribe. Подписка на lobby мобильному не нужна (он не слушает lobby, он туда отправляет).

Для `session:` каналов клиент должен использовать авто-подписку через `channels` claim в JWT (что и реализовано в `connectToSession`). Ручной subscribe в `connectToLobby` — действительно лишний и будет отклонён сервером.

**План:** убрать ручной subscribe в `connectToLobby`, оставить только publish. В `connectToSession` всё корректно.

### [P1] Claude без системного промпта и MCP-конфига

**Статус: подтверждено, известное ограничение Фазы 1.**

Текущая реализация запускает `claude -p --output-format stream-json` — минимальный вариант для проверки стриминга. Системный промпт («ты — Лира»), MCP-конфиг (Vega, mcp-1c-docs), модель и permissions — запланированы.

**План:** добавить `--system-prompt`, `--mcp-config`, `--model`, `--permission-mode` при проверке шага 11. Промпт уже есть в `docs/prompts/lyra-analyst-prompt.md`.

### [P2] `ЕХТ_Центрифуга_ВебСокетКлиент.Подключиться()` — «сломан»

**Статус: некорректная интерпретация.**

JWT сохраняется в заголовок `centrifugo-jwt` при вызове `Подключиться()`. Этот заголовок передаётся платформой 1С при открытии WebSocket-соединения. Однако Centrifugo v6 не использует заголовки для аутентификации — аутентификация происходит через JSON-команду `connect` с токеном внутри.

`WS_ПриОткрытии` пуста **намеренно** — комментарий в коде: `// Для программного API connect отправляется через ОтправитьConnect()`. Это двухэтапный процесс: (1) платформа открывает WebSocket, (2) модуль-обработчик (например `ЕХТ_Лира_Роутер`) отправляет connect-команду с JWT.

JWT из заголовка `centrifugo-jwt` читается модулем-обработчиком в `ПриОткрытииСоединения` и отправляется как `{"id":1,"connect":{"token":"..."}}`. Это **работающий код** — подтверждено трейс-логами Centrifugo (клиент `server-Роутер` успешно аутентифицируется).

## Уточнение по спорному пункту: ЕХТ_Центрифуга_ВебСокетКлиент

Дата: 2026-03-09
Автор: Claude Opus 4.6

Замечание аудитора справедливо: **по коду самого API** (`Подключиться()` + `WS_ПриОткрытии`) путь от JWT до connect-команды не прослеживается. Объясню цепочку полностью.

### Как это работает

Платформа 1С 8.3.27 использует модель **WebSocket-клиент + модуль-обработчик**:

1. `ЕХТ_Центрифуга_ВебСокетКлиент.Подключиться()` — сохраняет JWT в заголовок `centrifugo-jwt` и вызывает платформенный `WebSocketКлиент.Подключить()` (или ранее `ОткрытьСоединение()`)
2. Платформа открывает TCP/WebSocket соединение и вызывает callback `ПриОткрытииСоединения` в **модуле-обработчике** (указанном в настройках WebSocket-клиента, заголовок `centrifugo-handler`)
3. Модуль-обработчик (например `ЕХТ_Лира_Роутер`) в своём `ПриОткрытииСоединения`:
   - Читает JWT из заголовка `centrifugo-jwt`
   - Формирует JSON `{"id":1,"connect":{"token":"<JWT>"}}`
   - Отправляет через WebSocket
4. Centrifugo получает connect-команду, проверяет JWT, аутентифицирует клиента

`WS_ПриОткрытии` в `ЕХТ_Центрифуга_ВебСокетКлиент` — пуста **намеренно**, потому что это НЕ модуль-обработчик. Это утилитный модуль, который предоставляет API. Логика connect живёт в модуле-обработчике конкретного потребителя.

### Доказательство из трейс-лога Centrifugo

```json
{"level":"trace","client":"b20bba59-...","command":"{\"id\":1,\"connect\":{\"token\":\"eyJ...\"}}","user":"","message":"<-in--"}
{"level":"debug","client":"b20bba59-...","user":"server-Роутер","message":"client authenticated"}
```

Клиент `server-Роутер` успешно аутентифицирован — connect-команда с JWT отправлена и принята.

### Вывод

API не сломан, но **разделение ответственности неочевидно** при статическом аудите. JWT сохраняется в одном модуле, а используется в другом (через заголовок как промежуточное хранилище). Это можно было бы задокументировать яснее в коде `ЕХТ_Центрифуга_ВебСокетКлиент`, например комментарием: «JWT будет прочитан модулем-обработчиком в ПриОткрытииСоединения».
