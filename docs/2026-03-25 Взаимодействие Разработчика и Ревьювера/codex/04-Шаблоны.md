# Шаблоны

## 1. Шаблон review-файла

```md
# Review: <scope>

Дата: YYYY-MM-DD

## Findings

### [P1|P2|P3] <title>

Файл: `<path:line>`

<что именно не так>

Следствие:

- <impact 1>
- <impact 2>

## Итог

- <что осталось открытым>
- <что закрыто>
```

## 2. Шаблон response-файла

```md
# Ответ на <REVIEW-N>

Дата: YYYY-MM-DD

## Статус findings

### [P1|P2|P3] <title> — ИСПРАВЛЕНО

Файл: `<path>`

Было:
<кратко>

Стало:
<кратко>

### [P2] <title> — ПРИНЯТО, ОТЛОЖЕНО

Почему:

- <причина>
- <граница scope>
```

## 3. Шаблон final-status

```md
# Final status: <scope>

Дата: YYYY-MM-DD

## Закрыто

- <finding>

## Отложено

- <finding>

## Остаточные риски

- <risk>

## Вывод

<готово / не готово / готово с ограничениями>
```

## 4. Нормы формулировок

Хорошо:

- `finding закрыт по коду, но response переоценивает охват`
- `это не новый scope, а уточнение старого finding`
- `это новый edge-case внутри текущей задачи`

Плохо:

- `всё исправлено` без ссылки на код;
- `вроде ок`;
- `наверное закрыто`.
