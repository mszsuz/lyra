# Взаимодействие Разработчика и Ревьювера

Дата: 2026-03-25

## Зачем этот пакет

Этот набор документов фиксирует рабочую модель взаимодействия между:

- Разработчиком, который меняет план или код;
- Ревьювером, который независимо проверяет результат;
- Оркестратором, который связывает проходы в воспроизводимый цикл.

Цель: превратить ручной цикл `review -> response -> re-review` в полуавтоматический или полностью автоматический процесс с понятными артефактами и правилами остановки.

## Что здесь лежит

- `01-Роли-и-контракт.md` — кто за что отвечает.
- `02-Цикл-и-артефакты.md` — как идут проходы, какие файлы рождаются, когда работа считается завершённой.
- `03-Контур-автоматизации.md` — как это запускать автоматически.
- `04-Шаблоны.md` — рекомендуемые шаблоны review/response/status файлов.

## Ключевая идея

Разработчик и Ревьювер не должны спорить в чате. Они должны обмениваться формальными артефактами:

- `REVIEW` фиксирует findings;
- `RESPONSE` фиксирует реакцию на findings;
- код и документы меняются отдельно;
- оркестратор принимает решение: следующий проход, завершение, либо эскалация.

## Наблюдение по прошедшему взаимодействию

Лучше всего сработал такой паттерн:

1. Ревьювер пишет краткий, точный review-файл.
2. Разработчик отвечает отдельным response-файлом и при необходимости меняет код.
3. Ревьювер проверяет не только код, но и сам response-файл: не переобещал ли он.
4. Каждый следующий проход должен либо:
   - закрыть finding,
   - оставить его открытым,
   - или перевести в осознанно отложенный scope.

Главный риск без формализации: бесконечные recheck-проходы, где findings переименовываются, но не теряют неоднозначность.
