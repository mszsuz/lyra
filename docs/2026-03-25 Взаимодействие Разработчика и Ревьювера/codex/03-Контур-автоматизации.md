# Контур автоматизации

## 1. Цель автоматизации

Автоматизация должна делать не «магический code review», а управлять понятным циклом:

1. взять текущую задачу;
2. запустить Разработчика;
3. получить ответ и изменения;
4. запустить Ревьювера;
5. сопоставить findings с прошлым проходом;
6. либо закрыть задачу, либо открыть следующий проход.

## 2. Минимальная модель состояния

Для каждой задачи оркестратор хранит:

- `task_id`
- `scope`
- `current_phase`
- `current_pass`
- `open_findings`
- `closed_findings`
- `deferred_findings`
- `artifacts`

Для каждого finding:

- `finding_id`
- `title`
- `severity`
- `kind`
- `file`
- `line`
- `pass_opened`
- `status`
- `linked_response_pass`
- `notes`

`kind`:

- `code`
- `plan`
- `response-doc`
- `residual-risk`

## 3. Рекомендуемая логика оркестратора

### Шаг 1. Старт

Оркестратор создаёт базовый пакет:

- task folder
- исходный plan/task file
- `REVIEW-1`

### Шаг 2. Проход Разработчика

Разработчику передаются:

- текущий scope;
- последний `REVIEW`;
- список ещё открытых findings;
- правило: отвечать поштучно.

Выход:

- изменения в коде/документах;
- `RESPONSE-N`.

### Шаг 3. Проход Ревьювера

Ревьюверу передаются:

- текущий код;
- предыдущий `REVIEW`;
- новый `RESPONSE`;
- открытые findings из state.

Ревьювер обязан ответить в терминах:

- `closed`
- `still_open`
- `new`
- `doc-overclaim`

### Шаг 4. Решение оркестратора

Если:

- `new` пусто,
- `still_open` пусто,
- `P1/P2` не осталось,

то задача переходит в `done`.

Иначе:

- создаётся следующий `REVIEW-(N+1)`;
- цикл повторяется.

## 4. Что важно автоматизировать обязательно

### Дедупликацию findings

Иначе один и тот же риск будет переоткрываться как «новый» в каждом проходе.

### Разделение code-fix и doc-fix

Иначе исправленный код будет выглядеть как незакрытая задача только из-за неточного response-файла.

### Проверку claims against code

Это обязательный тип проверки. В прошедшем взаимодействии он оказался очень полезным.

## 5. Что можно автоматизировать позже

- автогенерацию короткого `FINAL-STATUS`;
- подсветку diff только по файлам, относящимся к finding;
- автоматическую классификацию severity;
- построение сводной таблицы `finding -> status -> pass`.

## 6. Стоп-условия

Оркестратор должен останавливать цикл, если:

- три прохода подряд не дали содержательного движения;
- findings уже только `P3` и касаются wording;
- задача вышла за исходный scope;
- разработчик пометил пункт как deferred и это принято.

## 7. Практическое правило

Автоматизация должна считать нормой, что:

- код может быть уже исправлен;
- документ о нём может быть неточен;
- reviewer должен уметь закрыть кодовый finding и открыть документальный.

Именно это делает цикл устойчивым, а не бесконечным.
