# Автоматизация review-цикла

## Архитектура

```
Пользователь
    │
    ▼
Оркестратор (Claude Code / скрипт)
    │
    ├── Разработчик (Claude — пишет код, отвечает на findings)
    │   Контекст: план, код, открытые findings
    │   Выход: изменения в коде + RESPONSE-N.md
    │
    └── Ревьювер (Claude — другая сессия, read-only)
        Контекст: код (текущий), план, предыдущий REVIEW, RESPONSE
        Выход: REVIEW-N.md
```

Ключевое: **ревьювер не видит процесс разработки**. Он получает результат и проверяет его. Это обеспечивает независимость.

## Вариант 1: Ручной (текущий)

Как работало в задаче adapter-timeout:

1. Пользователь просит ревьювера (отдельная сессия Claude) проверить файл
2. Ревьювер пишет `REVIEW-N.md`
3. Пользователь передаёт файл разработчику (эта сессия)
4. Разработчик исправляет код, пишет `RESPONSE-N.md`
5. Repeat

**Плюсы:** полный контроль, пользователь видит каждый шаг.
**Минусы:** ручная передача файлов, пользователь — bottleneck.

## Вариант 2: Полуавтоматический (Claude Code agents)

Оркестратор — основная сессия Claude Code. Запускает ревьювера как subagent:

```js
// Псевдокод оркестратора
const reviewResult = await Agent({
  subagent_type: 'Plan',  // read-only
  prompt: `
    Ты — независимый ревьювер. Проверь реализацию.

    План: ${planFile}
    Код: ${changedFiles}
    Предыдущий review: ${prevReview}
    Response разработчика: ${response}

    Формат: docs/plan-adapter-timeout-REVIEW-TEMPLATE.md

    Проверь:
    1. Код соответствует плану
    2. Все error-paths: billing, history, UI
    3. Interrupt trace: от abort до полной остановки
    4. Claims в response соответствуют коду
    5. Resource cleanup: timers, readers, controllers
  `
});
```

**Плюсы:** один цикл = одна команда пользователя.
**Минусы:** subagent может не иметь полного контекста; ограничение на размер prompt.

## Вариант 3: Полностью автоматический (цикл)

```
while (true) {
  // 1. Разработчик исправляет
  const response = await developerAgent(openFindings, code);
  saveFile(`RESPONSE-${pass}.md`, response);

  // 2. Ревьювер проверяет
  const review = await reviewerAgent(code, response, prevReview);
  saveFile(`REVIEW-${pass + 1}.md`, review);

  // 3. Оркестратор решает
  const findings = parseFindings(review);
  const p1p2 = findings.filter(f => f.severity <= 2);

  if (p1p2.length === 0) break;           // готово
  if (pass >= maxPasses) break;            // лимит
  if (noProgress(findings, prevFindings)) break; // застряли

  pass++;
}

saveFile(`FINAL-STATUS.md`, summarize(allFindings));
```

**Плюсы:** полная автоматизация, можно запустить и уйти.
**Минусы:** риск бесконечного цикла на doc-only findings; стоимость токенов; без пользователя нет решений по ОТЛОЖЕНО vs ИСПРАВЛЕНО.

## Рекомендация

Начать с **Варианта 2** — полуавтоматический. Причины:

1. Пользователь остаётся в цикле для решений (ОТЛОЖЕНО / вне scope)
2. Ревьювер запускается одной командой, не нужно переключать сессии
3. Контекст передаётся через prompt, не через файлы
4. Можно постепенно перейти к Варианту 3, добавив цикл

## Практические советы из опыта

### Ревьюверу нужен доступ к реальному коду

Ревьювер adapter-timeout был эффективен потому что читал `billing.mjs`, `server.mjs`, адаптеры — а не только план. Абстрактный review по описанию даёт P3 findings. Review по коду даёт P1.

### Разделять plan-review и impl-review

6 + 5 раундов — потому что фазы разные. Plan-review ловит архитектурные проблемы. Impl-review ловит runtime-баги. Смешивать их в одном цикле — путаница.

### Лимит на раунды

В нашей задаче: 6 раундов плана (ОК — архитектурные решения), 5 раундов реализации (последние 2 — по сути doc-only). Рекомендация: **5 раундов на фазу**, после чего оркестратор требует FINAL-STATUS даже если есть P3.

### Стоимость

11 раундов ревью ≈ 11 вызовов модели (каждый ~5-15К токенов input). При Sonnet это ~$0.15-0.50 за полный цикл. При Opus — $1-3. Приемлемо для задач уровня adapter-timeout. Для тривиальных задач — overkill.

### Когда НЕ нужен review-цикл

- Однофайловые правки без стыков
- Конфиг-изменения
- Документация
- Рефакторинг с тестами (тесты = ревьювер)

### Когда НУЖЕН review-цикл

- Изменения затрагивают 3+ файлов
- Есть error-paths / interrupt-paths
- Billing / security / auth
- Новый протокол между компонентами
- Нет тестов (как в нашем случае)
