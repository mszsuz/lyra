# Что ломалось

## 1. Псевдокод в плане расходился с контрактом

Самая частая проблема: верхняя часть плана описывала правильный контракт (`user_abort` отделён от `adapter_timeout`), а детальные snippets ниже по тексту реализовывали старое поведение.

Пример (RECHECK-2): раздел «Разделение abort-причин» описывает три канонических механизма, а snippet `openai.mjs` ниже маппит любой `AbortError` в `adapter_timeout`.

**Урок:** при итеративном обновлении длинного документа snippets в середине текста отстают от контракта в начале. Ревьювер должен проверять **согласованность** всех snippets с контрактом, а не только сам контракт.

## 2. Billing при аварийных выходах — слепая зона

Три раунда подряд (RECHECK-3, 4, 5) находили варианты потери `accumulatedCostUsd`:

- RECHECK-3: `billAccumulatedCost` с `_internal: true` → billing отбрасывает
- RECHECK-4: helper вызывает несуществующую `billingProcessEvent` вместо `processEvent`
- RECHECK-5: при аварийной ветке теряется `event.cost_usd` последнего запроса

**Урок:** каждый `return` в error-path — это потенциальная потеря billing. Ревьювер должен проверять **все пути выхода** из функции на предмет ресурсов (billing, history, state cleanup).

## 3. Abort-path — бесконечный источник стыковых багов

5 раундов implementation review — и каждый находил новый аспект abort:

1. `_currentAbort` уже null во время executeTool
2. `handleAbort` публикует terminal event, но loop продолжает
3. Новое сообщение не ставит `_aborted`
4. Tool history добавляется после abort
5. `tool_status` публикуется мимо guard

**Урок:** interrupt-path — это cross-cutting concern. Его нельзя проверить, глядя на одну функцию. Нужен **сквозной trace** от точки входа abort до полной остановки всех побочных эффектов.

## 4. Разработчик иногда исправлял finding, не закрывая причину

Пример: добавление `_aborted` check перед `executeTool` (IMPL-REVIEW-2) — но не после `await executeTool()`. Следующий раунд нашёл что tool_result всё равно утекает в историю.

**Урок:** ревьювер должен проверять не «исправлен ли конкретный симптом», а «закрыта ли вся поверхность атаки данного finding».

## 5. Документальные findings затягивают цикл

IMPL-REVIEW-5 (P3) — единственное замечание к формулировке response-файла. Код уже был ОК. Но цикл продолжился ещё на один раунд.

**Урок:** нужно разделять `code-blocking` и `doc-only` findings. Doc-only не должны блокировать завершение цикла, если код ready. Можно исправлять документацию в том же раунде без нового review.
