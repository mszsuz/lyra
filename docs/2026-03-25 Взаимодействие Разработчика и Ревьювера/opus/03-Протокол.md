# Протокол review-цикла

## Две фазы

### Фаза 1: Review плана (до написания кода)

```
PLAN → REVIEW-1 → правки плана → REVIEW-2 → ... → REVIEW-N (нет findings) → план утверждён
```

Что проверяет ревьювер плана:
- Согласованность контракта (error types, retry policy, state management)
- Scope: что включено, что исключено, почему
- Все snippets соответствуют контракту в начале документа
- Каждый error-path имеет определённое поведение (billing, history, UI)
- Passthrough vs managed — явное разделение

Критерий завершения: **нет P1/P2 findings**.

### Фаза 2: Review реализации (после написания кода)

```
Код написан → IMPL-REVIEW-1 → правки кода + RESPONSE-1 → IMPL-REVIEW-2 → ... → IMPL-REVIEW-N (нет findings)
```

Что проверяет ревьювер реализации:
- Код соответствует утверждённому плану
- Все error-paths корректно обрабатывают ресурсы (billing, history, state)
- Interrupt-path: сквозной trace от abort до полной остановки
- Claims в response-файлах соответствуют коду
- Нет cross-attempt state bleed (mutable state на инстансе)

Критерий завершения: **нет P1/P2 findings по коду**. P3 (doc-only) не блокируют.

## Формат finding

```
### [P1|P2|P3] Краткий заголовок

Файл: `path/file.mjs:line`

Что не так (1-3 предложения).

Следствие:
- конкретный негативный эффект

Что бы я поправил:
- конкретное предложение (опционально — code snippet)
```

Правила:
- Один finding = один дефект
- Severity по влиянию: P1 — ломает основной сценарий, P2 — edge-case/неполнота, P3 — документация/стиль
- `Файл:строка` обязательна
- `Следствие` — не «может сломаться», а «сломается когда X»

## Формат response

```
### [P1] Заголовок — ИСПРАВЛЕНО | ПРИНЯТО, ОТЛОЖЕНО | НЕ СОГЛАСЕН

Файл: `path/file.mjs`

Было: ...
Стало: ...
```

Правила:
- Отвечать на каждый finding поштучно
- ИСПРАВЛЕНО — обязательно указать где и что изменено
- ОТЛОЖЕНО — обязательно указать почему и границу scope
- Не утверждать больше, чем сделано в коде

## Обязательные проверки ревьювера

### По плану

1. **Контракт ↔ snippets** — все code examples в плане реализуют заявленный контракт
2. **Error-paths** — каждый `return` в error-ветке: что с billing, history, UI notification
3. **Scope boundaries** — что явно исключено и почему
4. **State management** — mutable state на инстансе vs локальные переменные

### По реализации

1. **Plan compliance** — код делает то, что утверждено в плане
2. **Interrupt trace** — от точки входа abort до полной остановки (все publish, history, billing подавлены)
3. **Claims verification** — response утверждает X, код действительно делает X
4. **Resource cleanup** — все пути выхода: billing, timers (clearTimeout), readers (releaseLock), abort controllers
5. **Cross-component** — вызывающий код (server.mjs) совместим с контрактом адаптера

## Стоп-условия

Цикл завершается когда:
- Последний review не содержит P1/P2
- Или 3 раунда подряд только P3 (doc-wording)
- Или findings вышли за scope задачи → фиксируются в бэклоге, цикл завершается
