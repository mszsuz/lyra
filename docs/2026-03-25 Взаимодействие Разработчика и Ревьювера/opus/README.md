# Взаимодействие Разработчика и Ревьювера — наблюдения из практики

Дата: 2026-03-25
Задача: adapter-timeout (6 раундов плана + 5 раундов реализации)

## Что здесь

Практические выводы из 11 раундов review-цикла на реальной задаче. Не теоретическая модель, а зафиксированные паттерны, которые работали и которые ломались.

- `01-Что-сработало.md` — паттерны, которые давали прогресс
- `02-Что-ломалось.md` — антипаттерны, на которые мы наступили
- `03-Протокол.md` — минимальный протокол для автоматизации
- `04-Автоматизация.md` — как это запускать без человека в цикле

## Ключевое наблюдение

Главная ценность ревьювера — не поиск багов в коде (линтер справится), а **верификация стыков**: что происходит когда компонент A обещает одно, компонент B ожидает другое, а документация утверждает третье. В нашем случае 80% findings были именно про стыки: abort vs timeout, billing vs error exits, plan vs pseudocode vs implementation.

## Сравнение с codex-версией

Codex дал хорошую формальную модель (роли, контракт, шаблоны). Моя версия дополняет её практикой: какие проверки давали реальные баги, а какие были шумом. Обе версии совместимы — протокол из `03-Протокол.md` можно использовать вместе с шаблонами из `codex/04-Шаблоны.md`.
