# Что сработало

## 1. Ревьювер проверяет стыки, а не синтаксис

Самые ценные findings были на стыках компонентов:

| Раунд | Finding | Суть стыка |
|-------|---------|-----------|
| AUDIT | retry привязан к whole tool loop | transport retry vs semantic turn — разные циклы |
| RECHECK | user_abort и adapter_timeout неразличимы | один AbortController, две разных причины |
| RECHECK-2 | chunk-timeout описан двумя механизмами | AdapterTimeoutError vs _abortReason — конкуренция |
| IMPL-3 | tool history leaks после abort | executeTool уже выполнен, а _aborted проверяется до |
| IMPL-4 | tool_status публикуется мимо guard | прямой apiPublish vs handleAdapterEvent |

Паттерн: **ревьювер эффективен когда проверяет flow целиком**, а не отдельные функции.

## 2. Claims-vs-code верификация

Ревьювер проверял не только код, но и утверждения в response-файлах:

- RECHECK-3: `billAccumulatedCost` с `_internal: true` — plan обещает списание, а `billing.mjs` отбрасывает `_internal`
- IMPL-5: response утверждает «подавляются все события», а `tool_status` публикуется напрямую

Это отдельный класс проверки: **документ обещает больше, чем код делает**. Без неё баги в response-файлах накапливаются и вводят в заблуждение в следующих раундах.

## 3. Severity работает

P1/P2/P3 — не формальность. На практике:

- **P1** (6 штук) — каждый раз реальный баг или архитектурное расхождение. Все потребовали изменения кода.
- **P2** (8 штук) — edge-cases и неполнота. Половина исправлена, половина отложена с обоснованием.
- **P3** (1 штука) — документальная неточность. Исправлена заодно с кодом.

Правило: **P1 блокирует, P2 обсуждается, P3 по желанию**.

## 4. Каждый finding — один дефект

Ревьювер ни разу не смешал два дефекта в один абзац. Это позволяло:
- отвечать поштучно (ИСПРАВЛЕНО / ОТЛОЖЕНО)
- отслеживать закрытие по номерам
- не спорить о severity составного finding

## 5. Готовые правки в review

Начиная с RECHECK-3 ревьювер давал не только findings, но и готовые code snippets для исправления. Это ускоряло цикл — разработчик мог взять и вставить, а не додумывать намерение ревьювера.

## 6. Фаза плана отдельно от фазы реализации

6 раундов плана выловили архитектурные проблемы **до** написания кода:
- retry per-session vs per-request
- passthrough адаптеры в scope / вне scope
- connect-timeout без retry
- maxTurns съедается retry

Без этих раундов все 6 проблем вылезли бы в code review и потребовали бы переписывания.
