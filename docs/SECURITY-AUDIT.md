# Lyra — аудит безопасности

**Дата:** 2026-03-20
**Контекст:** EPF уйдёт в открытый доступ, код полностью виден злоумышленнику.
**Области:** Chat EPF, Router (server.mjs), Centrifugo config, мобильное приложение.

---

## Критические

### SEC-01. JWT mobile_jwt передаётся на внешний API (api.qrserver.com)

**Файл:** `Chat/.../МодульСессия/Module.bsl:510-512`

QR-код генерируется через внешний сервис `api.qrserver.com` — JWT передаётся в URL query parameter. Любой перехват (логи CDN, кэш, история) даёт `mobile_jwt` = доступ к каналу сессии.

**Рекомендация:** Генерировать QR локально (библиотека 1С или встроенный генератор). Полностью убрать внешний API.

---

### SEC-02. Выполнить() / Вычислить() — произвольное выполнение кода 1С

**Файл:** `Chat/.../МодульОбработка/Module.bsl:330, 356`

Инструменты `lyra_data_eval` и `lyra_data_exec` принимают код от Claude и выполняют через `Вычислить()` / `Выполнить()`. Даже в безопасном режиме позволяют: чтение любых данных базы, доступ к системным объектам, потенциальная модификация данных.

Вектор: prompt injection → Claude генерирует вредоносный код 1С → код выполняется на сервере пользователя.

**Рекомендация:**
- Удалить `lyra_data_exec` и `lyra_data_eval` из продакшена
- Если нужны — реализовать whitelist разрешённых операций, не произвольный код
- Как минимум: подтверждение пользователем перед каждым выполнением

---

### SEC-03. Секреты в git (hmac_secret_key, api_key, admin password)

**Файлы:** `centrifugo/config.json`, `centrifugo/test-full-flow.mjs`

Все ключи Centrifugo скомичены в репо:
- `hmac_secret_key` — позволяет подделывать JWT для любого канала
- `http_api.key` — полный доступ к Server API (publish/subscribe/disconnect)
- `admin.password` + `admin.secret` — вход в Admin UI

**Рекомендация:**
- Ротировать все секреты немедленно
- Хранить в переменных окружения или .env (не в git)
- Добавить `centrifugo/config.json` в `.gitignore`, использовать `config.example.json`
- Очистить историю git (`git filter-repo`)

---

### SEC-04. verifyAuth() — MVP-заглушка, принимает любой user_id

**Файл:** `Router/users.mjs:18-26`

Функция `verifyAuth(userId, deviceId)` безусловно возвращает `{ok: true}` и создаёт пользователя на лету. Атакующий может отправить `auth` с чужим `user_id` → получить доступ к чужой сессии.

**Рекомендация:** Реализовать проверку пары (user_id, device_id) в MDM. Отклонять если пара не зарегистрирована при SMS-регистрации.

---

### SEC-05. namespace `pipe:` — клиенты могут publish

**Файл:** `centrifugo/config.json:38-42`

`allow_publish_for_client: true` для namespace `pipe:`. Атакующий может отправить поддельные сообщения в `pipe:<session_id>` (fake assistant_end, fake tool_result). Роутер примет их как настоящие от centrifugo_stdio.

**Рекомендация:** `allow_publish_for_client: false`. Публикация только через Server API.

---

### SEC-06. namespace `service:` — открыт для всех клиентов

**Файл:** `centrifugo/config.json:43-47`

`allow_subscribe_for_client: true` + `allow_publish_for_client: true`. Любой клиент может:
- Подписаться на `service:events` → видеть session_id/user_id всех сессий
- Публиковать fake `event_billing_balance_checked` → заблокировать чужую сессию

**Рекомендация:** Оба флага в `false`. Серверные компоненты работают через Server API.

---

### SEC-07. Wildcard CORS в Centrifugo

**Файл:** `centrifugo/config.json:14`

`allowed_origins: ["*"]` — подключение с любого домена. CSRF: злоумышленный сайт может установить WebSocket-соединение с Centrifugo от имени браузера жертвы.

**Рекомендация:** Указать конкретные домены. Для dev: `["http://localhost:8300"]`.

---

## Высокие

### SEC-08. Нет rate limiting на регистрацию (SMS flood)

**Файл:** `Router/server.mjs:439-464`

`handleMobileRegister` не проверяет частоту запросов по номеру телефона. Атакующий может:
- Отправить тысячи register → исчерпать память (_pendingRegistrations)
- При реальной SMS-отправке → flood жертвы SMS-ами

**Рекомендация:** Rate limit по номеру телефона (1 SMS / 15 мин). Валидация формата номера (`/^\+?[0-9]{10,15}$/`).

---

### SEC-09. 4-значный код подтверждения — brute force

**Файл:** `Router/server.mjs:453`

Код: 4 цифры (9000 комбинаций), 3 попытки на reg_id, но можно запрашивать новый reg_id неограниченно → 3 × N попыток.

**Рекомендация:**
- Увеличить до 6 цифр (900 000 комбинаций)
- Глобальный лимит попыток на номер (не только на reg_id)
- Задержка между попытками (500мс)

---

### SEC-10. Tool call HTTP endpoint без авторизации

**Файл:** `Router/tools.mjs:36-62`

`POST /tool-call` проверяет только существование session_id, но не авторизацию вызывающего. Любой процесс на localhost может отправлять tool call.

**Рекомендация:** Генерировать session-специфичный bearer token при спавне Claude CLI, требовать его в Authorization header.

---

### SEC-11. Нет whitelist инструментов на стороне Чата

**Файл:** `Chat/.../МодульЧат/Module.bsl:431`

Чат выполняет любой tool_call от Claude без фильтрации. Prompt injection может заставить Claude вызвать опасные инструменты.

**Рекомендация:** Whitelist допустимых инструментов в Чате. Опасные (`lyra_data_exec`, `lyra_data_eval`) — только с подтверждением пользователя.

---

### SEC-12. Ошибки раскрывают внутренние детали

**Файлы:** `Chat/.../МодульОбработка/Module.bsl`, `Router/centrifugo.mjs`

Исключения возвращаются Claude без фильтрации: имена таблиц, пути, структура конфигурации. Claude пересказывает ошибки пользователю → информация утекает.

**Рекомендация:** Возвращать обобщённые сообщения («Ошибка выполнения запроса»). Детали — только в серверный лог.

---

### SEC-13. Регистрационный код логируется в открытом виде

**Файл:** `Router/server.mjs:460`

```
📱 REGISTRATION CODE for +79001234567: 4521 (reg_id=abc)
```

При утечке логов — полный доступ к регистрации любого номера.

**Рекомендация:** Не логировать код. Логировать только факт отправки: `SMS sent to ...4567`.

---

### SEC-14. Admin API Centrifugo включён

**Файл:** `centrifugo/config.json:16-20`

Admin UI с паролем доступен на порту Centrifugo. Пароль скомичен в git.

**Рекомендация:** `"enabled": false` для production. Или скрыть за firewall / VPN.

---

### SEC-15. Нет TLS/SSL

**Файл:** `centrifugo/config.json`

WebSocket работает по `ws://` (не `wss://`). Весь трафик (JWT, сообщения, данные базы) передаётся открытым текстом.

**Рекомендация:** TLS через reverse proxy (Caddy/Nginx) или нативно в Centrifugo. Обязательно для production.

---

### SEC-16. Токен Напарника (code.1c.ai) передаётся стороннему API

**Файл:** `Chat/.../МодульНапарник/Module.bsl:140-155`

Bearer-токен пользователя отправляется на code.1c.ai. При компрометации стороннего сервиса — утечка токена.

**Рекомендация:** Прокси через собственный сервер. Или предупреждать пользователя о передаче данных третьей стороне.

---

## Средние

### SEC-17. Нет аудит-лога tool call

Нет записи о том, какие инструменты вызвал Claude, с какими параметрами, какой результат получен. При инциденте невозможно понять, что произошло.

**Рекомендация:** Логирование каждого tool_call (имя, параметры, результат, timestamp) в отдельный audit log.

---

### SEC-18. Нет rate limiting на tool calls

Одна сессия может вызывать инструменты без ограничений → исчерпание ресурсов сервера 1С.

**Рекомендация:** Лимит на стороне Роутера: N вызовов в минуту на сессию.

---

### SEC-19. Настройки из hello_ack не валидируются

**Файл:** `Chat/.../МодульСессия/Module.bsl:328-343`

Настройки от Роутера (user_name, user_level) принимаются без проверки типов и значений.

**Рекомендация:** Whitelist допустимых ключей + валидация типов.

---

### SEC-20. Path traversal в userDir()

**Файл:** `Router/users.mjs:40-43`

Фильтрация `userId` неполная (убирает `/\` и leading dots, но не все варианты). Node.js `resolve()` защищает на практике, но паттерн ненадёжный.

**Рекомендация:** Whitelist: `/^[a-zA-Z0-9_-]+$/`.

---

### SEC-21. Нет timeout активности на стороне Чата

Если JWT украден, сессия остаётся активной бессрочно (пока Роутер не истечёт по TTL 30 мин).

**Рекомендация:** Client-side timeout при отсутствии активности (15-30 мин) → автоматическое отключение.

---

### SEC-22. Lobby JWT с долгим сроком жизни

**Файлы:** `МодульТранспорт/Module.bsl:51`, `centrifugo_config.dart:8`

Общие JWT для lobby живут до 2027+ года. Если скомпрометирован — работает годами.

**Рекомендация:** Уменьшить TTL до 30 дней. Обновлять при каждом запуске / при старте приложения с сервера.

---

### SEC-23. WebSocket-сообщения не валидируются по структуре

**Файл:** `Chat/.../МодульТранспорт/Module.bsl:337-386`

JSON от Centrifugo парсится без проверки обязательных полей, типов, размера.

**Рекомендация:** Валидация размера (отклонять > 1MB), обязательных полей, типов значений.

---

## Низкие

### SEC-24. Нет CSRF-защиты на tool call endpoint

**Файл:** `Router/tools.mjs:14-23`

CORS: `Access-Control-Allow-Origin: *`. Сейчас `127.0.0.1` — защищено. Но при изменении конфигурации на `0.0.0.0` станет уязвимо.

**Рекомендация:** Явный whitelist в CORS. Binding строго на `127.0.0.1`.

---

### SEC-25. Memory hint response пишется в историю

**Файл:** `Router/server.mjs:596-605`

Подавленные ответы Claude записываются в файл истории. При утечке файлов — информационное раскрытие.

**Рекомендация:** Помечать `_hidden: true` или не записывать вовсе.

## Критические (дополнение — углублённый аудит по акторам)

### SEC-26. Reconnect hijack через form_id

**Файл:** `Router/server.mjs:228-262`

Если атакующий узнал `form_id` чужой сессии (из логов, перебора, трафика), он отправляет hello с этим form_id → Роутер находит живую сессию → выдаёт новый `chat_jwt` для неё → атакующий получает полный доступ к чужой сессии.

Роутер перезаписывает `clientId` при reconnect, не проверяя, что новый клиент — тот же, что и старый.

**Рекомендация:** При reconnect проверять дополнительный секрет (nonce из hello_ack) или clientUUID.

---

### SEC-27. Спуфинг register_ack / sms_sent от мобильного клиента

**Файл:** `centrifugo/config.json` (mobile: allow_publish_for_client: true)

Мобильное приложение может **публиковать** в mobile:lobby. Атакующий отправляет fake `register_ack` с чужим `reg_id` и своим `user_id` → жертва (если не фильтрует по reg_id) сохранит чужой user_id.

Текущая защита: Flutter-клиент фильтрует по phone/reg_id. Но модифицированный клиент может это обойти.

**Рекомендация:** `allow_publish_for_client: false` для mobile:. Мобильное публикует через Server API (proxy) или отдельный канал для исходящих.

---

### SEC-28. Утечка номеров телефонов через mobile:lobby

**Файл:** `Router/server.mjs:463`

`sms_sent` публикуется в shared mobile:lobby с полным номером телефона. ВСЕ мобильные клиенты видят номера всех регистрирующихся пользователей. Нарушение конфиденциальности персональных данных.

**Рекомендация:** Не включать phone в sms_sent. Или публиковать только через Server API subscribe для конкретного клиента (аналогично hello_ack).

---

### SEC-29. DoS: массовый спам hello без ограничений

**Файл:** `Router/server.mjs:152-170`

Дедупликация hello по `clientUUID` с TTL 10 секунд. Обходится: новое WS-соединение = новый clientUUID. 1000 соединений × 1 hello = 1000 сессий + 1000 Claude CLI процессов → OutOfMemory.

**Рекомендация:**
- Глобальный лимит одновременных сессий (max 100-500)
- Rate limit по IP (max 5 hello / мин)
- Не спавнить Claude до auth_ack ok

---

### SEC-30. Самоавторизация без мобильного (bypass QR)

**Файл:** `Router/server.mjs:340-383` + `users.mjs:18-26`

Неавторизованный атакующий: извлекает lobby JWT из EPF → connect → hello → получает session_id + chat_jwt + mobile_jwt → подключается с chat_jwt → отправляет `{type: "auth", user_id: "any-uuid", device_id: "any"}` → verifyAuth() возвращает ok → сессия active → Claude спавнится.

Полный bypass авторизации без мобильного приложения.

**Рекомендация:** SEC-04 (verifyAuth) + проверять, что auth пришёл от mobile_jwt (sub начинается с `mobile-`), а не от chat_jwt.

---

### SEC-31. Мобильное спуфит сообщения Claude (assistant_end, tool_call) на канале сессии

**Файл:** `centrifugo/config.json` (session: allow_publish_for_client: true)

После авторизации мобильное подключено к session:\<id\> и может публиковать любой JSON. Чат EPF не проверяет источник сообщений — отобразит fake assistant_end от мобильного как настоящий ответ Claude. Хуже: fake tool_call заставит Чат выполнить запрос к базе 1С.

**Рекомендация:** Роутер подписывает свои сообщения (HMAC). Чат проверяет подпись. Или: разделить каналы (session:\<id\>:upstream и session:\<id\>:downstream).

---

### SEC-32. Модификация tool_result пользователем перед отправкой

**Файл:** `Router/tools.mjs:122-134`

Чат получает tool_call, выполняет в 1С, отправляет tool_result. Модифицированный EPF может подменить результат → Claude получит ложные данные → неверные ответы / решения.

**Рекомендация:** Подписание tool_result (HMAC с session secret) или контрольная сумма от параметров запроса.

---

### SEC-33. Мобильное может отправить chat вместо media → Claude выполнит

**Файл:** `Router/server.mjs:193`

Роутер обрабатывает `{type: "chat"}` от любого клиента на канале сессии, не проверяя источник (Chat vs Mobile). Мобильное может отправить полноценный chat-запрос → Claude выполнит tool calls на сервере 1С.

**Рекомендация:** Проверять sub из JWT (chat- vs mobile-) через pub.info или разделить каналы.

---

### SEC-34. Одна auth не отменяет другую (race condition)

**Файл:** `Router/server.mjs:340-383`

Если два клиента отправят auth на одну сессию — оба пройдут, последний перезапишет userId. Нет блокировки повторной auth.

**Рекомендация:** После первого успешного auth → отклонять последующие: `if (session.userId) return already_authorized`.

---

### SEC-35. get_sessions не проверяет владельца

**Файл:** `Router/server.mjs:518-554`

`handleGetSessions` принимает `user_id` из сообщения и возвращает список сессий. Не проверяет, что запрашивающий — владелец этого user_id. Атакующий может перебирать UUID и получать списки чужих сессий.

**Рекомендация:** Проверять, что user_id совпадает с авторизованным пользователем текущего соединения.
