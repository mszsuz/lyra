## Функциональность

### 1. Редактор кода (BSL Console / Monaco Editor)

Полноценный редактор кода BSL на базе Monaco Editor, загружаемый в ПолеHTML формы 1С.

**Возможности:**
- Подсветка синтаксиса BSL и языка запросов
- Автодополнение (метаданные конфигурации, общие модули, пользовательские сниппеты)
- Hover-подсказки с реальными значениями переменных после выполнения кода
- Выполнение кода на клиенте и на сервере с перехватом переменных
- Конструктор запросов и конструктор форматной строки
- Diff-сравнение версий кода (diff2html)
- Темы оформления, карта кода, настройки шрифта
- Сохранение/загрузка из файлов .bsl

**Инициализация Monaco:**
Исходники упакованы в макет `src` (ZIP). При открытии:
1. ZIP извлекается во временный каталог (тонкий клиент) или загружается с GitHub (веб-клиент)
2. Создаётся `index.html` с уникальным именем (для обхода кеширования)
3. Доступ к Monaco через `Элементы.ПолеHTML.Документ.defaultView`

Цепочка инициализации — 8 асинхронных колбэков через `ОписаниеОповещения`.

### 2. Чат с ИИ

**Протокол:** REST API с Bearer-токеном через `HTTPСоединение`.

**Цикл работы:**
1. Пользователь вводит промпт + передаётся текущий код из редактора
2. POST на `/api/manager-1c/code-console/sessions/{uuid}/entries/`
3. Поллинг GET каждые 2 секунды — проверка наличия ответа
4. При получении ответа — обновление HTML чата
5. Таймаут ожидания: 300 секунд

**HTML чата** формируется на сервере в ObjectModule: полная HTML-страница с CSS, marked.js (markdown), highlight.js (подсветка кода). Хранится в реквизите формы `ИсторияЧата`.

**Блокировка UI:** Во время ожидания ответа блокируются поле промпта, кнопка "Отправить" и редактор кода (`setReadOnly`).

### 3. Выгрузка метаданных конфигурации

Полная выгрузка структуры конфигурации в XML (~1945 строк кода). Покрывает:

- Конфигурация (имя, синоним, версия, дата обновления)
- Примитивные типы (Число, Строка, Дата, Булево, ХранилищеЗначения, УИД)
- Общие реквизиты (с учётом автоиспользования и разделения данных)
- Справочники, документы, перечисления, ПВХ
- Регистры сведений, накопления, бухгалтерии, расчёта (опционально)
- Бизнес-процессы, задачи

**Для каждого объекта:**
- Имя, синоним, комментарий, тип
- Настраиваемые свойства (иерархичность, периодичность, подчинённость)
- Системные реквизиты (Код, Наименование, Родитель, Дата, Номер, Проведен и т.д.)
- Пользовательские реквизиты с типами (измерения, ресурсы, реквизиты)
- Табличные части и их реквизиты
- Значения перечислений
- Предопределённые элементы (через запрос к БД)
- Регистраторы (для регистров)
- Предопределённые ТЧ для ПВР (базовые, ведущие, вытесняющие)

**Процесс загрузки на сервер:**
1. XML-файл кодируется в base64
2. POST `/api/file-manager/files/` — загрузка файла
3. POST `/api/manager-1c/configs/` — создание конфигурации
4. Поллинг статуса: `indexing` → `active`

### 4. Проверка запросов (примитивный tool-use)

Единственный "инструмент" — проверка синтаксиса запросов 1С:
1. В цикле поллинга чата проверяется флаг `need_check`
2. GET `/api/manager-1c/query-console/checks/?status=pending` — получение задачи
3. 1С выполняет `Запрос.НайтиПараметры()` для валидации синтаксиса
4. PATCH `/api/manager-1c/query-console/checks/{id}/` — возврат результата

### 5. История версий кода

- GET `.../sessions/{uuid}/versions/` — список версий
- Отображение diff через diff2html (CDN)
- Тёмная тема для diff (background: #0f172a)

## REST API (aether-lab.ru)

### Endpoints

| Метод | Путь | Назначение |
|---|---|---|
| POST | `/api/file-manager/files/` | Загрузка файла (base64) |
| POST | `/api/manager-1c/configs/` | Создание конфигурации |
| GET | `/api/manager-1c/configs/{uuid}/` | Проверка конфигурации |
| GET | `/api/manager-1c/configs/` | Список конфигураций |
| POST | `.../code-console/sessions/` | Создание сессии |
| GET | `.../code-console/sessions/{uuid}/` | Проверка сессии |
| POST | `.../sessions/{uuid}/entries/` | Отправка сообщения |
| GET | `.../sessions/{uuid}/entries/` | История сообщений |
| POST | `.../sessions/{uuid}/clear/` | Очистка диалога |
| GET | `.../sessions/{uuid}/versions/` | История версий |
| GET | `.../sessions/{uuid}/versions/{ver}/` | Конкретная версия |
| GET | `.../query-console/checks/?status=pending` | Ожидающие проверки |
| PATCH | `.../query-console/checks/{id}/` | Завершение проверки |

### HTTP-транспорт

- `HTTPСоединение` + `ВызватьHTTPМетод()`
- SSL: `ЗащищенноеСоединениеOpenSSL()`
- Авторизация: `Authorization: Bearer {ТокенДоступа}`
- Ответы: `{Успех, Ответ, Ошибка}` — унифицированная структура
- Коды: 200/201/204 — успех, 401 — авторизация, 404 — не найдено

## Интересные паттерны

### Goto для опциональных свойств метаданных

В `ВыгрузитьОсновныеСвойства()` используется `Перейти ~Метка` для пропуска свойств, отсутствующих у объекта. Не самый чистый подход, но эффективный для рефлексии 1С, где набор свойств зависит от типа объекта.

### Кеширование типов и UUID

- `мСоответствиеИменТиповОбъектовУИ` — кеш UUID по имени типа
- `мПоследнееЗначениеТипа` / `мПоследнийТипДанных` — кеш последнего преобразования
- `мСоответствиеРегистровИРегистраторов` — ленивая инициализация

### Перехват переменных после выполнения кода

Имена переменных получаются через Monaco API (`View().getVarsNames()`). Для каждой генерируется:
```bsl
Попытка SPd201bf6.Вставить("ИмяПеременной", Вычислить("ИмяПеременной")); Исключение КонецПопытки;
```
Имена `RVS6e6c2a` и `SPd201bf6` — случайные, чтобы не конфликтовать с пользовательским кодом.

### Настройки с профилями по доменам

Два уровня: legacy (`AetherAIConsole`) и V2 (`AetherAIConsoleV2` — профили по нормализованному домену API-хоста). При чтении — сначала V2, затем legacy с автоматической миграцией.

### Передача current_code при каждом сообщении

При отправке промпта передаётся параметр `current_code` — текущий код в редакторе. ИИ видит контекст "над чем работает пользователь".

### Отложенная установка текста

Если Monaco ещё не инициализирован, текст сохраняется в `ОтложенныйКодДляРедактора` и применяется в обработчике `ConsoleOnReady`.

### Защита от повторных диалогов ошибок

Флаги `ДиалогОшибкиСессииПоказан` и `ДиалогОшибкиКонфигурацииПоказан` предотвращают множественное показывание ошибок при поллинге.

## Слабые стороны

### Поллинг вместо стриминга

Нет стриминга (SSE/WebSocket). Ответ ИИ получается целиком после завершения генерации. Задержка до 2 секунд между проверками. Таймаут 300 секунд блокирует серверный вызов.

### Монолитная архитектура

Весь код в двух файлах (2800 + 5600 строк). Нет разделения на модули. ObjectModule содержит два несвязанных блока: выгрузка метаданных (старый код с `Перейти`) и современный API-клиент.

### Генерация HTML строками в BSL

Функция `СформироватьHTMLЧата()` (180 строк) генерирует полную HTML-страницу конкатенацией строк. Хрупко и трудно поддерживать.

### Облачная зависимость

Весь ИИ-функционал зависит от сервиса aether-lab.ru. Нет локального режима. Требуется Bearer-токен (подписка).

### Один tool-вызов

Единственный "инструмент" — проверка синтаксиса запроса. Нет возможности выполнить произвольный код, получить метаданные по запросу, вычислить выражение.

### Безопасный режим отключён

`БезопасныйРежим = Ложь` — необходимо для HTTP-запросов, но открывает полный доступ к системе.

## Сравнение с Lyra-Chat

| Параметр | Aether Lab | Lyra-Chat |
|---|---|---|
| Связь с ИИ | REST API + поллинг (2 сек) | WebSocket + стриминг по буквам |
| Архитектура | Монолит (1 форма + ObjectModule) | Модульная (6 форм-модулей) |
| ИИ-сервис | aether-lab.ru (облако) | Claude Code (локально через Bridge) |
| Инструменты | 1 (проверка запроса) | 4 MCP (v8_query, v8_eval, v8_metadata, v8_exec) |
| Контекст конфигурации | Полная выгрузка МД в XML | MCP-инструмент v8_metadata (по запросу) |
| JSON на клиенте | Всё на сервере | ПрочитатьЗначениеJSON/ЗаписатьЗначениеJSON (8.3.23+) |
| Редактор кода | Monaco Editor (BSL Console) | Нет (чат-интерфейс) |
| Отображение ответа | HTML с markdown (полный после генерации) | Стриминг thinking + text |
| Стоимость | Подписка aether-lab.ru | API-ключ Anthropic |

## Что можно позаимствовать

1. **Выгрузка метаданных** — готовый код для полного контекста конфигурации (2800 строк). Может дополнить MCP-инструмент `v8_metadata`.

2. **Передача current_code** — при каждом сообщении передавать текущий контекст пользователя.

3. **Блокировка UI** — блокировка ввода во время генерации ответа.

4. **Унифицированный HTTP-клиент** — паттерн `{Успех, Ответ, Ошибка}` для REST-интеграций.

5. **Отложенная инициализация** — сохранение данных в реквизит, если HTML-компонент ещё не готов.
