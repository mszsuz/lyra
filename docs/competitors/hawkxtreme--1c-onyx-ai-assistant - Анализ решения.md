# Анализ конкурента: hawkxtreme / 1C Onyx AI Assistant

## 1. Общее описание

**1C Onyx AI Assistant** — расширение для 1С:Предприятие, интегрирующее AI-чат на базе платформы [Onyx](https://github.com/onyx-dot-app/onyx) (ранее Danswer). Onyx — open-source RAG-платформа (Retrieval-Augmented Generation) с поддержкой коннекторов к различным источникам данных. Решение ориентировано на **конечных пользователей 1С**, которым нужен AI-ассистент с доступом к базе знаний (документации ИТС, внутренним инструкциям).

**Слоган:** *"Интеграция чат-функционала Onyx в 1С:Предприятие"*

**Позиционирование:** чат-бот поддержки для пользователей 1С, работающий на основе RAG (ответы по документации). Не имеет доступа к данным базы 1С — работает только с загруженными документами.

**Версия:** 1.0.2 | **Лицензия:** MIT (с обязательным указанием авторства @hawkxtreme)

## 2. Технологический стек

| Компонент | Технология |
|-----------|-----------|
| Язык | 1C BSL (расширение конфигурации) |
| AI-платформа | Onyx (open-source RAG, Docker) |
| LLM | Любой через Onyx: OpenRouter, RouterAI, Ollama (локальный) |
| Протокол | REST API (Onyx Chat API) |
| Транспорт | HTTP request-response (синхронный + фоновые задания) |
| UI | Управляемая форма 1С (HTML-поле для истории чата) |
| HTTP-клиент | Библиотека Connector (onyx_КоннекторHTTP) |
| Развёртывание | Расширение .cfe + Onyx (Docker CPU/GPU) |
| Совместимость | 1С:Предприятие 8.3.27+ (управляемые формы) |
| RAG (база знаний) | Onyx Connectors (файлы, URL, Slack, и т.д.) |

## 3. Архитектура

```
Пользователь 1С
     |
     v
Расширение Onyx (.cfe) ──HTTP REST──> Onyx Server (Docker)
     |                                      |
     |  POST /api/chat/send-message         |── LLM Provider (OpenRouter / RouterAI / Ollama)
     |  GET  /api/chat/get-chat-session     |── RAG Engine (векторный поиск по документам)
     |  GET  /api/persona                   |── Connectors (файлы ИТС, инструкции)
     |                                      |
     v                                      v
HTML-чат в форме 1С              Onyx Admin UI (:3000)
(история, агенты, сессии)        (настройка ботов, RAG, LLM)
```

**Ключевое:** Onyx — это отдельный сервер (Docker), который берёт на себя всю AI-логику: RAG, LLM, агенты, сессии. Расширение 1С — тонкий HTTP-клиент к API Onyx.

### Поток отправки сообщения

1. Пользователь вводит текст, нажимает "Отправить"
2. Форма блокирует ввод, запускает **фоновое задание** на сервере 1С
3. Фоновое задание: `POST /api/chat/send-message` — получает `reserved_assistant_message_id`
4. **Polling** (до 6 попыток с задержкой 0.7 сек): `GET /api/chat/get-chat-session/{id}` — ищет ответ ассистента по message_id
5. Ответ помещается во временное хранилище — клиент (обработчик ожидания, 1 сек) забирает результат
6. HTML-история обновляется целиком

## 5. MCP-инструменты

**MCP-инструментов НЕТ.** Решение не использует протокол MCP и не предоставляет AI-агенту доступ к данным базы 1С. AI работает исключительно с базой знаний Onyx (загруженные документы, RAG).

| Возможность | Onyx AI Assistant | Аналог в Lyra |
|-------------|-------------------|---------------|
| Чат с AI из 1С | Да (HTTP -> Onyx API) | Да (WebSocket -> Centrifugo -> Claude) |
| Доступ к данным базы | **Нет** | Да (exec-команды, MCP) |
| RAG по документации | Да (через Onyx Connectors) | Да (mcp-1c-docs — поиск по документации ИТС) |
| Выполнение запросов 1С | **Нет** | Да (`v8_query`) |
| Выполнение кода 1С | **Нет** | Да (`v8_exec`) |
| Метаданные конфигурации | **Нет** | Да (`v8_metadata`, Vega MCP) |
| Навигационные ссылки | **Нет** | Да (`v8_navlink`) |

## 6. Ключевые технические решения

### 6.1 Onyx как AI-бэкенд

Вся AI-логика вынесена в Onyx — open-source RAG-платформу. Onyx предоставляет:
- Управление агентами (persona) с системными промптами
- Сессии чата с историей сообщений
- RAG: векторный поиск по загруженным документам
- Подключение любых LLM (OpenRouter, Ollama, и др.)
- Web UI для администрирования (порт 3000)

Расширение 1С — тонкий HTTP-клиент, ~2500 строк BSL (без onyx_КоннекторHTTP).

### 6.2 Синхронный request-response (без стриминга)

Нет стриминга ответов. Отправка сообщения -> ожидание полного ответа через polling (каждые 0.7 сек, до 6 попыток). Время ответа 30-60 секунд по документации. Пользователь видит только индикатор "печатает..." и затем готовый ответ целиком.

### 6.3 Фоновые задания для асинхронности

Отправка сообщения выполняется в фоновом задании (`ФоноваяОтправкаСообщения`). Клиент подключает `ОбработчикОжидания` с интервалом 1 сек, проверяя завершение задания. Поддержка нескольких параллельных чатов через таблицу `Чаты` на форме.

### 6.4 HTML-рендеринг истории чата

История чата отображается через HTML-поле формы. Сообщения пользователя — справа (синий), ассистента — слева (белый). При каждом обновлении HTML перестраивается целиком (вызов `ПолучитьHTMLИсторииЧатаНаСервере`).

### 6.5 Библиотека Connector

Для HTTP-запросов используется библиотека [Connector](https://github.com/vbondarevsky/Connector) — популярная BSL-библиотека для HTTP. Модуль `onyx_КоннекторHTTP` — полная копия библиотеки (~1700 строк), встроенная в расширение.

### 6.6 Журнал работы с API

Все HTTP-запросы логируются в регистр сведений `onyx_ЖурналРаботыСAPI` с типом запроса, типом операции, статусом и деталями ответа.

### 6.7 Множественные контуры (Production / Sandbox)

Константа `onyx_ТекущийКонтурAPI` позволяет переключаться между Production и Sandbox окружениями Onyx.

### 6.8 Поддержка Ollama (локальный AI)

Решение акцентирует внимание на конфиденциальности: при использовании Docker + Ollama данные не покидают инфраструктуру. Это серьёзный маркетинговый аргумент для компаний с закрытой документацией.

### 6.9 Scraping ИТС

Отдельный проект автора ([scraping_its](https://github.com/hawkxtreme/scraping_its)) — Python-скрипт для скачивания документации ИТС в форматах Markdown/JSON/PDF для загрузки в Onyx как базу знаний.

### 6.10 RouterAI — российский LLM-провайдер

Поддержка [RouterAI](https://routerai.ru) — российский сервис (ООО "Интер", ИНН 7017351722), серверы в РФ, оплата в рублях. Подключается через Custom LLM Provider в Onyx (OpenAI-совместимый API).

# Сравнение с Lyra

## Принципиальные отличия

| Аспект | Onyx AI Assistant | Lyra |
|--------|-------------------|------|
| **Целевая аудитория** | Конечные пользователи 1С (поддержка) | Конечные пользователи 1С (ассистент) |
| **AI-платформа** | Onyx (RAG, open-source) | Claude (Anthropic) через Роутер |
| **Доступ к данным базы** | **Нет** (только RAG по документам) | **Да** (exec, MCP, запросы) |
| **LLM** | Любой через Onyx (OpenRouter, Ollama) | Claude (Haiku/Sonnet/Opus) |
| **Стриминг ответа** | **Нет** (polling, 30-60 сек ожидание) | **Да** (text_delta, thinking) |
| **Транспорт** | HTTP REST (синхронный) | WebSocket (Centrifugo, реальное время) |
| **Метаданные конфигураций** | **Нет** | Vega MCP (парсер, граф) |
| **RAG по документации** | **Да** (Onyx Connectors, ИТС) | **Да** (mcp-1c-docs) |
| **Локальный AI** | **Да** (Ollama, Docker GPU) | **Нет** (облачный Claude) |
| **Мобильное приложение** | **Нет** | **Да** (Flutter, QR) |
| **Биллинг** | **Нет** (свой API ключ LLM) | **Да** (серверный) |
| **Мультитенантность** | Один пользователь Onyx | Сессии через Centrifugo |
| **Форма поставки** | Расширение .cfe | Расширение .cfe + внешняя обработка .epf |
| **Зависимости** | Onyx (Docker, 4-16 GB RAM, GPU опц.) | Centrifugo + Claude API |
| **Время ответа** | 30-60 сек (polling) | 1-3 сек до первого токена (стриминг) |
| **Модератор** | **Нет** | **Да** (проверка сообщений, легенда "Лира") |

## Наши преимущества

1. **Доступ к данным базы** — AI может выполнять запросы, читать метаданные, вызывать код 1С. Onyx работает только по документам
2. **Стриминг** — ответ виден по мере генерации (text_delta), thinking-блоки. Onyx — ожидание 30-60 сек без обратной связи
3. **WebSocket транспорт** — Centrifugo обеспечивает реальное время, push-уведомления. Onyx — HTTP polling
4. **Vega MCP** — знание структуры типовых конфигураций без подключения к базе
5. **Мобильное + авторизация** — QR, SMS, профиль пользователя. Onyx — только из 1С
6. **Биллинг** — серверный учёт, контроль расходов клиентов
7. **Качество AI** — Claude (Opus/Sonnet) существенно превосходит модели, доступные через Ollama
8. **Архитектура масштабирования** — Centrifugo + сессии, готово для мультитенантности
9. **Модератор** — проверка входящих сообщений, соблюдение легенды "Лира"

## Преимущества конкурента

1. **RAG по документации ИТС** — ответы на основе официальной документации, с citation. У нас есть mcp-1c-docs (MCP-сервер с поиском по документации), но без citation и Onyx-стиля коннекторов
2. **Локальный AI (Ollama)** — полная конфиденциальность, данные не покидают инфраструктуру. У нас только облачный Claude
3. **Множество LLM** — OpenRouter даёт доступ к GPT, Claude, Gemini, Llama и др. через единый API
4. **Простота развёртывания** — расширение .cfe + Docker. Не нужен Centrifugo, Node.js, Rust-бинарники
5. **Open-source AI-платформа** — Onyx бесплатный, код открыт, можно кастомизировать
6. **Scraping ИТС** — готовый инструмент для скачивания документации ИТС для RAG
7. **Журнал API-запросов** — полное логирование всех обращений к AI (тип, статус, время). Удобно для аудита
8. **Контуры (Production/Sandbox)** — разделение окружений для тестирования и продакшена
9. **RouterAI** — поддержка российского сервиса с серверами в РФ и оплатой в рублях
10. **Мастер первого запуска** — пошаговая настройка при первом использовании

# Что полезного взять в Lyra

## Приоритет 1 -- Высокий

### 1. ~~RAG по документации (база знаний)~~ — уже реализовано
**Статус:** У нас есть **mcp-1c-docs** — MCP-сервер для поиска по документации ИТС. Claude получает доступ к документации через MCP-инструменты (search_docs, scrape_docs и др.).
**Что можно улучшить:** citation (ссылки на конкретные разделы документации в ответах), коннекторы к внутренним инструкциям пользователей (как у Onyx).

### 2. Журнал работы с AI
**Что:** Логировать все обращения к AI: запрос, ответ, время, токены, модель, ошибки.
**Зачем:** Аудит использования, диагностика проблем ("почему AI ответил так?"), основа для аналитики качества ответов и биллинга. У конкурента это `onyx_ЖурналРаботыСAPI` — простой регистр сведений, но полезный.
**Реализация:** Регистр сведений на стороне Роутера. Записывать: session_id, user_id, текст вопроса (хеш), длина ответа, модель, токены (input/output), время генерации, статус.

## Приоритет 2 -- Средний

### 3. Поддержка локальных моделей (Ollama)
**Что:** Опциональная возможность использовать локальные LLM вместо Claude.
**Зачем:** Конфиденциальность — крупные компании не хотят отправлять данные в облако. Onyx делает на этом сильный маркетинговый акцент. Также снижает стоимость при высоком объёме запросов.
**Реализация:** centrifugo_stdio уже поддерживает произвольные CLI. Добавить конфиг для Ollama API вместо Claude CLI. Качество ответов будет ниже — но для простых вопросов по документации достаточно.

### 4. Scraping ИТС для базы знаний
**Что:** Инструмент для скачивания документации ИТС (ERP, БП, УТ) в формате, пригодном для RAG.
**Зачем:** Если делаем RAG (п.1), нужен источник данных. Конкурент уже решил эту задачу — [scraping_its](https://github.com/hawkxtreme/scraping_its), MIT лицензия.
**Реализация:** Можно использовать готовый инструмент конкурента или написать свой. Форматы: Markdown, JSON, PDF.

### 5. Мастер первоначальной настройки
**Что:** Пошаговый помощник настройки при первом запуске.
**Зачем:** Снижает порог входа. У конкурента `onyx_ПомощникПервогоЗапуска` проводит через настройку API, загрузку агентов, проверку соединения.
**Реализация:** Аналогичный мастер в Чате: проверка Centrifugo, настройка подключения, тестовое сообщение.

## Приоритет 3 -- Низкий

### 6. Контуры окружения (Production/Sandbox)
**Что:** Переключение между тестовым и рабочим окружением.
**Зачем:** Удобно для разработки и тестирования без риска для продакшена.

### 7. Множественные LLM-провайдеры
**Что:** Возможность выбирать модель (GPT, Claude, Gemini) через единый API (OpenRouter).
**Зачем:** Гибкость, выбор по цене/качеству. Но для Lyra Claude — базовый бренд, смена модели размывает позиционирование.

### 8. Маркетинговый акцент на конфиденциальность
**Что:** Явно описывать политики конфиденциальности для каждого варианта развёртывания.
**Зачем:** Конкурент уделяет этому серьёзное внимание — подробные описания для каждого варианта (облако, локальный, Ollama). Это важно для корпоративных клиентов.
**Реализация:** Добавить раздел в документацию и на сайт.
