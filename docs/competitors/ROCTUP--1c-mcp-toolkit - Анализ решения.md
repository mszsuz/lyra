# Анализ конкурента: ROCTUP / 1C MCP Toolkit

## 1. Общее описание

**1C MCP Toolkit** — система интеграции AI-агентов (Kiro, Claude Desktop и др.) с базами 1С:Предприятие через стандартный MCP-протокол. Ориентирован на **разработчиков и аналитиков**, которые используют AI-IDE (Kiro, Claude Desktop, Cursor) для работы с данными 1С.

**Слоган:** *"Система интеграции AI-агентов с базами данных 1С:Предприятие через MCP и REST API"*

**Позиционирование:** инструмент для разработчиков, не продукт для конечных пользователей.

## 2. Технологический стек

| Компонент | Технология |
|-----------|-----------|
| Язык | 1C BSL (обработка) + Python (прокси-сервер) |
| Протокол | MCP (streamable-http) + REST API |
| LLM | Любой AI-агент (Kiro, Claude Desktop, Cursor) |
| Развёртывание | Внешняя обработка (.epf) + Python/Docker (прокси) или NativeAPI (встроенный сервер) |
| Транспорт | HTTP long polling (прокси) или встроенный HTTP-сервер (NativeAPI) |
| Совместимость | 1С:Предприятие 8.2.13+ / 8.3.25 |

## 3. Архитектура

### Вариант 1: Встроенный сервер (рекомендуемый)

```
AI-агент (Kiro, Claude) ──► HTTP ──► 1С EPF (NativeAPI HTTP-сервер) ──► База 1С
                          MCP/REST     порт 6003
```

HTTP-сервер прямо внутри обработки 1С через NativeAPI компоненту `MCPHttpTransport` (C++). Без Python.

### Вариант 2: Прокси (Python)

```
AI-агент ──► MCP/REST ──► Python Proxy (Starlette + MCP SDK) ──► HTTP Long Polling ──► 1С EPF ──► База 1С
                           порт 6003                               /1c/poll, /1c/result
```

**Ключевое:** Python-прокси + обработка 1С через long polling. Docker-ready.

## 4. MCP-инструменты (8 штук)

| Инструмент | Назначение | Аналог в Lyra |
|------------|-----------|---------------|
| `execute_query` | Запрос на языке 1С + параметры + schema + limit | `v8_query` (без params, schema, limit) |
| `execute_code` | Выполнение кода (результат через переменную `Результат`) | `v8_exec` (без возврата значения) |
| `get_metadata` | Детальная структура: filter, sections, extensions, пагинация | `v8_metadata` (только список) |
| `get_event_log` | Журнал регистрации с фильтрами и курсорной пагинацией | **Нет аналога** |
| `get_link_of_object` | Навигационная ссылка из результатов запроса | `v8_navlink` (наш новый) |
| `get_object_by_link` | Данные объекта по навигационной ссылке | **Нет аналога** |
| `find_references_to_object` | Поиск всех ссылок на объект (where-used) | **Нет аналога** |
| `get_access_rights` | Права ролей + эффективные права пользователя | **Нет аналога** |

## 5. Ключевые технические решения

### 5.1 Автоконвертация ссылочных типов (`_objectRef`)

Ссылочные поля в результатах запроса автоматически конвертируются в структурированные описания:

```json
{
  "_objectRef": true,
  "УникальныйИдентификатор": "a1b2c3d4-...",
  "ТипОбъекта": "СправочникСсылка.Контрагенты",
  "Представление": "ООО \"Рога и Копыта\""
}
```

Эти описания можно передавать обратно как параметры запроса — система автоматически конвертирует в ссылки. AI может «ходить» по связям объектов без знания внутренней структуры.

**Это очень сильная идея** — у нас `v8_query` возвращает `Строка(Ссылка)` и теряет UUID.

### 5.2 TOON-формат (экономия токенов 30-60%)

Собственный формат сериализации (NativeAPI компонента `ToonConverter`), компактнее JSON для табличных данных. Формат по умолчанию для ответов.

### 5.3 Детальные метаданные (`get_metadata`)

Единый инструмент с несколькими режимами:
- **Сводка** (без параметров) → корневые типы с количеством + info о конфигурации
- **Список** (`meta_type`, `name_mask`) → объекты с пагинацией (limit/offset)
- **Детали** (`filter`) → реквизиты с типами, табличные части, формы, команды, макеты
- **Расширения** (`extension_name`) → объекты из расширений

### 5.4 Журнал регистрации (`get_event_log`)

Мощный инструмент для диагностики: фильтры по уровню, событию, пользователю, приложению, метаданным, комментарию, транзакции. Курсорная пагинация.

### 5.5 Поиск ссылок (`find_references_to_object`)

Двухэтапный: обход метаданных → запросы. С timeout budget и limit. Находит где используется объект (документы, регистры).

### 5.6 Skills для AI-агентов

Готовые инструкции (skills) для AI-агентов:
- `composing-1c-queries` — справочник синтаксиса запросов 1С (CASE, UNION, виртуальные таблицы, оптимизация)
- `calling-1c-rest-api-via-curl` — полное руководство по REST API

### 5.7 Изоляция каналов

`?channel=dev-environment` в URL — маршрутизация команд к конкретному клиенту 1С. Несколько баз через один прокси.

### 5.8 Безопасность

- Настраиваемый чёрный список опасных операций
- Режим `ALLOW_DANGEROUS_WITH_APPROVAL` — подтверждение пользователем
- `autoApprove` для безопасных инструментов

## 6. NativeAPI компоненты (C++)

Две компоненты, встроенные в макеты обработки:

| Компонента | Назначение |
|------------|-----------|
| `MCPHttpTransport` | HTTP-сервер + SSE потоки, прямо в процессе 1С. Обрабатывает MCP и REST запросы |
| `ToonConverter` | Конвертация данных в TOON-формат для экономии токенов |

# Сравнение с Lyra

## Принципиальные отличия

| Аспект | ROCTUP (MCP Toolkit) | Lyra |
|--------|---------------------|------|
| **Целевая аудитория** | Разработчики/аналитики (AI-IDE) | Конечные пользователи 1С |
| **UI** | Нет своего UI (используется IDE) | HTML-чат в интерфейсе 1С |
| **LLM** | Любой (через IDE) | Claude (через Роутер) |
| **Стриминг ответа** | Через IDE | Собственный (text_delta, thinking) |
| **Транспорт** | HTTP (MCP streamable-http) | WebSocket (Centrifugo) |
| **Метаданные типовых** | Нет (только из подключённой базы) | Vega MCP (парсер конфигураций, граф) |
| **Мобильное приложение** | Нет | Да (Flutter, авторизация по QR) |
| **Биллинг** | Нет (API ключ пользователя) | Да (серверный, учёт токенов) |
| **Протокол инструментов** | Стандартный MCP | MCP (через tools-mcp.mjs) |
| **Мультитенантность** | Каналы (channel isolation) | Сессии (Centrifugo channels) |

## Наши преимущества

1. **Продукт для конечных пользователей** — чат в интерфейсе 1С, не нужна IDE
2. **Стриминг** — ответ виден по мере генерации, thinking-блоки
3. **Vega** — знание структуры типовых конфигураций без подключения к базе
4. **Мобильное + авторизация** — QR, SMS, профиль пользователя
5. **Системный промпт** — Лира скрывает техническую сложность от пользователя
6. **Биллинг** — серверный учёт, контроль расходов

## Преимущества конкурента

1. **Встроенный HTTP-сервер** — NativeAPI, без Bridge/Node.js/Python, всё в одной EPF
2. **Автоконвертация ссылок** (`_objectRef`) — AI ходит по связям без знания UUID
3. **Детальные метаданные** — реквизиты с типами, табличные части, формы, extensions в одном инструменте
4. **Журнал регистрации** — мощный диагностический инструмент
5. **Поиск ссылок** (where-used) — «где используется этот контрагент?»
6. **Права доступа** — проверка прав конкретного пользователя
7. **TOON-формат** — экономия 30-60% токенов на табличных данных
8. **Параметризованные запросы** — `params` с автоконвертацией ссылок
9. **Skills** — готовые инструкции по синтаксису запросов для AI
10. **Schema в результатах** — типы колонок без дополнительного запроса
11. **Стандартный MCP** — streamable-http, совместимость с любой IDE
12. **Docker** — готовый образ для развёртывания

# Что полезного взять в Lyra

## Приоритет 1 — Высокий

### 1. Автоконвертация ссылочных типов (`_objectRef`)
**Что:** В результатах `v8_query` возвращать ссылки как `{_objectRef, UUID, Тип, Представление}`. Принимать эти объекты как параметры в следующих запросах.
**Зачем:** Сейчас `ЗначениеДляJSON()` делает `Строка(Ссылка)` — UUID теряется. AI не может использовать результат одного запроса как параметр другого. Это **ключевое ограничение** текущей реализации.
**Реализация:** Доработать `ЗначениеДляJSON()` в МодульMCP — определять ссылочные типы, возвращать структурированное описание. В `v8_query` принимать `params` и конвертировать `_objectRef` обратно в ссылки.

### 2. Детальная структура метаданных (`v8_structure`)
**Что:** Расширить `v8_metadata` (или новый `v8_structure`) — возвращать реквизиты с типами, табличные части, владельцев.
**Зачем:** Сейчас AI вынужден делать `ВЫБРАТЬ ПЕРВЫЕ 1 * ИЗ ...` чтобы понять структуру. У конкурента это одним вызовом `get_metadata(filter="Справочник.Номенклатура")`.
**Реализация:** Обход `МетаданныеОбъекта.Реквизиты`, `.ТабличныеЧасти` и т.д.

### 3. Журнал регистрации (`v8_event_log`)
**Что:** Инструмент для чтения журнала регистрации с фильтрами.
**Зачем:** Диагностика проблем — «почему документ не проводится?», «кто удалил контрагента?», «какие ошибки были вчера?». Очень ценно для роли analyst/developer.
**Реализация:** `ЖурналРегистрации.Выбрать()` на сервере, сериализация результатов.

## Приоритет 2 — Средний

### 4. Параметризованные запросы в `v8_query`
**Что:** Добавить `params` в `v8_query` с автоконвертацией `_objectRef` в ссылки.
**Зачем:** Итеративные запросы: нашёл контрагента → ищу его документы. Без параметров AI пишет `ГДЕ Контрагент = &Контрагент` и не может подставить значение.
**Реализация:** `Запрос.УстановитьПараметр(Имя, Значение)` с конвертацией.

### 5. Получение объекта по ссылке (`v8_get_object`)
**Что:** Получить все реквизиты объекта по UUID/навигационной ссылке.
**Зачем:** «Покажи детали этого документа» — без написания запроса.
**Реализация:** `ПолучитьСсылку(UUID) → ПолучитьОбъект() → сериализация реквизитов`.

### 6. Поиск ссылок (`v8_find_references`)
**Что:** «Где используется этот объект?» — поиск во всех документах/регистрах.
**Зачем:** Проверка зависимостей, аудит использования.
**Реализация:** Обход метаданных + запросы по кандидатным полям (как у конкурента).

### 7. Schema в результатах запроса
**Что:** `include_schema: true` в `v8_query` — возвращать типы колонок.
**Зачем:** AI сразу понимает структуру данных без дополнительных вызовов.
**Реализация:** `Результат.Колонки[i].ТипЗначения` → массив типов.

## Приоритет 3 — Низкий

### 8. Права доступа (`v8_access_rights`)
**Что:** Проверить права конкретного пользователя на объект метаданных.
**Зачем:** «Почему я не могу открыть этот документ?» — диагностика прав.

### 9. Оптимизация размера ответов
**Что:** Компактный формат для табличных данных (аналог TOON или column-oriented JSON).
**Зачем:** Экономия токенов при больших результатах.

### 10. Skills / справочник синтаксиса
**Что:** Детальный справочник синтаксиса запросов 1С как часть промпта или отдельный ресурс.
**Зачем:** У конкурента хорошие reference-документы по CASE, UNION, виртуальным таблицам. У нас в промпте только базовый синтаксис.
