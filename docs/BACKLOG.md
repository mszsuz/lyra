## Загрузчик и подключение

- [ ] **Конфигурация подключения через GitHub** — JSON-файл в публичном репо (`github.com/mszsuz/lyra/config.json`) содержит адрес сервера, версию EPF и URL для скачивания. EPF и мобильное при запуске делают GET на `raw.githubusercontent.com/.../config.json`, получают актуальный `ws_url`. Менять сервер — один коммит в GitHub, без пересборки клиентов. Формат:
  ```json
  {"ws_url": "wss://lyra.1ext.com/connection/websocket", "epf_version": "26.03.22.2", "epf_url": "https://..."}
  ```
  Кэш в `ХранилищеОбщихНастроек` — если GitHub недоступен, подключаться по сохранённому адресу. GitHub CDN кэширует ~5 мин — допустимо для смены сервера. Когда появится свой домен — endpoint переедет на `https://lyra.1ext.com/config.json`, в GitHub — redirect
- [ ] **Автообновление обработки** — Загрузчик сравнивает `epf_version` из config.json с `Комментарий` метаданных. Если новее — предлагает скачать по `epf_url`, сохранить, переоткрыть

## UI рабочая область (workspace)

Концепция: чат справа, рабочая область слева. Рабочая область — **программно генерируемые страницы** формы обработки с динамически добавляемым содержимым (реквизиты, элементы формы, таблицы). Лира через инструменты создаёт страницы и наполняет их данными. Внизу — вкладки страниц.

- [ ] **Архитектура рабочей области** — форма обработки с программно добавляемыми страницами (ДобавитьСтраницу). Каждый вызов ui_* создаёт новую страницу или обновляет существующую. Страницы содержат динамически добавленные элементы формы (ТабличныйДокумент, ПолеHTMLДокумента, ТаблицаФормы и т.д.)
- [ ] **Инструменты отображения (в рабочей области):**
  - `ui_table` — страница с ТабличнымДокументом (данные, форматирование, печать, экспорт в Excel)
  - `ui_chart` — страница с диаграммой (встроенные диаграммы 1С в ТабличномДокументе)
- [ ] **Инструменты навигации:**
  - `ui_open_list` — страница с ДинамическимСписком с заданными отборами. Самый частый запрос — «покажи документы за период по контрагенту»
  - `ui_open_form` — открыть форму объекта (перекрывается с `v8_navlink`, но может задавать параметры формы)
  - `ui_open_report` — страница с результатом отчёта (предзаполненные настройки)

## Оптимизация промпта

- [ ] **Параллельные tool calls в системном промпте** — подсказать модели использовать параллельные вызовы инструментов (несколько search_code + lyra_data_query за один turn). В кейсе БП "18 контрагентов сервис сверка" (2026-03-13) модель сделала 21 turn / 2 мин 18 сек — последовательные Vega-запросы и data_query. Параллельные вызовы могли бы сократить до ~10-12 turns. **Риск:** параллельные вызовы быстрее расходуют токены (больше контекста за turn). Нужно замерить: экономия времени vs рост расхода токенов. Возможно, включать только для определённых моделей или ролей
- [ ] **"Один поиск — одно действие"** — усилить правило: получил результат из Vega → сразу выполняй запрос, не делай повторных search_code для перепроверки

## Память

- [ ] **Записная книжка пользователя** (`lyra_note_save`) — личные заметки, не связанные с 1С (предпочтения, напоминания). Отдельное хранилище от `lyra_memory_*`. Реализовать только при реальном запросе от пользователей
- [ ] **Память как часть документации в Vega** — знания Лиры = подмножество документации в едином индексе Neo4j. Один `search_docs` ищет по всем источникам. Текущий файловый MVP (registry.md + skills/*.md) заменяется на Vega. Разделение по `source`:
  - `1c-docs` — официальная документация 1С
  - `lyra-config` — общие знания по конфигурации (проверенные курирующей моделью)
  - `lyra-user` — личные знания пользователя
  - Методы: `memory_save` (сохранить с source + scope), `memory_delete` (удалить устаревшее). Поиск и чтение — через существующий `search_docs`
  - Реестр в промпте убрать — вместо него подсказка «перед исследованием вызови search_docs»
- [ ] **Курирующий агент** — фоновый процесс (Haiku или Sonnet) обходит `users/*/memory/` всех пользователей по расписанию. Задачи:
  1. Рецензирует сохранённые запросы/знания — проверяет корректность, полноту
  2. Обезличивает — убирает конкретные имена контрагентов, суммы, даты, оставляет структуру и паттерн
  3. Сохраняет проверенные знания в общую базу (`memory/<config>/skills/`) — доступны всем пользователям этой конфигурации
  4. Помечает обработанные файлы чтобы не проверять повторно
  - Пример: пользователь сохранил «дебиторка ООО Ромашка за март» → агент обезличивает → «запрос дебиторской задолженности по контрагенту за период» → общий навык
- [ ] **Поиск по истории диалогов** (`lyra_history_search`) — поиск по прошлым сессиям пользователя. «Я вчера спрашивал про...» → находит в JSONL-логах. **Критично:** доступ только к `.users/<свой_user_id>/` — чужие диалоги недоступны. Фильтрация по user_id на уровне инструмента, не промпта

## Будущее развитие

- [ ] **Чат в мобильном приложении** — полноценный чат с Claude прямо в мобильном. Пользователь может отойти от компьютера, продолжить общение голосом через телефон. Чат в 1С остаётся открытым (позже — работа в фоне). Канал сессии единый — мобильное видит весь стриминг Claude и отображает в своём чате. Голосовой ввод через STT уже в плане, нужен ещё UI чата и отображение ответов Claude
- [ ] **Темы оформления (скины)** — переключаемые темы в мобильном приложении. 7 дизайн-концептов уже готовы (Glassmorphism, Minimal, Neobrutalism, Cyberpunk, Organic, Neumorphism, Flat). Реализация через кастомный ThemeData во Flutter. MVP — одна тема + dark/light, потом 3-4 темы на выбор в настройках
- [ ] **Геймификация (ачивки)** — система достижений для пользователей Лиры. Ref: [habr.com/mosigra/228593](https://habr.com/ru/companies/mosigra/articles/228593/). Идеи:
  - **Онбординг**: «Первый вопрос», «Первый exec-запрос», «Подключил мобильное» — обучают функциям через награды
  - **Вовлечение**: «10 сессий», «Подключил 3 базы», «Голосовой вопрос» — мотивируют использовать разные возможности
  - **Экспертиза**: «100 запросов к метаданным», «Нашёл баг через v8_query» — показывают глубину использования
  - **Социальная роль**: видимые значки в профиле — «старожил», «бета-тестер», «power user»
  - **Лояльность**: серии дней использования, бонусы за безупречные серии (как в Мосигре — серии безошибочных заказов)
  - Отображение: мобильное приложение (профиль) + сайт (личный кабинет). В чате 1С — не нужно (отвлекает)
  - Без манипулятивной монетизации — не привязывать ачивки к платным действиям
